## 1. 프로젝트 개요

### 1.1 연구 배경

최근 ChatGPT, Claude, Gemini 등 대형 언어 모델(LLM) API를 활용한 서비스가 급증하고 있습니다. 그러나 다수의 사용자가 동시에 요청을 보내면 다음과 같은 문제가 발생합니다:

- **비용 폭증**: API 호출 비용이 기하급수적으로 증가
- **응답 지연**: 동시 요청 처리로 인한 전반적인 지연 시간 증가
- **불공정한 처리**: 모든 요청을 동등하게 처리하면 긴급한 요청도 대기해야 함
- **자원 낭비**: 비효율적인 요청 관리로 인한 시스템 자원 낭비

### 1.2 연구 목적

본 프로젝트는 운영체제(OS) 과목에서 학습한 프로세스 스케줄링 알고리즘을 LLM API 요청 관리에 적용하여, 효율적이고 공정한 요청 처리 시스템을 구축하는 것을 목표로 합니다.

### 1.3 핵심 아이디어

| OS 개념 | LLM 적용 |
|---------|----------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 호출 권한 |
| 우선순위 | 사용자 등급, 요청 긴급도 |
| 스케줄링 알고리즘 | 요청 처리 순서 결정 |
| 타임 퀀텀 | 요청 처리 시간 제한 |
| 멀티레벨 큐 | 우선순위별 요청 분류 |

### 1.4 구현 범위

본 프로젝트에서 구현한 스케줄링 알고리즘:

1. **FCFS (First-Come, First-Served)**: 도착 순서대로 처리하는 기본 알고리즘
2. **Priority Queue**: 우선순위 기반 처리 알고리즘
3. **MLFQ (Multi-Level Feedback Queue)**: 적응형 다단계 피드백 큐
4. **WFQ (Weighted Fair Queuing)**: 가중치 기반 공정 큐잉

## 3. 구현 상세

### 3.1 도메인 모델

#### 3.1.1 요청 우선순위 (RequestPriority)

```typescript
enum RequestPriority {
  LOW = 0,      // 낮은 우선순위 (배치 작업)
  NORMAL = 1,   // 일반 우선순위 (기본값)
  HIGH = 2,     // 높은 우선순위 (유료 사용자)
  URGENT = 3,   // 긴급 우선순위 (실시간 응답 필요)
}
```

#### 3.1.2 요청 상태 (RequestStatus)

```typescript
enum RequestStatus {
  PENDING = 'pending',       // 대기 중
  QUEUED = 'queued',         // 큐에 등록됨
  PROCESSING = 'processing', // 처리 중
  COMPLETED = 'completed',   // 완료됨
  FAILED = 'failed',         // 실패함
  CANCELLED = 'cancelled',   // 취소됨
}
```

#### 3.1.3 LLM 요청 (LLMRequest)

```typescript
interface LLMRequest {
  id: string;           // UUID
  prompt: string;       // 프롬프트
  provider: LLMProvider; // LLM 제공자 (ollama/openai)
  priority: RequestPriority;
  status: RequestStatus;
  metadata?: Record<string, any>;
  createdAt: Date;
  updatedAt: Date;
}
```

### 3.2 스케줄러 인터페이스

모든 스케줄러가 구현하는 공통 인터페이스:

```typescript
interface IScheduler {
  initialize(): Promise<void>;           // 초기화
```

### 3.3 FCFS 스케줄러 구현

**First-Come, First-Served** - 가장 단순한 스케줄링 알고리즘

**핵심 특징:**
- 요청 도착 순서대로 처리
- BullMQ의 기본 큐 사용
- 우선순위 필드가 있으면 존중 (확장된 FCFS)

**시간 복잡도:**
- 삽입: O(1)
- 선택: O(1)
- 공간: O(n)

### 3.4 Priority 스케줄러 구현

**Priority Queue** - 우선순위 기반 스케줄링

**핵심 특징:**
- 높은 우선순위 요청 먼저 처리
- 에이징 메커니즘으로 기아 현상 방지
- 비선점형 스케줄링 (실행 중인 작업은 중단하지 않음)

**우선순위 매핑:**

| RequestPriority | BullMQ Priority | 처리 순서 |
|-----------------|-----------------|-----------|
| URGENT (3) | 0 | 1순위 |
| HIGH (2) | 2 | 2순위 |
| NORMAL (1) | 4 | 3순위 |
| LOW (0) | 6 | 4순위 |

**에이징 메커니즘:**
- 에이징 간격: 60초
- 에이징 임계값: 2분 (120,000ms)
- 최대 승격 횟수: 2단계

### 3.5 MLFQ 스케줄러 구현

**Multi-Level Feedback Queue** - OSTEP 교과서 기반 구현

**MLFQ 5가지 규칙:**

| 규칙 | 설명 | 구현 |
|------|------|------|
| Rule 1 | Priority(A) > Priority(B)이면 A만 실행 | 큐 레벨 순회로 구현 |
| Rule 2 | Priority(A) = Priority(B)이면 라운드로빈 | BullMQ 기본 동작 활용 |
| Rule 3 | 새 작업은 최고 우선순위(Q0)에서 시작 | submit()에서 Q0에 추가 |
| Rule 4 | 타임 슬라이스 소진 시 강등, 양보 시 유지 | processJob()에서 시간 추적 |
| Rule 5 | 주기적으로 모든 작업을 Q0로 부스팅 | BoostManager 사용 |

**큐 구성:**

| 큐 | 타임 퀀텀 | 용도 |
|----|-----------|------|
| Q0 | 1,000ms | 새 작업, 부스팅된 작업 |
| Q1 | 3,000ms | Q0 퀀텀 초과 작업 |
| Q2 | 8,000ms | Q1 퀀텀 초과 작업 |
| Q3 | Unlimited | 장시간 실행 작업 (FCFS 동작) |

**부스팅 메커니즘:**
- 부스팅 간격: 5초
- 모든 대기 중인 작업을 Q0로 이동
- 기아 현상 방지

### 3.6 WFQ 스케줄러 구현

**Weighted Fair Queuing** - 멀티테넌트 공정 스케줄링

**핵심 개념:**
- **가상 시간 (Virtual Time)**: GPS(Generalized Processor Sharing) 근사
- **가상 종료 시간 (Virtual Finish Time)**: 작업 스케줄링 순서 결정
- **테넌트 가중치**: 대역폭 비례 할당

**테넌트 계층 및 가중치:**

| 계층 | 가중치 | 대역폭 비율 |
|------|--------|-------------|
| Enterprise | 100 | 100x |
| Premium | 50 | 50x |
| Standard | 10 | 10x |
| Free | 1 | 1x (기준) |

**가상 종료 시간 계산:**

```
F_i = max(V(t), F_i_prev) + (L_i / r_i)

여기서:
- F_i: 요청 i의 가상 종료 시간
- V(t): 현재 가상 시간
- F_i_prev: 테넌트 i의 이전 종료 시간
- L_i: 서비스 시간 (예상 처리 시간)
- r_i: 테넌트 가중치
```

**Jain's Fairness Index:**

```
J = (sum x_i)^2 / (n * sum x_i^2)

여기서:
- x_i: 테넌트 i가 받은 서비스 시간
- n: 활성 테넌트 수
- J: 공정성 지수 (0~1, 1이 완벽한 공정성)
```

|--------|------|------|
| 평균 처리 시간 | 159.25 | ms |
| 평균 대기 시간 | 48.25 | ms |
| 시스템 처리량 | ~6.3 | RPS |

### 5.2 알고리즘 비교

| 알고리즘 | 평균 대기시간 | 처리량 | 공정성 | 적합한 사용 사례 |
|----------|--------------|--------|--------|------------------|
| FCFS | 기준 | 기준 | 낮음 | 단순 배치 작업, 우선순위 불필요 |
| Priority | 30% 개선 | 유지 | 낮음 | 유료 사용자 우선, SLA 보장 |
| MLFQ | 40% 개선 | 20% 증가 | 높음 | 혼합 워크로드, 짧은 요청 최적화 |
| WFQ | 35% 개선 | 유지 | 매우 높음 | 멀티테넌트 SaaS, QoS 보장 |

### 5.3 예상 성능 개선

**Priority 스케줄러:**
- URGENT 요청: 60-80% 대기시간 감소
- HIGH 요청: 40-60% 대기시간 감소
- LOW 요청: 약간 증가 (에이징으로 기아 방지)

**MLFQ 스케줄러:**
- 짧은 요청 (< 1초): 70-90% 대기시간 감소
- 중간 요청 (1-8초): 30-50% 대기시간 감소
- 긴 요청 (> 8초): FCFS와 유사 (Q3에서 무제한)

**WFQ 스케줄러:**
- Enterprise 테넌트: 보장된 대역폭
- Free 테넌트: 비례적 서비스 (1/100 비율)
- 동일 계층 내: 완전한 공정성

### 5.4 시간/공간 복잡도

| 알고리즘 | 삽입 | 선택 | 공간 | 비고 |
|----------|------|------|------|------|
| FCFS | O(1) | O(1) | O(n) | 가장 효율적 |
| Priority | O(log n) | O(1) | O(n) | 힙 기반 우선순위 큐 |
| MLFQ | O(1) | O(k) | O(n) | k = 큐 레벨 수 (4) |
| WFQ | O(log n) | O(1) | O(n + t) | t = 테넌트 수 |
