## 2. 상세 비교 분석

### 2.1 LLM Scheduler (후보 1)

#### 개요
OS CPU 스케줄링 알고리즘(FCFS, Priority, MLFQ, WFQ)을 LLM API 요청 처리에 적용한 시스템입니다.

#### 구현된 알고리즘

| 알고리즘 | 설명 | 테스트 커버리지 | 핵심 특징 |
|---------|------|---------------:|----------|
| **FCFS** | 선입선처리 | 100% (20/20) | 가장 단순, 기준선 |
| **Priority** | 우선순위 기반 | 65% (13/20) | Aging으로 기아 현상 방지 |
| **MLFQ** | 다단계 피드백 큐 | 68% (25/37) | OSTEP 5규칙 구현, 적응형 |
| **WFQ** | 가중치 공평 큐 | 85.7% (18/21) | 테넌트별 공평 자원 분배 |

#### 기술 스택
- TypeScript 5.9, Node.js 20 LTS
- BullMQ + Redis (큐 시스템)
- MongoDB (요청 로그)
- Ollama (LLM 추론)

#### 성능 메트릭 (FCFS 기준)

| 지표 | 값 |
|------|-----|
| 평균 처리 시간 | 159.25ms |
| 평균 대기 시간 | 48.25ms |
| 처리량 | ~6.3 RPS |

#### 강점
- ✅ 4가지 알고리즘 비교 가능 (학술적 가치 높음)
- ✅ 정량적 성능 비교 용이
- ✅ 실제 서비스에 즉시 적용 가능
- ✅ 통합 테스트 완료 (6/6 시나리오)

#### 약점
- ⚠️ 일부 알고리즘 테스트 커버리지 낮음 (65%, 68%)
- ⚠️ LLM 추론 의존도로 인한 성능 편차

### 2.2 Memory Manager (후보 2)

#### 개요
OS 페이징과 가상메모리 개념을 AI 에이전트 컨텍스트 관리에 적용한 3계층 계층형 메모리 시스템입니다.

#### 아키텍처

| 계층 | 기술 | 목적 | 접근 속도 |
|:----:|------|------|----------:|
| L1 | Redis | 핫 데이터 캐시 | ~1ms |
| L2 | ChromaDB | 의미론적 검색 | ~10ms |
| L3 | MongoDB | 장기 저장 | ~50ms |

#### 핵심 기능
- LRU (Least Recently Used) 캐시 교체 알고리즘
- 페이지 폴트 처리 및 자동 승격
- 벡터 임베딩 기반 의미론적 검색
- O(1) get/put/evict 연산

#### 기술 스택
- TypeScript 5.9, Node.js 20 LTS
- Redis 7.2 (L1 캐시)
- ChromaDB 1.8 (L2 벡터 DB)
- MongoDB 7.0 (L3 저장소)
- Ollama nomic-embed-text (임베딩)

#### 테스트 결과
- **57/57 테스트 통과 (100%)**
- **코드 커버리지: 94.44%**

#### 강점
- ✅ 가장 높은 테스트 커버리지 (94.44%)
- ✅ RAG(검색 증강 생성)와 연계 가능
- ✅ 실제 AI 에이전트 메모리 관리에 바로 적용
- ✅ 의미론적 검색으로 지능형 컨텍스트 관리

#### 약점
- ⚠️ 3개 데이터베이스 동시 운영으로 복잡도 높음
- ⚠️ 벡터 임베딩 생성에 추가 비용

### 2.3 Deadlock Detector (후보 3)

#### 개요
OS 교착상태 탐지 및 회복 기법을 AI 멀티 에이전트 시스템에 적용한 시스템입니다.

#### 핵심 컴포넌트

| 컴포넌트 | 기능 | 알고리즘/기법 |
|---------|------|--------------|
| **CycleDetector** | 사이클 탐지 | DFS, O(V+E) |
| **VictimSelector** | 희생자 선택 | 5가지 전략 |
| **RollbackManager** | 상태 복구 | 스냅샷 기반 |
| **BankersAlgorithm** | 예방 시스템 | 은행원 알고리즘 |

#### 희생자 선택 전략 (5가지)
1. **LowPriorityFirst**: 가장 낮은 우선순위 에이전트
2. **ShortestWaitTime**: 가장 짧게 대기한 에이전트
3. **MostResourcesHeld**: 가장 많은 자원을 보유한 에이전트
4. **FewestDependencies**: 의존성이 가장 적은 에이전트
5. **YoungestAgent**: 가장 최근 생성된 에이전트

#### 기술 스택
- TypeScript 5.3, Node.js 20 LTS
- MongoDB 7.0+ (Mongoose)
- Redis 7.2+ (캐시)
- Socket.IO 4.6 (실시간 알림)

#### 테스트 결과
- **63/63 테스트 통과 (100%)**
- **코드 커버리지: 66.64%**

#### 성능 메트릭

| 작업 | 시간 |
|------|------|
| 사이클 탐지 (10 노드) | < 1ms |
| 사이클 탐지 (100 노드) | ~5ms |
| 희생자 선택 | < 1ms |
| 총 회복 시간 | ~100ms |

#### 강점
- ✅ 데드락 탐지/예방/회복 3가지 기능 완비
- ✅ 5가지 희생자 선택 전략 비교 가능
- ✅ 실시간 모니터링 (Socket.IO)
- ✅ 은행원 알고리즘으로 예방 시스템 구현

#### 약점
- ⚠️ 실제 멀티 에이전트 환경에서 데드락 발생 빈도 낮아 데모 어려움
- ⚠️ 단일 에이전트 시나리오에서는 효과 없음

### 2.4 Checkpointing (후보 4)

#### 개요
OS 프로세스 체크포인팅을 AI 에이전트에 적용하여 장애 복구 시스템을 구현했습니다.

#### 핵심 컴포넌트

| 컴포넌트 | 기능 | 커버리지 |
|---------|------|---------:|
| **StateSerializer** | 상태 직렬화/역직렬화 | 89.88% |
| **CheckpointManager** | 체크포인트 생성/관리 | 75.8% |
| **RecoveryManager** | 복구 오케스트레이션 | 67.5% |
| **RollbackExecutor** | 상태 롤백 실행 | 42.85% |

#### 체크포인트 타입
- **Full Checkpoint**: 전체 상태 스냅샷
- **Incremental Checkpoint**: 변경사항만 저장 (diff 기반)

#### 기술 스택
- TypeScript 5.9, Node.js 20 LTS
- MongoDB 7.0 (체크포인트 저장소)
- Zod 3.22 (스키마 검증)

#### 테스트 결과
- **46/46 테스트 통과 (100%)**
- **코드 커버리지: 50.66%**

#### 성능 메트릭

| 작업 | 시간 |
|------|------|
| 전체 체크포인트 생성 | ~5-10ms |
| 증분 체크포인트 생성 | ~3-5ms |
| 무결성 검증 | ~1-2ms |
| 전체 체크포인트 복원 | ~10-20ms |

#### 강점
- ✅ 구현 난이도 가장 낮음 (안정성 높음)
- ✅ 실제 장애 복구 시나리오에 바로 적용
- ✅ 증분 체크포인트로 저장소 효율화
- ✅ 주기적 자동 체크포인트 기능

#### 약점
- ⚠️ 단일 에이전트에만 적용 (멀티 에이전트 미지원)
- ⚠️ 외부 시스템 상태는 복구 불가
