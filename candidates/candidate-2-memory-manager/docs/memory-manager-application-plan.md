## LLM 에이전트를 위한 계층형 메모리 시스템

## 1. 프로젝트 개요

### 1.1 프로젝트 정보

- **프로젝트 명칭:** Memory Manager - OS 페이징 개념을 활용한 LLM 에이전트 메모리 관리 시스템
- **대학:** 홍익대학교 컴퓨터공학과
- **학년도:** 2025년 졸업 프로젝트
- **개발 기간:** 2025년 1월 ~ 2025년 2월
- **프로젝트 상태:** 완료 (100% 구현)

### 1.2 문제 정의

대규모 언어 모델(LLM) 기반 AI 에이전트 시스템에서 다음과 같은 심각한 기술적 문제가 존재합니다:

1. **컨텍스트 윈도우 제약:** LLM은 제한된 토큰 윈도우(예: GPT-4 8K/32K, Claude 200K) 내에서만 정보를 처리할 수 있어 장기 대화나 복잡한 작업에서 중요한 맥락이 손실됩니다.

2. **메모리 관리 부재:** 기존 RAG(Retrieval-Augmented Generation) 시스템은 벡터 데이터베이스에만 의존하여 빈번하게 접근하는 핫 데이터(Hot Data)에 대한 최적화가 부족합니다.

3. **검색 효율성 저하:** 모든 검색을 벡터 유사도 검색에만 의존하여, 최근 접근한 중요한 정보의 빠른 검색이 어렵습니다.

4. **비용 효율성 낮음:** 항상 전체 데이터베이스를 검색하여 API 호출 비용이 높고 응답 시간이 길어집니다.

### 1.3 해결 방안 개요

운영체제(OS)의 메모리 계층 구조와 페이징(Paging) 기법을 LLM 에이전트의 메모리 관리에 적용하여 다음과 같은 혁신적인 해결책을 제안합니다:

- **3계층 메모리 계층 구조:** L1 캐시(Redis) → L2 벡터 DB(ChromaDB) → L3 스토리지(MongoDB)로 구성하여 속도와 용량의 최적화
- **LRU 캐시 교체 정책:** 가장 최근에 사용되지 않은 페이지를 자동으로 교체하여 캐시 효율성 극대화
- **페이지 폴트 처리:** 상위 계층에 데이터가 없을 때 하위 계층에서 자동으로 데이터를 가져와 투명하게 승급
- **의미론적 검색:** 벡터 임베딩을 활용한 의미 기반 검색으로 문맥 이해도 향상

## 2. 기술적 배경

### 2.1 OS 메모리 계층 구조

현대 운영체제는 속도와 용량의 트레이드오프를 해결하기 위해 다층 메모리 계층 구조를 사용합니다:

```
┌─────────────────────────────────────────────────────────┐
│                   메모리 계층 구조                       │
├─────────────────────────────────────────────────────────┤
│  L1 캐시    │  ~1ns   │  매우 빠름  │  KB 단위  │ 핫 데이터  │
│  L2 캐시    │  ~10ns  │  빠름      │  MB 단위  │  │
│  L3 캐시    │  ~50ns  │  중간      │  수십 MB  │  │
│  메인 메모리│  ~100ns │  느림      │  GB 단위  │  │
│  디스크     │  ~10ms  │  매우 느림 │  TB 단위  │ 콜드 데이터 │
└─────────────────────────────────────────────────────────┘
```

**핵심 원리:**
- **지역성(Locality):** 프로그램은 최근 접근한 데이터를 다시 접근하려는 경향(시간적 지역성)
- **계층형 이동:** 자주 접근하는 데이터는 빠른 상위 계층으로, 그렇지 않은 데이터는 하위 계층으로 이동
- **투명성:** 프로그램은 데이터가 어디에 있는지 알 필요 없음

### 2.2 페이징(Paging)과 가상 메모리

**페이징이란?**
- 물리 메모리를 고정 크기의 블록(페이지)으로 분할하여 관리하는 기법
- 논리 주소(Logical Address)와 물리 주소(Physical Address)의 매핑을 통해 메모리 관리

**핵심 개념:**

1. **페이지(Page):** 메모리 관리의 기본 단위 (일반적으로 4KB)
2. **페이지 테이블(Page Table):** 논리 주소를 물리 주소로 변환하는 매핑 테이블
3. **페이지 폴트(Page Fault):** 접근하려는 페이지가 메모리에 없을 때 발생하는 인터럽트
4. **페이지 교체(Page Replacement):** 메모리가 가득 찼을 때 어떤 페이지를 내릴지 결정

### 2.3 LRU (Least Recently Used) 캐시 교체 정책

**LRU 원리:**
- 가장 오랫동안 사용되지 않은 페이지를 교체 대상으로 선택
- 시간적 지역성(Temporal Locality) 원리에 기반
- 최근 접근한 페이지는 곧 다시 접근할 가능성이 높음 가정

**구현 방법:**
1. **해시맵 + 이중 연결 리스트:** O(1) 조회, 삽입, 삭제
2. **접근 순서 추적:** 최신 접근 데이터를 리스트 앞부분으로 이동
3. **용량 초과 시:** 리스트 미부(가장 오래된 데이터) 제거

### 2.4 LLM 에이전트와 메모리 관리

**LLM 컨텍스트 윈도우의 문제점:**
- 제한된 토큰 윈도우로 인한 맥락 손실
- 장기 대화에서 초기 정보 누락
- 복잡한 작업에서 정보 조각화

**기존 RAG 시스템의 한계:**
- 벡터 데이터베이스에만 의존한 단일 계층 구조
- 빈번하게 접근하는 데이터에 대한 최적화 부족
- 항상 전체 검색 수행으로 비용 효율성 저하

**OS 컨셉 적용의 장점:**
- 핫 데이터를 L1 캐시에서 즉시 검색 (1ms)
- 의미론적 검색은 L2 벡터 DB에서 수행 (~10ms)
- 장기 보관은 L3 MongoDB에서 담당 (~50ms)
- 전체 검색 비용과 응답 시간 획기적 개선

### 3.2 3계층 메모리 계층 구조

```
┌────────────────────────────────────────────────────────────────────┐
│                    AI Agent Application Layer                       │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│                 Hierarchical Memory Manager                        │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                       │
│  │    L1    │   │    L2    │   │    L3    │                       │
│  │  Redis   │◄──┤ ChromaDB │◄──┤ MongoDB  │                       │
│  │  ~1ms    │   │  ~10ms   │   │  ~50ms   │                       │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘                       │
│       │              │              │                               │
│       ▼              ▼              ▼                               │
│  Fast Cache    Vector Search    Long-term                          │
│  (Hot Data)    (Semantic)       Storage                            │
│  - 최근 접근    - 의미론적 검색   - 장기 보관                        │
│  - 자주 사용    - 임베딩 저장    - 백업                            │
└────────────────────────────────────────────────────────────────────┘
                               │
                               ▼
┌────────────────────────────────────────────────────────────────────┐
│              Ollama Embedding Service                              │
│          (nomic-embed-text model)                                  │
│      - 로컬 임베딩 생성                                            │
│      - 코사인 유사도 계산                                           │
└────────────────────────────────────────────────────────────────────┘
```

**L1 캐시 (Redis - 인메모리):**
- **목적:** 가장 빠른 액세스를 위한 핫 데이터 캐싱
- **접근 속도:** 약 1ms
- **용량:** 기본 100페이지 (구성 가능)
- **TTL:** 5분 (300초)
- **용도:** 최근 접근한 메모리 페이지의 즉시 검색

**L2 벡터 DB (ChromaDB):**
- **목적:** 의미론적 검색을 위한 벡터 임베딩 저장
- **접근 속도:** 약 10ms
- **용량:** 실제로 무제한 (디스크 기반)
- **용도:** 쿼리와 의미적으로 유사한 컨텍스트 검색

**L3 스토리지 (MongoDB):**
- **목적:** 장기 보관을 위한 영구 저장소
- **접근 속도:** 약 50ms
- **용량:** TB 단위 확장 가능
- **용도:** 모든 메모리 페이지의 영구 보관

### 3.3 LRU 캐시 구현

**데이터 구조:**
- **해시맵:** O(1) 키 조회
- **이중 연결 리스트:** O(1) 삽입, 삭제, 접근 순서 업데이트

**핵심 연산:**

1. **GET 연산:**
   ```
   1. 해시맵에서 키 조회
   2. 찾은 노드를 리스트 두부로 이동 (최신 접근 표시)
   3. 값 반환
   ```

2. **PUT 연산:**
   ```
   1. 이미 존재하는 키: 값 업데이트 후 리스트 두부로 이동
   2. 새로운 키:
      - 캐시가 가득 찼으면 리스트 미부 제거 (LRU 교체)
      - 새 노드를 리스트 두부에 삽입
      - 해시맵에 추가
   ```

3. **TTL(만료) 지원:**
   ```
   - 각 페이지에 마지막 접근 시간 저장
   - TTL 경과 시 자동 삭제
   - 주기적 만료 검사
   ```

### 3.4 페이지 폴트 처리

**페이지 폴트란?**
- 요청한 데이터가 L1, L2 캐시에 없어 L3 스토리지까지 내려가는 상황

**처리 흐름:**
```
GET 요청 → L1 캐시 확인
           ↓ (miss)
           L2 벡터 DB 확인
           ↓ (miss)
           L3 MongoDB 확인 (페이지 폴트)
           ↓ (hit)
           데이터를 L1, L2에 승급(promotion)
           데이터 반환
```

**투명한 승급(Promotion):**
- L3에서 데이터를 찾으면 자동으로 L2, L1에도 복사
- 다음 접근부터는 빠른 계층에서 검색 가능
- 사용자는 승급 과정을 인식하지 못함 (투명성)

### 3.5 의미론적 검색

**임베딩 생성:**
- Ollama nomic-embed-text 모델 사용
- 텍스트를 고정 차원 벡터(예: 768차원)로 변환
- 로컬에서 실행하여 API 비용 절감

**유사도 계산:**
- 코사인 유사도(Cosine Similarity) 사용
- 범위: -1(반대) ~ 1(동일), 일반적으로 0.7 이상이면 유사
- 쿼리 벡터와 가장 유사한 top-K 개 반환

**검색 프로세스:**
```
1. 사용자 쿼리 입력
2. Ollama로 쿼리 임베딩 생성
3. ChromaDB에서 벡터 유사도 검색
4. 가장 유사한 top-K 개 결과 반환
```

### 3.6 REST API 설계

**API 엔드포인트:**

| 메서드 | 엔드포인트 | 설명 |
|--------|-----------|------|
| GET | `/api/health` | 시스템 건강 상태 확인 |
| POST | `/api/memory/get` | 메모리 계층 구조에서 값 검색 |
| POST | `/api/memory/put` | 모든 계층에 값 저장 |
| DELETE | `/api/memory` | 모든 계층에서 값 삭제 |
| POST | `/api/memory/search` | 의미론적 검색 수행 |
| GET | `/api/stats` | 메모리 관리자 통계 조회 |
| POST | `/api/memory/clear` | 모든 메모리 초기화 |

**API 예시:**

```bash
# 값 저장
curl -X POST http://localhost:3001/api/memory/put \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "agent-001",
    "key": "conversation:123",
    "value": "사용자가 날씨에 대해 질문함"
  }'

# 값 검색
curl -X POST http://localhost:3001/api/memory/get \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "agent-001",
    "key": "conversation:123"
  }'

# 의미론적 검색
curl -X POST http://localhost:3001/api/memory/search \
  -H "Content-Type: application/json" \
  -d '{
    "agentId": "agent-001",
    "query": "날씨 토론",
    "topK": 5
  }'

# 통계 조회
curl http://localhost:3001/api/stats
```

### 3.7 도메인 모델

**MemoryPage (메모리 페이지):**
```typescript
{
  id: string,           // 고유 식별자
  agentId: string,      // 소속 에이전트 ID
  key: string,          // 키
  value: string,        // 값
  embedding: number[],  // 벡터 임베딩
  level: MemoryLevel,   // 현재 계층 (L1|L2|L3)
  status: PageStatus,   // 페이지 상태
  accessCount: number,  // 접근 횟수
  lastAccessed: Date,   // 마지막 접근 시간
  createdAt: Date,      // 생성 시간
  updatedAt: Date       // 수정 시간
}
```

**PageTableEntry (페이지 테이블 엔트리):**
```typescript
{
  pageNumber: number,   // 논리 페이지 번호
  frameNumber: number,  // 물리 프레임 번호
  present: boolean,     // 메모리 내 존재 여부
  referenced: boolean,  // 참조 비트
  modified: boolean,    // 수정 비트
  protection: string    // 접근 권한
}
```

### 3.8 멀티 에이전트 지원

- 각 에이전트(agentId)는 독립적인 메모리 공간 보유
- 에이전트 간 메모리 격리 보장
- Composite Key: `agentId:key` 형태로 메모리 페이지 식별
- 통계는 에이전트별로 개별 추적

## 6. 향후 계획

### 6.1 추가 기능 구현

**1. 다른 캐시 교체 정책:**
- **LFU (Least Frequently Used):** 접근 횟수 기반 교체
- **ARC (Adaptive Replacement Cache):** 적응형 교체 정책
- **CLOCK 알고리즘:** LRU의 근사치로 오버헤드 감소
- 목표: 각 정책의 성능 비교 연구

**2. 쓰기 지연(Write-back) 캐싱:**
- L3 쓰기를 배치로 지연 처리
- 쓰기 버퍼링으로 디스크 I/O 감소
- 더티 페이지(Dirty Page) 추적

**3. 프리페칭(Pre-fetching):**
- 접근 패턴 학습으로 예측적 페이지 로드
- 순차 접근 패턴 감지
- 의미론적 유사도 기반 프리페칭

**4. 데이터 압축:**
- 콜드 데이터의 L3 저장 전 압축
- 압축 알고리즘 비교(gzip, brotli, lz4)
- 압축률과 성능 트레이드오프 분석

### 6.2 성능 최적화
