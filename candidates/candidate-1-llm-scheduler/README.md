# LLM 추론 요청 스케줄러

> OS 스케줄링 알고리즘을 LLM API 요청 관리에 적용

## 왜 이 주제?

요즘 ChatGPT, Claude 같은 LLM API를 쓰는 서비스가 많아졌는데, 여러 사용자가 동시에 요청하면 비용도 폭증하고 응답도 느려진다. 근데 모든 요청을 똑같이 처리하면 급한 요청도 한참 기다려야 하는 문제가 생긴다.

OS 수업에서 배운 프로세스 스케줄링(MLFQ, Priority Queue 등)을 여기에 적용하면 어떨까?

## 핵심 아이디어

| OS 개념 | LLM에 적용하면 |
|---------|---------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 호출 권한 |
| 우선순위 | 사용자 등급, 요청 긴급도 |
| 스케줄링 알고리즘 | 요청 처리 순서 결정 |

## 구현할 것

1. **Express 프록시 서버**: 모든 LLM 요청이 여기를 거침
2. **요청 분류기**: 우선순위, 토큰 예산 분석
3. **스케줄러**: FCFS, Priority, MLFQ, WFQ 등 구현
4. **대시보드**: 실시간 대기열 상태, 메트릭 시각화

## 비교 실험

| 알고리즘 | 측정할 것 |
|----------|----------|
| FCFS | 기본 베이스라인 |
| Priority Queue | 평균 대기시간 개선 |
| MLFQ | 공정성 + 응답성 균형 |
| WFQ | 멀티테넌트 환경 |

## 기술 스택

```
Express.js, BullMQ (Redis), MongoDB
Ollama (로컬 LLM) 또는 OpenAI API
Socket.io (실시간 대시보드)
```

## 예상 일정

- 설계: 1주
- MVP (FCFS, Priority): 4주
- 고도화 (MLFQ, WFQ): 3주
- 실험 및 논문: 2주

## 왜 이게 좋은지

- OS 스케줄링 알고리즘을 **직접** 구현하고 비교 실험
- 정량적 성능 측정 가능 (처리량, 대기시간, 공정성)
- 논문 작성 가능: "LLM 추론 요청을 위한 적응형 스케줄링 알고리즘"
