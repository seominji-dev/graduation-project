## 1. 개요

### 1.1 OS 스케줄링과 LLM 스케줄링의 매핑

운영체제의 프로세스 스케줄링 개념을 LLM API 요청 관리에 적용하면 다음과 같이 매핑됩니다:

| OS 개념 | LLM 스케줄링 매핑 | 구현 세부사항 |
|---------|-------------------|---------------|
| 프로세스 (Process) | LLM API 요청 | LLMRequest 객체 |
| CPU | LLM 서비스 (Ollama/OpenAI) | LLMService 클래스 |
| CPU 시간 | API 호출 권한 | BullMQ Worker 처리 |
| 프로세스 상태 | 요청 상태 | RequestStatus enum |
| Ready Queue | 대기열 | BullMQ Queue |
| Context Switch | 요청 전환 | Worker job 완료 후 다음 job 처리 |
| Time Quantum | 처리 시간 제한 | MLFQ의 타임 슬라이스 |
| Priority | 요청 우선순위 | RequestPriority enum |

## 2. FCFS 알고리즘 상세 분석

### 2.1 알고리즘 원리

**First-Come, First-Served (FCFS)**는 가장 단순한 스케줄링 알고리즘으로, 요청이 도착한 순서대로 처리합니다.

### 2.2 시간 복잡도 분석

| 연산 | 시간 복잡도 | 설명 |
|------|-------------|------|
| 삽입 (Enqueue) | O(1) | 큐 끝에 추가 |
| 선택 (Dequeue) | O(1) | 큐 앞에서 제거 |
| 검색 (getStatus) | O(1) | jobId로 직접 접근 |
| 취소 (cancel) | O(1) | jobId로 직접 접근 |

**공간 복잡도**: O(n), n = 대기 중인 요청 수

### 2.3 장단점 분석

**장점:**
- 구현이 매우 간단함
- 예측 가능한 동작 (도착 순서 보장)
- 기아 현상 없음 (모든 요청이 결국 처리됨)
- 오버헤드 최소

**단점:**
- Convoy Effect (긴 요청이 뒤의 짧은 요청들을 블로킹)
- 평균 대기 시간이 길어질 수 있음
- 우선순위 처리 불가 (확장된 FCFS로 부분 해결)
- 응답성이 중요한 시스템에 부적합

## 3. Priority 알고리즘 상세 분석

### 3.1 알고리즘 원리

**Priority Scheduling**은 각 요청에 우선순위를 부여하고, 높은 우선순위의 요청을 먼저 처리합니다.

### 3.2 우선순위 매핑

| RequestPriority | BullMQ Priority | 처리 순서 |
|-----------------|-----------------|-----------|
| URGENT (3) | 0 | 1순위 |
| HIGH (2) | 2 | 2순위 |
| NORMAL (1) | 4 | 3순위 |
| LOW (0) | 6 | 4순위 |

### 3.3 에이징 메커니즘

기아 현상(Starvation)을 방지하기 위한 에이징 구현:

- 에이징 간격: 60초마다 검사
- 에이징 임계값: 2분 후 에이징 시작
- 최대 승격 횟수: 2단계

### 3.4 시간 복잡도 분석

| 연산 | 시간 복잡도 | 설명 |
|------|-------------|------|
| 삽입 | O(log n) | 힙 기반 우선순위 큐 삽입 |
| 선택 | O(1) | 최우선순위 요소 접근 |
| 에이징 검사 | O(n) | 모든 대기 작업 순회 |
| 우선순위 변경 | O(log n) | 힙 재구성 |

## 4. MLFQ 알고리즘 상세 분석

### 4.1 알고리즘 원리

**Multi-Level Feedback Queue (MLFQ)**는 여러 우선순위 큐를 사용하고, 작업의 실행 특성에 따라 동적으로 우선순위를 조정합니다.

### 4.2 OSTEP 5 Rules 구현

| 규칙 | 설명 | 구현 방식 |
|------|------|-----------|
| Rule 1 | Priority(A) > Priority(B)이면 A만 실행 | 큐 레벨 순회로 구현 |
| Rule 2 | Priority(A) = Priority(B)이면 라운드로빈 | BullMQ 기본 동작 활용 |
| Rule 3 | 새 작업은 Q0에서 시작 | submit()에서 Q0에 추가 |
| Rule 4 | 타임 슬라이스 소진 시 강등 | processJob()에서 시간 추적 |
| Rule 5 | 주기적 부스팅 | BoostManager 사용 (5초 간격) |

### 4.3 큐 구성

| 큐 | 타임 퀀텀 | 용도 |
|----|-----------|------|
| Q0 | 1,000ms | 새 작업, 부스팅된 작업 |
| Q1 | 3,000ms | Q0 퀀텀 초과 작업 |
| Q2 | 8,000ms | Q1 퀀텀 초과 작업 |
| Q3 | Unlimited | 장시간 실행 작업 (FCFS 동작) |

### 4.4 시간 복잡도 분석

| 연산 | 시간 복잡도 | 설명 |
|------|-------------|------|
| 삽입 | O(1) | Q0에 직접 추가 |
| 선택 | O(k) | k개 큐 순회 (k=4) |
| 강등 | O(1) | 다음 레벨 큐로 이동 |
| 부스팅 | O(n) | 모든 대기 작업 순회 |

## 5. WFQ 알고리즘 상세 분석

### 5.1 알고리즘 원리

**Weighted Fair Queuing (WFQ)**는 GPS(Generalized Processor Sharing)를 근사하여 테넌트별 가중치에 비례한 공정한 자원 할당을 제공합니다.

### 5.2 가상 시간 시스템

**가상 종료 시간 계산:**

```
F_i = max(V(t), F_i_prev) + (L_i / r_i)

여기서:
- F_i: 요청 i의 가상 종료 시간
- V(t): 현재 가상 시간
- F_i_prev: 테넌트 i의 이전 종료 시간
- L_i: 서비스 시간 (예상 처리 시간)
- r_i: 테넌트 가중치
```

### 5.3 테넌트 계층 및 가중치

| 계층 | 가중치 | 대역폭 비율 |
|------|--------|-------------|
| Enterprise | 100 | 100x |
| Premium | 50 | 50x |
| Standard | 10 | 10x |
| Free | 1 | 1x (기준) |

### 5.4 공정성 계산 (Jain's Fairness Index)

```
J = (sum x_i)^2 / (n * sum x_i^2)

여기서:
- x_i: 테넌트 i가 받은 서비스 시간
- n: 활성 테넌트 수
- J: 공정성 지수 (0~1, 1이 완벽한 공정성)
```

### 5.5 시간 복잡도 분석

| 연산 | 시간 복잡도 | 설명 |
|------|-------------|------|
| 삽입 | O(log n) | 가상 종료 시간 기준 정렬 삽입 |
| 선택 | O(1) | 최소 가상 종료 시간 요소 |
| 가상 시간 업데이트 | O(1) | 단순 산술 연산 |
| 공정성 계산 | O(t) | t = 테넌트 수 |

## 6. 성능 비교 분석

### 6.1 정량적 비교

| 메트릭 | FCFS | Priority | MLFQ | WFQ |
|--------|------|----------|------|-----|
| 평균 대기 시간 | 기준 | -30% | -40% | -35% |
| 처리량 | 기준 | +0% | +20% | +0% |
| 응답성 | 낮음 | 중간 | 높음 | 중간 |
| 공정성 | 중간 | 낮음 | 높음 | 매우 높음 |
| 구현 복잡도 | 낮음 | 중간 | 높음 | 높음 |
| 오버헤드 | 최소 | 낮음 | 중간 | 중간 |

### 6.2 워크로드별 권장

| 워크로드 | 권장 알고리즘 |
|----------|--------------|
| 짧은 요청 위주 | MLFQ |
| 혼합 워크로드 | MLFQ 또는 WFQ |
| 멀티테넌트 환경 | WFQ |
| 단순 배치 처리 | FCFS |

### 7.1 의사결정 매트릭스

| 사용 사례 | 권장 알고리즘 | 이유 |
