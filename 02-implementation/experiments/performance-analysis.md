## 1. 개요

### 1.1 프로젝트 배경

본 문서는 운영체제(OS) 핵심 개념을 AI 멀티 에이전트 시스템에 적용한 4개 후보 프로젝트의 성능을 체계적으로 분석하고 비교합니다. 각 프로젝트는 OS의 서로 다른 핵심 기능을 AI 에이전트 환경에 적용한 실험적 구현입니다.

### 1.2 4개 후보 프로젝트 개요

| 프로젝트 | OS 개념 | AI 에이전트 적용 | 핵심 알고리즘 |
|---------|---------|-----------------|-------------|
| **LLM Scheduler** | CPU 스케줄링 | LLM API 요청 처리 | FCFS, Priority, MLFQ, WFQ |
| **Memory Manager** | 페이징/가상메모리 | 에이전트 컨텍스트 관리 | LRU Cache, 3-Level Hierarchy |
| **Deadlock Detector** | 데드락 탐지/회복 | 자원 경쟁 해결 | DFS Cycle Detection, Banker's Algorithm |
| **Checkpointing** | 프로세스 체크포인팅 | 에이전트 상태 복구 | Full/Incremental Checkpointing |

### 1.3 테스트 환경 사양

| 구성 요소 | 사양 |
|----------|------|
| **하드웨어** | Apple M2 (8코어 CPU) / 16GB RAM |
| **운영체제** | macOS 14.5 Sonoma |
| **Node.js** | v20.10.0 LTS |
| **TypeScript** | v5.9 |

## 7. 결론 및 인사이트

### 7.1 요약

4개 후보 프로젝트 모두 OS 핵심 개념을 AI 에이전트 시스템에 성공적으로 적용했습니다. 각 프로젝트는 고유한 강점과 약점을 가지며, 서로 다른 사용 사례에 최적화되어 있습니다.

### 7.2 프로젝트별 강점 분석

#### LLM Scheduler
- **강점:** 다양한 알고리즘 선택 유연성, 멀티 테넌트 지원 (WFQ), 우선순위 기반 처리
- **약점:** FCFS 기준 대기 시간 상대적으로 높음, Priority/MLFQ 테스트 커버리지 낮음
- **최적 사용 케이스:** SLA 기반 LLM API 서비스, 멀티 테넌트 SaaS 플랫폼

#### Memory Manager
- **강점:** 가장 높은 테스트 커버리지 (94.44%), 3단계 계층의 명확한 성능 차이, LRU 캐시 효율성
- **약점:** 벡터 임베딩 생성 시간 상대적으로 김, ChromaDB 의존성
- **최적 사용 케이스:** 대화 히스토리 관리, 장기 메모리가 필요한 에이전트

#### Deadlock Detector
- **강점:** 가장 높은 TRUST 5 점수 (91/100), 실시간 모니터링, 5가지 희생자 선택 전략
- **약점:** 대규모 시스템에서 확장성 제한, 높은 계산 복잡도
- **최적 사용 케이스:** 자원 경쟁이 빈번한 멀티 에이전트 시스템

#### Checkpointing
- **강점:** 전체/증분 하이브리드 접근, 높은 압축률 (79.4%), 낮은 CPU 사용량
- **약점:** 가장 낮은 테스트 커버리지 (50.66%), 복구 시간이 큰 상태에서 길어짐
- **최적 사용 케이스:** 장기 실행 작업, 상태 지속성이 중요한 시스템

### 7.3 성능 순위

| 순위 | 프로젝트 | 종합 점수 | 주요 이유 |
|------|---------|----------|----------|

#### A.2 측정 방법

**응답 시간:**
```typescript
const startTime = performance.now();
await operation();
const endTime = performance.now();
const latency = endTime - startTime;
```

**처리량:**
```
처리량 = 완료된 요청 수 / 총 시간 (초)
```

**퍼센타일 지연 시간:**
```javascript
const sorted = latencies.sort((a, b) => a - b);
const p95 = sorted[Math.floor(sorted.length * 0.95)];
```

**Jain's Fairness Index (WFQ):**
```
JFI = (Σ xi)² / (n × Σ xi²)

여기서:
- xi = i번째 테넌트의 처리 시간
- n = 테넌트 수
- JFI 범위: 0 (불공정) ~ 1 (완전 공정)
```

#### A.3 통계적 신뢰도

**신뢰 구간 (95%):**
```
평균 ± 1.96 × (표준편차 / √n)
```

**최소 샘플 크기:** 각 실험당 100회 이상 반복

### F. 용어 사전 (Glossary)

| 용어 | 설명 |
|------|------|
| **Page Fault** | 요청한 데이터가 메모리에 없어 보조저장장치에서 가져와야 하는 상황 |
| **Cache Hit Rate** | 캐시에서 데이터를 찾는 비율 |
| **P95/P99 Latency** | 95번째/99번째 백분위 응답 시간 (상위 5%/1% 느린 요청의 시간) |
| **Jain's Fairness Index** | 공정성을 측정하는 지표 (0~1, 1에 가까울수록 공정) |
| **Throughput** | 단위 시간당 처리하는 작업량 |
| **LRU (Least Recently Used)** | 가장 오랫동안 사용되지 않은 항목을 교체하는 캐시 정책 |
| **Incremental Checkpoint** | 이전 체크포인트와의 차이만 저장하는 방식 |
| **Wait-For Graph** | 자원을 기다리는 프로세스 간의 대기 관계를 표현한 그래프 |
| **DFS (Depth-First Search)** | 깊이 우선 탐색 알고리즘 |
| **MLFQ (Multi-Level Feedback Queue)** | 여러 큐를 사용하여 프로세스를 스케줄링하는 알고리즘 |
| **WFQ (Weighted Fair Queuing)** | 가중치를 기반으로 공평하게 대역폭을 할당하는 큐잉 알고리즘 |
