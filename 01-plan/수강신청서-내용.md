## 3. 프로젝트 개요

### 3.1 연구 배경

ChatGPT, Claude 등 대규모 언어 모델(LLM) API의 사용이 급증하면서, 다중 사용자 환경에서 API 요청을 효율적으로 관리해야 하는 필요성이 대두되고 있습니다. 현재 많은 LLM 서비스는 기본적인 선착순(FCFS) 처리나 단순 Rate Limiting에 의존하고 있어, 멀티테넌트 환경에서 다음과 같은 문제가 발생할 수 있습니다:

- **응답 지연**: 긴 요청이 짧은 요청을 지연시키는 Convoy Effect(호위 효과)
- **공정성 부재**: 모든 사용자가 동일하게 취급되어 VIP 사용자 차별화 불가
- **비용 비효율**: 우선순위 없이 처리하여 자원 낭비

운영체제 수업에서 프로세스 스케줄링을 학습하며 "실제 시스템에서도 이러한 알고리즘이 효과적일까?"라는 연구 동기를 갖게 되었으며, 최근 주목받는 LLM API 관리 분야에 적용하고자 본 주제를 선정하였습니다.

### 3.2 해결 방안

운영체제 수업에서 학습한 CPU 스케줄링 알고리즘을 LLM API 요청 관리에 적용합니다:

1. **FCFS (First-Come, First-Served)**: 기본 선착순 처리 (베이스라인)
2. **Priority Scheduling**: 긴급 요청 우선 처리 + Aging(노화)으로 기아(Starvation) 방지
3. **MLFQ (Multi-Level Feedback Queue)**: 짧은 요청 최적화, 동적 우선순위
4. **WFQ (Weighted Fair Queuing)**: 사용자별 가중치 기반 공정한 자원 배분

### 3.3 OS 수업 연관성

| OS 개념 | 본 프로젝트 적용 |
|---------|----------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 응답 시간 |
| Ready Queue | 요청 대기열 |
| Context Switch | 요청 전환 |
| Starvation(기아 현상) | 장시간 대기 요청 |
| Aging(노화 기법) | 대기 시간에 따른 우선순위 상승 |

## 8. 알고리즘 설계

### 8.1 FCFS (First-Come, First-Served)

- **원리**: 먼저 도착한 요청을 먼저 처리합니다.
- **LLM 적용**: 모든 LLM 요청을 도착 순서대로 처리하는 베이스라인 알고리즘입니다.
- **장점**: 구현이 간단하고, 모든 요청을 동등하게 처리합니다.
- **단점**: Convoy Effect(호위 효과) — 긴 요청이 짧은 요청을 지연시킵니다.

### 8.2 Priority Scheduling

- **원리**: 우선순위가 높은 요청을 먼저 처리합니다.
- **LLM 적용**: 긴급한 LLM 요청(예: 실시간 고객 응대)을 일반 요청보다 먼저 처리합니다.
- **우선순위**: URGENT(3) > HIGH(2) > NORMAL(1) > LOW(0)
- **Aging(노화 기법)**: 5초 주기로 대기 중인 요청의 우선순위를 1단계 상승시켜 기아 현상을 방지합니다.

### 8.3 MLFQ (Multi-Level Feedback Queue)

- **원리**: 4개 큐(Q0-Q3)를 두고, 요청의 실행 시간에 따라 큐를 이동시킵니다.
- **LLM 적용**: 짧은 LLM 요청(간단한 질문)은 높은 큐에서 빠르게 처리하고, 긴 요청(문서 요약 등)은 낮은 큐로 이동시켜 짧은 요청의 응답성을 보장합니다.
- **Time Quantum**: Q0(1초) < Q1(3초) < Q2(8초) < Q3(무제한)
- **Boost**: 30초 주기로 모든 요청을 Q0으로 재배치하여 장기 대기를 방지합니다.

### 8.4 WFQ (Weighted Fair Queuing)

- **원리**: 테넌트별 가중치에 비례하여 자원을 배분합니다.
- **LLM 적용**: 유료 테넌트(Enterprise)의 LLM 요청이 무료 테넌트(Free)보다 가중치에 비례하여 빠르게 처리되도록 합니다.
- **가중치**: Enterprise(100) > Premium(50) > Standard(10) > Free(1)
- **공정성 측정**: Jain's Fairness Index를 활용하여 가중치 대비 서비스 차등화를 정량적으로 평가합니다.

## 9. 예상 성과

### 9.1 연구 질문 (Research Questions)

본 프로젝트는 사전에 목표 수치를 정하지 않고, 실험 결과를 관찰하여 분석하는 **탐구적 연구(exploratory research)** 방식으로 진행하며, 다음 연구 질문에 답하고자 합니다:

| 연구 질문 | 알고리즘 | 핵심 탐구 내용 |
|----------|----------|---------------|
| RQ1 | Priority | 긴급(URGENT) 요청은 낮은 우선순위 요청보다 얼마나 빠르게 처리되는가? |
| RQ2 | MLFQ | 다양한 길이의 작업이 혼재된 환경에서 MLFQ는 어떤 적응성을 보이는가? |
| RQ3 | WFQ | 가중치에 비례하는 서비스 차등화를 달성하는가? |

### 9.2 예상되는 결과

며칠간 간단한 프로토타입을 만들어 사전 테스트해 본 결과, FCFS 대비 Priority 스케줄링에서 URGENT 요청의 평균 대기시간이 약 81% 감소하는 양상을 확인하였습니다 (간이 프로토타입 결과이며, 본격적인 실험에서는 조건에 따라 달라질 수 있습니다). 이를 바탕으로 다음과 같은 결과를 기대합니다:

- **Priority Scheduling**: URGENT 요청이 FCFS 대비 의미 있게 빠르게 처리될 것으로 예상 (프로토타입에서 확인 완료)
- **MLFQ**: 짧은 요청과 긴 요청이 혼재된 환경에서 적응적 성능을 보일 것으로 예상
- **WFQ**: 테넌트 가중치에 비례하는 서비스 차등화가 관찰될 것으로 예상

### 9.3 품질 목표

| 항목 | 목표 |
|------|------|
| 테스트 커버리지 | 85%+ |
| 테스트 통과율 | 100% |
| 코드 라인 | 스케줄러 ~460줄, 전체 ~1,000줄 |

### 9.4 프로젝트 한계점

- 본 프로젝트는 **로컬 환경에서의 개념 증명(PoC)** 수준으로, 실제 대규모 트래픽 환경에서의 검증은 포함하지 않습니다.
- Ollama 로컬 LLM 사용으로, 실제 클라우드 LLM API(OpenAI 등)와 응답 특성이 다를 수 있습니다.
- 단일 서버 환경에서의 실험이므로, 분산 환경에서의 확장성은 향후 과제로 남깁니다.
