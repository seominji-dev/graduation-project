## 3. 프로젝트 개요

### 3.1 연구 배경

ChatGPT, Claude 등 대규모 언어 모델(LLM) API의 사용이 급증하면서, 다중 사용자 환경에서 API 요청을 효율적으로 관리해야 하는 필요성이 대두되고 있습니다. 현재 대부분의 LLM 서비스는 단순한 선착순(FCFS) 방식으로 요청을 처리하여 다음과 같은 문제가 발생합니다:

- **응답 지연**: 긴 요청이 짧은 요청을 지연시키는 Convoy Effect
- **공정성 부재**: 모든 사용자가 동일하게 취급되어 VIP 사용자 차별화 불가
- **비용 비효율**: 우선순위 없이 처리하여 자원 낭비

### 3.2 해결 방안

운영체제 수업에서 학습한 CPU 스케줄링 알고리즘을 LLM API 요청 관리에 적용합니다:

1. **FCFS (First-Come, First-Served)**: 기본 선착순 처리 (베이스라인)
2. **Priority Scheduling**: 긴급 요청 우선 처리 + Aging으로 기아 방지
3. **MLFQ (Multi-Level Feedback Queue)**: 짧은 요청 최적화, 동적 우선순위
4. **WFQ (Weighted Fair Queuing)**: 사용자별 가중치 기반 공정한 자원 배분

### 3.3 OS 수업 연관성

| OS 개념 | 본 프로젝트 적용 |
|---------|----------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 응답 시간 |
| Ready Queue | 요청 대기열 |
| Context Switch | 요청 전환 |
| Starvation | 장시간 대기 요청 |
| Aging | 대기 시간에 따른 우선순위 상승 |

## 8. 알고리즘 설계

### 8.1 FCFS (First-Come, First-Served)

- **원리**: 먼저 도착한 요청을 먼저 처리
- **장점**: 구현 간단, 공정함
- **단점**: Convoy Effect (긴 요청이 짧은 요청 지연)

### 8.2 Priority Scheduling

- **원리**: 우선순위가 높은 요청 먼저 처리
- **우선순위**: URGENT(3) > HIGH(2) > NORMAL(1) > LOW(0)
- **Aging**: 5초 주기로 대기 중인 요청의 우선순위 1단계 상승 (기아 방지)

### 8.3 MLFQ (Multi-Level Feedback Queue)

- **원리**: 4개 큐(Q0-Q3), 요청 행동에 따라 큐 이동
- **Time Quantum**: Q0(1초) < Q1(3초) < Q2(8초) < Q3(무제한)
- **MLFQ 5규칙**:
  1. 높은 큐 우선 실행
  2. Time Quantum 초과 시 강등
  3. 새 요청은 Q0 시작
  4. 주기적 Boost (30초)
  5. Aging 적용

### 8.4 WFQ (Weighted Fair Queuing)

- **원리**: 테넌트별 가중치에 비례하여 자원 배분
- **가중치**: Enterprise(100) > Premium(50) > Standard(10) > Free(1)
- **공정성**: 가중치에 비례하는 서비스 차등화 달성 여부로 평가
