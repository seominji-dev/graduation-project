# 멀티테넌트 LLM 게이트웨이 스케줄러 프로젝트 제안서

**프로젝트명**: OS 스케줄링 알고리즘을 활용한 멀티테넌트 LLM 게이트웨이 공정 요청 관리 시스템

**소속**: 홍익대학교 C235180 서민지

**학술년도**: 2026년 졸업프로젝트

---

## 1. 서론 (Introduction)

현대 AI 서비스에서 ChatGPT, Claude와 같은 대규모 언어 모델(LLM) API 사용이 급증하고 있습니다. 이에 따라 **여러 사용자(테넌트)에게 LLM 서비스를 제공하는 멀티테넌트 LLM 게이트웨이**의 필요성이 증가하고 있습니다. 그러나 이러한 환경에서 요청을 효율적으로 관리하지 못하면 특정 테넌트의 자원 독점, 낮은 등급 테넌트의 기아 현상, 서비스 품질 저하 등의 문제가 발생합니다.

본 프로젝트는 운영체제의 검증된 프로세스 스케줄링 알고리즘을 멀티테넌트 LLM 게이트웨이에 적용하여, **테넌트 간 공정한 자원 분배**와 **차등화된 서비스 품질(QoS)**을 동시에 달성하는 시스템을 구현합니다.

## 2. 문제 정의 (Problem Statement)

### 2.1 멀티테넌트 LLM 게이트웨이의 문제점

**여러 테넌트가 공유하는 LLM 게이트웨이에서 발생하는 4가지 핵심 문제:**

1. **자원 독점 (Resource Monopolization)**
   - Enterprise 테넌트가 대량 요청으로 자원 독점
   - 다른 테넌트의 서비스 품질 저하
   - 공정한 자원 배분 메커니즘 부재

2. **기아 현상 (Starvation)**
   - 낮은 등급(Free/Standard) 테넌트의 무기한 대기
   - 높은 등급 테넌트의 요청만 계속 처리
   - 서비스 약속(SLA) 위반

3. **차등 서비스 부재**
   - 모든 테넌트 요청을 동등하게 처리
   - 유료 테넌트와 무료 테넌트 구분 없음
   - 비즈니스 모델 지원 불가

4. **공정성 측정 불가**
   - 공정성을 정량적으로 측정할 방법 부재
   - 서비스 품질 보장 입증 불가능
   - 테넌트 불만 대응 어려움

### 2.2 실제 문제 시나리오

**시나리오 예시: 멀티테넌트 LLM 게이트웨이**
- 테넌트 A (Enterprise): 대량 분석 작업 100건 (가중치: 100)
- 테넌트 B (Premium): 실시간 고객 응대 (가중치: 50)
- 테넌트 C (Free): 개인 학습 용도 (가중치: 1)

**기존 시스템의 문제:**
- 도착 순서대로 처리 (FIFO) 또는 단순 우선순위
- Enterprise 테넌트가 자원 독점, Free 테넌트는 무기한 대기
- 결과: SLA 위반, 테넌트 이탈, 서비스 신뢰도 하락

## 3. 해결 방안 (Proposed Solution)

### 3.1 핵심 아이디어

운영체제의 프로세스 스케줄링 문제와 LLM API 요청 관리 문제의 본질적 유사성:

| OS 개념 | LLM 시스템 적용 |
|---------|----------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 호출 권한 |
| 우선순위 | 사용자 등급, 요청 긴급도 |
| 스케줄링 알고리즘 | 요청 처리 순서 결정 |

### 3.2 구현할 4가지 스케줄링 알고리즘

**1. FCFS (First-Come, First-Served)**
- 도착 순서대로 처리
- 베이스라인 알고리즘
- 시간 복잡도: O(1)

**2. Priority Scheduling**
- 우선순위 기반 처리 (URGENT > HIGH > NORMAL > LOW)
- Aging 메커니즘으로 기아(Starvation) 방지
- 선점형 구현

**3. MLFQ (Multi-Level Feedback Queue)**
- 4단계 큐 (Q0: 1초, Q1: 3초, Q2: 8초, Q3: 무제한)
- 5가지 MLFQ 규칙 구현
- Boost 메커니즘으로 응답성 보장

**4. WFQ (Weighted Fair Queuing)**
- 가중치 기반 공정 스케줄링
- Virtual Time 개념으로 GPS 근사
- Jain's Fairness Index로 공정성 측정

### 3.3 기술적 접근

**기술 스택:**
- **런타임:** Node.js 20 LTS
- **언어:** JavaScript (ES2024)
- **웹 프레임워크:** Express.js 4.18
- **큐 시스템:** 메모리 배열 기반 큐
- **데이터 저장:** JSON 파일
- **LLM 통합:** Ollama (로컬)

**시스템 아키텍처:**
- API Layer: REST API (폴링 방식)
- Scheduler Engine: 4가지 알고리즘 구현
- Storage Layer: SQLite

## 4. 연구 목표 (Research Objectives)

### 4.1 개발 목표

1. **알고리즘 구현:** 4가지 OS 스케줄링 알고리즘의 멀티테넌트 LLM 게이트웨이 적용
2. **공정성 분석:** 알고리즘별 Jain's Fairness Index 측정 및 비교
3. **차등 서비스:** 테넌트 등급(Enterprise/Premium/Standard/Free)별 차등 QoS 구현
4. **품질 보장:** 85% 이상 테스트 커버리지
5. **기아 방지:** Aging/Boosting 메커니즘으로 낮은 등급 테넌트 보호

### 4.2 연구 질문 (Research Questions)

본 프로젝트는 다음 연구 질문에 답하고자 한다:

**RQ1. Priority Scheduling의 효과**
- 긴급(URGENT) 요청은 낮은 우선순위 요청보다 얼마나 빠르게 처리되는가?
- Aging 메커니즘은 기아 현상을 효과적으로 방지하는가?

**RQ2. MLFQ의 적응성**
- MLFQ는 다양한 길이의 작업이 혼재된 환경에서 어떤 성능을 보이는가?
- Boost 메커니즘은 장기 대기 요청의 응답성을 개선하는가?

**RQ3. WFQ의 차등 서비스**
- WFQ는 가중치에 비례하는 서비스 차등화를 달성하는가?
- 높은 가중치 테넌트(Enterprise)와 낮은 가중치 테넌트(Free)의 대기시간 비율은 어떠한가?

### 4.3 품질 기준 (Quality Criteria)

| 기준 | 목표 | 측정 방법 |
|------|------|----------|
| 알고리즘 구현 | 4개 모두 | FCFS, Priority, MLFQ, WFQ |
| 테스트 커버리지 | 85%+ | Jest 커버리지 리포트 |
| 테스트 통과율 | 100% | npm test 실행 |
| 코드 품질 | 학부생 수준 | 함수 30줄 이하, 의존성 4개 이하 |

**참고**: 성능 수치(대기시간 개선율, 공정성 지수 등)는 실험 결과로 측정하며, 사전에 특정 수치를 목표로 설정하지 않는다. 이는 탐구적 연구(exploratory research)의 특성을 반영한다.

## 5. 기대 효과 (Expected Outcomes)

### 5.1 학술적 기여

1. **OS 이론의 멀티테넌트 시스템 응용 (차별화)**
   - 기존 연구: LLM API 관리에 단순 Rate Limiting 사용
   - 본 연구: CPU 스케줄링의 검증된 알고리즘(FCFS, Priority, MLFQ, WFQ)을 멀티테넌트 LLM 게이트웨이에 적용
   - 차별점: 테넌트 간 공정성 보장 및 기아 방지 메커니즘 통합

2. **멀티테넌트 환경 특화 메커니즘**
   - WFQ + TenantRegistry: 테넌트 등급별 가중치 기반 공정 분배
   - Aging: 낮은 등급 테넌트의 기아(Starvation) 방지
   - Jain's Fairness Index: 공정성의 정량적 측정 및 입증

3. **정량적 공정성 분석**
   - 다양한 테넌트 구성에서의 공정성 측정
   - 알고리즘별 Jain's Fairness Index 비교
   - 기아 현상 발생 여부 실증 분석

4. **통합 프레임워크 제공**
   - 4가지 알고리즘을 런타임에 교체 가능한 통합 시스템
   - REST API 및 실시간 모니터링 대시보드

5. **오픈소스 기여**
   - 완전한 소스코드 공개 (MIT 라이선스)
   - 재현 가능한 연구 결과 제공
   - LLM 게이트웨이 구축 시 즉시 적용 가능

### 5.2 실용적 가치

1. **멀티테넌트 LLM 서비스 지원**
   - 공정성 보장: WFQ 기반 테넌트 간 자원 분배
   - 차등 서비스: 테넌트 등급별 QoS 제공
   - SLA 준수: 기아 방지로 서비스 약속 이행

2. **LLM 게이트웨이 구축 지원**
   - 즉시 통합 가능한 REST API
   - 실시간 공정성 모니터링 대시보드
   - 다양한 LLM 제공자 지원

3. **확장 가능성**
   - 분산 스케줄링 지원 (향후 계획)
   - 적응형 알고리즘 선택 (향후 계획)

## 9. 실험 결과 요약 (Experimental Results Summary)

본 프로젝트의 실험 결과는 다음과 같다:

### 9.1 품질 기준 달성

| 기준 | 목표 | 결과 | 상태 |
|------|------|------|------|
| 알고리즘 구현 | 4개 | 4개 (FCFS, Priority, MLFQ, WFQ) | 달성 |
| 테스트 통과율 | 100% | 41/41 (100%) | 달성 |
| 테스트 커버리지 | 85%+ | 98.72% | 초과 달성 |
| 코드 품질 | 학부생 수준 | 3개 의존성, 함수 30줄 이하 | 달성 |

### 9.2 연구 질문에 대한 답변

**RQ1. Priority Scheduling의 효과**
- URGENT 요청: FCFS 대비 **62% 빠른 처리** (2,971ms → 1,122ms)
- LOW 요청: FCFS 대비 16% 느린 처리 (예상된 트레이드오프)
- Aging 메커니즘: 구현 완료, 기아 방지 효과 확인

**RQ2. MLFQ의 적응성**
- 현재 실험(단기 작업만): FCFS와 동일한 성능
- 분석: 모든 요청이 Q0 타임 슬라이스(1초) 내 완료되어 MLFQ의 피드백 메커니즘이 발동되지 않음
- 향후 과제: 장기 작업 포함 실험으로 MLFQ 효과 검증 필요

**RQ3. WFQ의 차등 서비스**
- Enterprise (weight: 100): 849ms
- Free (weight: 1): 4,894ms
- **Enterprise가 Free보다 5.8배 빠른 응답** (가중치 비례 차등 서비스 성공)
- Jain's Fairness Index: 0.32 (가중치 차이로 인한 의도된 불균형)
