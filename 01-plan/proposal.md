# LLM 스케줄러 프로젝트 제안서

**프로젝트명**: OS 스케줄링 알고리즘을 활용한 LLM API 요청 최적화 스케줄러

**소속**: 홍익대학교 C235180 서민지

**학술년도**: 2025년 졸업프로젝트

---

## 1. 서론 (Introduction)

현대 AI 서비스에서 ChatGPT, Claude와 같은 대규모 언어 모델(LLM) API 사용이 급증하고 있습니다. 그러나 다중 사용자 환경에서 LLM API 요청을 효율적으로 관리하지 못하면 비용 폭증, 응답 지연, 자원 낭비 등의 문제가 발생합니다.

본 프로젝트는 운영체제의 검증된 프로세스 스케줄링 알고리즘을 LLM API 요청 관리에 적용하여, 효율성과 공정성을 동시에 달성하는 시스템을 구현합니다.

---

## 2. 문제 정의 (Problem Statement)

### 2.1 현재 LLM API 환경의 문제점

**다중 사용자 환경에서 발생하는 4가지 핵심 문제:**

1. **비용 폭증**
   - 무분별한 API 요청으로 인한 예산 초과
   - 대량 작업과 긴급 작업의 구분 없음
   - 비용 관리 메커니즘 부재

2. **응답 지연**
   - 대기열 관리 부재로 불규칙한 응답 시간
   - 긴급한 요청도 순서대로 대기
   - 사용자 경험 저하

3. **공정성 부재**
   - 모든 요청을 동등하게 처리
   - 중요도나 사용자 등급 고려 없음
   - 멀티테넌트 환경에서 자원 배분 불공정

4. **자원 낭비**
   - 우선순위 없는 비효율적 처리
   - 시스템 자원 활용도 저하
   - 처리량 최적화 부재

### 2.2 실제 문제 시나리오

**시나리오 예시:**
- 사용자 A: 긴급 고객 문의 응답 (우선순위: URGENT)
- 사용자 B: 대량 데이터 분석 100건 (우선순위: LOW)
- 사용자 C: 일반 채팅 (우선순위: NORMAL)

**기존 시스템의 문제:**
- 도착 순서대로 처리 (FIFO)
- 긴급 요청 A가 대량 작업 B 뒤에서 장시간 대기
- 결과: 고객 불만, 서비스 품질 저하, 비즈니스 손실

## 3. 해결 방안 (Proposed Solution)

### 3.1 핵심 아이디어

운영체제의 프로세스 스케줄링 문제와 LLM API 요청 관리 문제의 본질적 유사성:

| OS 개념 | LLM 시스템 적용 |
|---------|----------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 호출 권한 |
| 우선순위 | 사용자 등급, 요청 긴급도 |
| 스케줄링 알고리즘 | 요청 처리 순서 결정 |

### 3.2 구현할 4가지 스케줄링 알고리즘

**1. FCFS (First-Come, First-Served)**
- 도착 순서대로 처리
- 베이스라인 알고리즘
- 시간 복잡도: O(1)

**2. Priority Scheduling**
- 우선순위 기반 처리 (URGENT > HIGH > NORMAL > LOW)
- Aging 메커니즘으로 기아(Starvation) 방지
- 선점형 구현

**3. MLFQ (Multi-Level Feedback Queue)**
- 4단계 큐 (Q0: 1초, Q1: 3초, Q2: 8초, Q3: 무제한)
- 5가지 MLFQ 규칙 구현
- Boost 메커니즘으로 응답성 보장

**4. WFQ (Weighted Fair Queuing)**
- 가중치 기반 공정 스케줄링
- Virtual Time 개념으로 GPS 근사
- Jain's Fairness Index로 공정성 측정

### 3.3 기술적 접근

**기술 스택:**
- **런타임:** Node.js 20 LTS
- **언어:** TypeScript 5.9
- **웹 프레임워크:** Express.js 4.18
- **큐 시스템:** BullMQ 5.1 (Redis 기반)
- **데이터베이스:** MongoDB 8.0
- **LLM 통합:** Ollama (로컬), OpenAI API (클라우드)

**시스템 아키텍처:**
- API Layer: REST API + Socket.IO
- Scheduler Engine: 4가지 알고리즘 구현
- Storage Layer: Redis + MongoDB

## 5. 기대 효과 (Expected Outcomes)

### 5.1 학술적 기여

1. **OS 이론의 AI 시스템 응용 (차별화)**
   - 기존 연구: LLM API 관리에 단순 Rate Limiting 사용
   - 본 연구: CPU 스케줄링의 검증된 알고리즘(FCFS, Priority, MLFQ, WFQ)을 LLM 환경에 최초 적용
   - 차별점: Aging, Boost, Virtual Time 등 고급 메커니즘 통합

2. **LLM 환경 특화 메커니즘**
   - Aging: API 요청의 기아(Starvation) 방지
   - Boost: 장시간 대기 요청의 응답성 보장
   - Virtual Time: 멀티테넌트 환경에서 공정성 측정

3. **정량적 비교 분석**
   - 실제 LLM 환경에서의 성능 측정
   - 알고리즘별 장단점 실증 분석
   - Jain's Fairness Index를 통한 공정성 정량화

4. **통합 프레임워크 제공**
   - 4가지 알고리즘을 런타임에 교체 가능한 통합 시스템
   - REST API 및 실시간 모니터링 대시보드

5. **오픈소스 기여**
   - 완전한 소스코드 공개 (MIT 라이선스)
   - 재현 가능한 연구 결과 제공
   - 산업계 즉시 적용 가능

### 5.2 실용적 가치

1. **AI 서비스 최적화**
   - 비용 절감: 효율적인 API 호출 관리
   - 응답 개선: 우선순위 기반 처리
   - 공정성 보장: 멀티테넌트 환경 지원

2. **산업계 적용 가능성**
   - 즉시 통합 가능한 REST API
   - 실시간 모니터링 대시보드
   - 다양한 LLM 제공자 지원

3. **확장 가능성**
   - 분산 스케줄링 지원 (향후 계획)
   - 적응형 알고리즘 선택 (향후 계획)
   - 추가 LLM 제공자 통합 용이
