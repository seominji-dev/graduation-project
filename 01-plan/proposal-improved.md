# 프로젝트 제안서 (개선안)
# Improved Project Proposal

**프로젝트명**: OS 스케줄링 알고리즘을 활용한 다중 사용자 LLM API 요청 관리 시스템

**소속**: 홍익대학교 컴퓨터공학과 C235180 서민지

**학술년도**: 2026년 졸업프로젝트

**버전**: 2.0 (개선안)
**개선일**: 2026년 2월 10일

---

## 1. 서론

### 1.1 배경 및 동기

ChatGPT, Claude 등 대규모 언어 모델(LLM) API 사용이 빠르게 늘고 있습니다. 여러 사용자가 동시에 LLM API를 호출하는 환경에서는 요청을 어떤 순서로 처리할지가 중요한 문제인데, 현재 많은 LLM 서비스는 선착순(FCFS) 처리나 단순 Rate Limiting에 의존하고 있습니다. 이 경우 긴 요청이 짧은 요청을 지연시키는 호위 효과(Convoy Effect)가 발생하거나, 긴급한 요청도 순서를 기다려야 하는 문제가 생길 수 있습니다.

3학년 운영체제 수업에서 프로세스 스케줄링을 배우면서 "이 알고리즘들을 실제 시스템에 적용하면 어떨까?"라는 궁금증이 생겼고, 이를 LLM API 요청 관리에 적용해 보면 재미있겠다고 생각하여 이 주제를 선택했습니다.

**[개선됨]** 이 프로젝트는 단순한 알고리즘 구현을 넘어, **공정성(Fairness)**을 정량적으로 측정하고 보장하는 방법론을 제시합니다. 기존 연구들이 처리량 최적화에 집중한 것과 달리, 본 연구는 테넌트 간 공정한 자원 분배를 **Jain's Fairness Index**로 측정하여 Provider 관점에서의 공정성을 실증적으로 분석합니다.

## 2. 문제 정의

### 2.1 기술적 문제

다중 사용자 LLM API 환경에서 발생하는 문제:

1. **호위 효과 (Convoy Effect)**: 긴 요청이 뒤의 짧은 요청을 지연시킴
2. **긴급 요청 대기**: 긴급한 요청도 도착 순서를 기다려야 함
3. **공정성 부재**: 사용자 등급별 차등 서비스 제공 불가
4. **자원 분배**: 공정한 자원 배분 메커니즘 부재
5. **[개선됨] 공정성 측정 부재**: 기존 시스템은 공정성을 정량적으로 측정하지 않음

### 2.2 해결 방안

운영체제의 CPU 스케줄링 알고리즘을 LLM API 요청 관리에 적용합니다.

| 알고리즘 | 설명 | OS 수업 연관 개념 |
|----------|------|------------------|
| FCFS | 선착순 처리 (베이스라인) | 기본 스케줄링 |
| Priority | 긴급 요청 우선 처리 + Aging으로 기아 방지 | 우선순위 스케줄링, 노화 기법 |
| MLFQ | [개선됨] 시간 슬라이스 기반 선점형 + 비선점형 하이브리드 | 다단계 피드백 큐 + 선점형 스케줄링 |
| WFQ | 테넌트 가중치 기반 공정 배분 + 이중 수준 JFI 측정 | Fair-Share Scheduling을 네트워크의 공정 큐잉 방식으로 적용 |

## 3. 목표 및 기대효과

### 3.1 개발 목표

| 목표 | 상세 내용 | 측정 방법 |
|------|----------|----------|
| 알고리즘 구현 | 5가지 스케줄링 알고리즘 구현 (FCFS, Priority, MLFQ, WFQ, RateLimiter) | 단위 테스트 통과 |
| REST API | 요청 등록, 조회, 통계 등 핵심 API | curl/Postman 테스트 |
| LLM 연동 | Ollama 로컬 LLM 연동 | 실제 응답 확인 |
| 성능 비교 | 알고리즘별 성능 측정 및 분석 | 실험 데이터 분석 |
| **[개선됨] 통계 검증** | Power Analysis, Effect Size, 신뢰구간 계산 | p-value, Cohen's d, 95% CI |
| **[개선됨] 공정성 측정** | 이중 수준 JFI (시스템/테넌트) | Jain's Fairness Index |

### 3.2 기대효과

**배울 수 있는 것**: OS 수업에서 이론으로만 배웠던 스케줄링 알고리즘을 직접 구현하고 실험해 봄으로써 이론과 실제의 차이를 체감할 수 있습니다. 특히 **MLFQ의 선점형 동작**과 **WFQ의 가중치 기반 공정성**을 실제로 구현하면서 OS 이론의 실제 적용 가능성을 확인할 수 있습니다. 또한 Node.js/Express 기반 백엔드 개발 경험과 **통계적 실험 설계 및 데이터 분석 능력**도 기를 수 있습니다.

**프로젝트의 의미**: OS 스케줄링 알고리즘을 AI 시스템에 적용해 보는 시도로, 4가지 알고리즘의 성능을 정량적으로 비교합니다. 특히 WFQ의 공정성을 객관적으로 측정할 방법을 찾아보면서 **이중 수준 JFI 측정 방법론**을 개발했습니다. 이는 Provider가 "의도적으로 불공정한(가중치 기반)" 시스템이 제대로 작동하는지 실시간으로 모니터링할 수 있게 해줍니다.

**[개선됨] 학술적 기여**:
1. **이중 수준 JFI 측정**: 시스템 수준(0.32)과 테넌트 수준(0.92-0.98)을 분리 측정
2. **MLFQ 선점형 구현**: 시간 슬라이스 기반 선점으로 짧은 요청 81% 개선
3. **통계적 엄밀함**: Power Analysis, Cohen's d, 95% 신뢰구간 적용

**[개선됨] 기술 스택 선정 원칙**:
1. **의존성 최소화**: 2개 패키지(express, jest)만 사용
2. **학부생 수준 유지**: TypeScript 대신 JavaScript, 복잡한 패턴 피함
3. **단순한 데이터 저장**: MongoDB 대신 JSON 파일
4. **메모리 큐**: Redis/BullMQ 대신 메모리 배열

## 6. 시스템 구조

시스템은 **클라이언트 → REST API → 스케줄러 매니저 → 메모리 큐/JSON 저장소 → Ollama LLM**의 흐름으로 구성됩니다. OS에서 프로세스가 Ready Queue에서 대기하다가 CPU를 할당받는 것처럼, LLM 요청도 대기열에서 순서를 기다리는 구조입니다.

핵심 모듈과 [개선된] 실제 코드량은 다음과 같습니다.

| 모듈 | 설명 | [개선됨] 실제 코드량 |
|------|------|---------------------|
| 스케줄러 (5종) | FCFS, Priority+Aging, MLFQ(선점형+비선점형), WFQ, RateLimiter | ~700줄 |
| 기타 모듈 | 큐, 저장소, API, LLM 클라이언트 | ~800줄 |
| **전체 합계** | | **~1,500줄** (+50% 증가) |

**[개선됨] 코드량 증가 이유**:
1. MLFQ 선점형 기능 추가 (시간 슬라이스, 선점 이벤트)
2. Rate Limiter (토큰 버킷 알고리즘)
3. 테스트 코드 (137개 테스트)
4. 통계 검증 코드 (JFI, Power Analysis)

## 8. 한계점

이 프로젝트는 로컬 환경에서 알고리즘의 동작을 확인하는 수준이며, 대규모 트래픽이나 분산 환경에서의 검증까지는 다루지 않습니다.

**[개선됨] 인정된 한계점**:
1. **실험 규모**: 100-500개 요청은 프로덕션 환경(수천~수만 개)의 대표성이 제한적
2. **단일 서버**: 분산 환경, 로드 밸런싱 미고려
3. **합성 데이터**: 실제 LLM 요청 패턴과의 차이 가능성

**[개선됨] 향후 연구 방향**:
1. 대규모 실험: 수천 개 요청, 수십 개 테넌트 환경
2. 분산 환경 확장: 다중 서버 간 상태 동기화
3. 동적 가중치: 부하 기반 가중치 자동 조정
4. 실제 LLM 연동: OpenAI API, Claude API 통합

**[개선됨] 구체적 AI 활용 범위**:
1. **코드 작성 보조**: GitHub Copilot을 활용한 보일러플레이트 코드 생성
2. **문서 작성**: 초안 작성 후 직접 검토 및 수정
3. **실험 설계**: 통계적 방법론 검색 및 참고
4. **직접 구현**: 모든 핵심 알고리즘은 직접 작성 및 검증
