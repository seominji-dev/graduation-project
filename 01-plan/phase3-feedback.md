## 2. Phase 1 계획서 수정 제안

### 2.1 테스트 목표 조정

**현재 계획서 (proposal.md)**:
> 테스트 커버리지 (Lines): 85%+
> 테스트 통과율: 100%

**수정 제안**:
- 단순화된 구현으로 인해 테스트 수가 777개 → 41개로 감소
- 그러나 핵심 기능은 모두 테스트됨
- **제안**: "핵심 기능 100% 테스트 커버리지" 또는 "모든 스케줄러 알고리즘 테스트"로 변경

### 2.2 공정성 지수 측정 방식 명확화

**현재 계획서**:
> 공정성 지수 (개별 테넌트): 0.90+
> 공정성 지수 (가중치 정규화): 0.95+

**문제점**:
- WFQ의 공정성 지수가 0.32로 낮음
- 이는 **의도된 동작**: 가중치가 다른 테넌트에게 다른 서비스 품질 제공
- 측정 방식이 "동일 등급 테넌트 간 공정성"이 아닌 "전체 테넌트 간 공정성"으로 계산됨

**수정 제안**:
```
공정성 측정 방식:
1. 동일 등급 테넌트 간: 0.90+ 목표
2. 전체 시스템 (가중치 차이 반영): 0.30-0.40 예상 (가중치 비율에 따라 변동)
```

### 2.3 기술 스택 명확화

**현재 계획서**:
> 데이터베이스: SQLite

**실제 구현**:
> 저장소: JSON 파일 (네이티브 모듈 의존성 제거)

**수정 제안**:
- SQLite → JSON 파일 저장소로 변경
- 이유: better-sqlite3의 네이티브 컴파일 문제 회피, 학부생 수준 유지

### 2.4 성공 지표 업데이트

**추가 제안**:
| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| URGENT 요청 개선율 | 50%+ | Priority 스케줄러 실험 |
| Enterprise 테넌트 개선율 | 50%+ | WFQ 스케줄러 실험 |

## 3. 구현 차이점 문서화

### 3.1 계획 vs 실제

| 항목 | 계획 | 실제 | 사유 |
|------|------|------|------|
| 저장소 | SQLite | JSON 파일 | 네이티브 모듈 호환성 |
| 패키지 수 | 4개 이하 | 3개 | 목표 달성 |
| 코드 라인 수 | ~640줄 | ~1,500줄 | 테스트 및 실험 포함 |
| 실시간 통신 | HTTP 폴링 | HTTP 폴링 | 계획대로 |

### 3.2 추가된 기능

1. **실험 스크립트**: 4개 스케줄러 성능 비교 자동화
2. **JSON 저장소**: 요청 로그 영구 저장
3. **Aging/Boosting**: 기아 방지 메커니즘 구현
