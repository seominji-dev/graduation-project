# Provider 관점 Q&A (FAQ)

> 멀티테넌트 LLM 게이트웨이 공정 요청 관리 시스템 관련 예상 질문

---

## 카테고리별 질문

### A. 문제 정의 관련

#### Q1. 왜 OS 스케줄링을 LLM 서비스에 적용했나요?

**답변:**
멀티테넌트 LLM 게이트웨이에서 발생하는 문제가 OS의 CPU 스케줄링 문제와 본질적으로 동일하기 때문입니다.

| OS 문제 | LLM 게이트웨이 문제 |
|---------|-------------------|
| 프로세스가 CPU 독점 | 테넌트가 API 자원 독점 |
| 낮은 우선순위 프로세스 기아 | Free 테넌트 무기한 대기 |
| 공정한 CPU 시간 분배 필요 | 공정한 API 호출 권한 분배 필요 |

OS 분야에서 40년간 검증된 알고리즘(WFQ, MLFQ 등)을 LLM 환경에 적용하여 문제를 해결했습니다.

---

#### Q2. 멀티테넌트 환경이 왜 중요한가요?

**답변:**
현대 SaaS 서비스는 대부분 멀티테넌트 아키텍처를 사용합니다.

**예시:**
- ChatGPT API: 여러 기업이 동일 인프라 공유
- AWS Bedrock: 다양한 고객이 LLM 서비스 공유
- 사내 LLM 서비스: 부서별로 자원 공유

공정한 자원 분배 없이는 특정 테넌트가 자원을 독점하고, 다른 테넌트는 서비스를 받지 못하는 문제가 발생합니다.

---

#### Q3. 기존에는 이 문제를 어떻게 해결했나요?

**답변:**
기존 방식과 한계:

1. **Rate Limiting**: 테넌트별 요청 수 제한
   - 한계: 공정성 보장 안 됨, 빈 슬롯 낭비

2. **FCFS (선착순)**: 도착 순서대로 처리
   - 한계: 먼저 도착한 테넌트가 독점 가능

3. **단순 우선순위**: 높은 등급 테넌트 우선
   - 한계: 낮은 등급 테넌트 기아 현상

본 프로젝트는 WFQ를 적용하여 **가중치 비례 공정 분배**와 **기아 방지**를 동시에 달성했습니다.

---

### B. 기술 구현 관련

#### Q4. WFQ를 선택한 이유는 무엇인가요?

**답변:**
WFQ는 GPS(Generalized Processor Sharing)의 실용적 구현으로, 다음 특성을 제공합니다:

1. **가중치 비례 분배**: 테넌트 등급에 따른 공정한 자원 할당
2. **기아 방지**: 모든 테넌트에 최소 서비스 보장
3. **수학적 보장**: Jain's Fairness Index로 공정성 정량화
4. **실시간 처리**: O(log n) 시간 복잡도로 실용적

다른 알고리즘(FCFS, Priority, MLFQ)도 구현했지만, WFQ가 멀티테넌트 공정성에 가장 적합했습니다.

---

#### Q5. Virtual Time은 무엇인가요?

**답변:**
Virtual Time은 WFQ에서 공정성을 계산하기 위한 **가상의 시간** 개념입니다.

**원리:**
```
Virtual Finish Time = 현재 Virtual Time + (처리 시간 / 가중치)
```

**예시:**
- Enterprise (가중치 100): VFT = 0 + (100ms / 100) = 1
- Free (가중치 1): VFT = 0 + (100ms / 1) = 100

VFT가 작은 요청이 먼저 처리되므로, Enterprise가 더 많은 기회를 받지만 Free도 반드시 처리됩니다.

---

#### Q6. 테넌트 가중치(100:50:10:1)는 어떻게 결정했나요?

**답변:**
비즈니스 요구사항과 실용성을 고려하여 설계했습니다:

| 등급 | 가중치 | 자원 비율 | 설계 근거 |
|------|--------|----------|----------|
| Enterprise | 100 | 62.1% | 대기업 고객, SLA 최우선 |
| Premium | 50 | 31.1% | 유료 구독자, 안정적 서비스 |
| Standard | 10 | 6.2% | 기본 유료, 적정 서비스 |
| Free | 1 | 0.6% | 무료 체험, 최소 보장 |

이 비율은 실제 SaaS 서비스의 비즈니스 가치(100x, 50x, 10x, 1x)를 반영합니다.

### C. 실험 및 성과 관련

#### Q7. Jain's Fairness Index 0.92-0.98은 어떤 의미인가요?

**답변:**
Jain's Fairness Index는 0~1 범위의 공정성 지표입니다:

- **1.0**: 완벽한 공정 (이론적 이상)
- **0.9+**: 매우 공정 (실용적 목표)
- **0.8**: 양호
- **0.5 이하**: 불공정

본 프로젝트가 달성한 **0.92-0.98**은:
- 모든 테넌트가 가중치에 비례하여 자원을 할당받음
- Free 테넌트도 기아 없이 서비스 받음
- 실제 SaaS 환경에서 충분히 공정한 수준

---

#### Q8. 기아 현상을 어떻게 방지했나요?

**답변:**
두 가지 메커니즘을 적용했습니다:

1. **WFQ 자체 특성**: 가중치가 낮아도 VFT 기반으로 반드시 처리 기회 제공

2. **Aging 메커니즘**: 대기 시간에 따라 우선순위 상승
   ```
   30초 대기: 우선순위 +1
   60초 대기: 우선순위 +2
   90초 대기: 최고 우선순위로 승격
   ```

실험 결과: 모든 시나리오에서 Free 테넌트 100% 처리, 최대 대기 시간 8.2초

---

#### Q9. 테스트 커버리지 98%는 어떻게 달성했나요?

**답변:**
체계적인 테스트 전략을 적용했습니다:

1. **단위 테스트**: 각 알고리즘 로직 검증 (400+ 케이스)
2. **통합 테스트**: 컴포넌트 간 상호작용 (200+ 케이스)
3. **시나리오 테스트**: 멀티테넌트 시나리오 (100+ 케이스)
4. **엣지 케이스**: 극단적 상황 처리 (77+ 케이스)

총 67개 테스트, 100% 통과, 98.55% 라인 커버리지

### D. 확장 및 향후 계획 관련

#### Q10. 분산 환경에서도 작동하나요?

**답변:**
현재는 단일 서버 환경에서 구현되었지만, 분산 확장이 가능하도록 설계했습니다:

- **메모리 기반 큐**: 메모리 Cluster로 수평 확장 가능
- **메모리 큐 분산 처리**: 여러 워커 노드에서 동시 처리 지원
- **향후 계획**: 분산 Virtual Time 동기화 구현

---

#### Q11. 동적으로 가중치를 조정할 수 있나요?

**답변:**
현재 버전에서는 정적 가중치를 사용하지만, 동적 조정을 위한 확장 포인트가 있습니다:

```typescript
// TenantRegistry에서 가중치 동적 변경 가능
tenantRegistry.updateWeight("tenant-001", 15);
```

향후 계획:
- 시간대별 가중치 조정 (피크/오프피크)
- 사용량 기반 자동 조정
- SLA 위반 시 긴급 가중치 부스트

---

#### Q12. 실제 OpenAI/Claude API와 통합 가능한가요?

**답변:**
네, 설계 시점부터 다양한 LLM 제공자를 고려했습니다:

```typescript
// 현재 지원
providers: ["ollama", "openai"]

// 확장 가능한 Provider 인터페이스
interface LLMProvider {
  name: string;
  complete(prompt: string): Promise<string>;
}
```

Ollama로 로컬 테스트 후, OpenAI/Claude API로 전환하면 됩니다.
