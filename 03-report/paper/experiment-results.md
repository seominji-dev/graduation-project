# LLM 스케줄러 실험 결과 보고서

**프로젝트**: OS 스케줄링 기법을 활용한 LLM API 요청 최적화 시스템
**작성자**: 서민지
**작성일**: 2026-02-04

---

## 1. 실험 개요

### 1.1 실험 목적

4가지 OS 스케줄링 알고리즘(FCFS, Priority, MLFQ, WFQ)을 LLM API 요청 관리에 적용하여 성능을 비교 분석한다.

### 1.2 실험 환경

| 항목 | 값 |
|------|-----|
| 총 요청 수 | 100개 |
| 테넌트 수 | 4개 (Enterprise, Premium, Standard, Free) |
| 처리 시간 범위 | 10-100ms |
| 요청 도착 간격 | 1-10ms (평균 5ms) |

### 1.3 측정 지표

- **평균 대기 시간 (Average Wait Time)**: 요청이 큐에서 대기한 평균 시간
- **최대 대기 시간 (Max Wait Time)**: 가장 오래 대기한 요청의 시간
- **처리량 (Throughput)**: 초당 처리된 요청 수
- **공정성 지수 (Jain's Fairness Index)**: 테넌트 간 자원 분배 공정성 (0~1)

## 2. 실험 결과

### 2.1 기본 성능 지표

| 스케줄러 | 평균 대기(ms) | 최대 대기(ms) | 처리량(req/s) | 공정성 지수 |
|----------|--------------|--------------|---------------|------------|
| **FCFS** | 2,571.75 | 4,952.00 | 17.99 | N/A |
| **Priority** | 2,826.41 | 5,243.00 | 17.09 | N/A |
| **MLFQ** | 2,571.75 | 4,952.00 | 17.99 | N/A |
| **WFQ** | 2,819.32 | 5,536.00 | 16.84 | 0.3159 |

### 2.2 분석

#### FCFS (First-Come, First-Served)
- **특징**: 가장 단순한 베이스라인 알고리즘
- **결과**: 평균 대기 시간 2,571.75ms
- **장점**: 구현이 간단하고 예측 가능
- **단점**: 우선순위 구분 없이 모든 요청을 동등하게 처리

#### Priority Scheduling (with Aging)
- **특징**: 우선순위 기반 처리 + Aging으로 기아 방지
- **결과**:
  - URGENT 요청: **1,121.75ms** (FCFS 대비 62.3% 개선)
  - LOW 요청: 3,080.91ms
- **장점**: 긴급 요청의 빠른 처리 보장
- **단점**: 전체 평균 대기 시간은 FCFS보다 9.9% 증가

#### MLFQ (Multi-Level Feedback Queue)
- **특징**: 4단계 큐 + Boosting
- **결과**: FCFS와 동일한 성능 (단일 실행 시뮬레이션 한계)
- **장점**: 대화형 작업과 배치 작업 혼합 환경에 적합
- **참고**: 실시간 환경에서는 더 나은 응답성 기대

#### WFQ (Weighted Fair Queuing)
- **특징**: 가중치 기반 공정 스케줄링
- **결과**:
  - Enterprise (weight: 100): **849.32ms**
  - Premium (weight: 50): 2,103.04ms
  - Standard (weight: 10): 3,431.20ms
  - Free (weight: 1): 4,893.72ms
- **장점**: 멀티테넌트 환경에서 차등 서비스 제공
- **공정성**: 가중치에 비례한 서비스 품질 보장

## 3. 우선순위별 성능 비교

### 3.1 Priority 스케줄러 vs FCFS

| 우선순위 | Priority 스케줄러 | FCFS | 개선율 |
|----------|------------------|------|--------|
| URGENT | 1,121.75ms | 2,971.00ms | **+62.3%** |
| HIGH | 3,164.47ms | 2,735.20ms | -15.7% |
| NORMAL | 2,830.23ms | 2,386.05ms | -18.6% |
| LOW | 3,080.91ms | 2,648.27ms | -16.3% |

**핵심 발견**: Priority 스케줄러는 URGENT 요청의 대기 시간을 62.3% 단축했다.

## 4. 테넌트별 성능 비교 (WFQ)

| 테넌트 | 가중치 | 평균 대기(ms) | FCFS 대비 |
|--------|--------|--------------|----------|
| Enterprise | 100 | 849.32 | **+67.0%** |
| Premium | 50 | 2,103.04 | +18.2% |
| Standard | 10 | 3,431.20 | -33.4% |
| Free | 1 | 4,893.72 | -90.3% |

**핵심 발견**: WFQ는 가중치에 비례하여 서비스 품질을 차등 제공한다.

## 5. 결론

### 5.1 알고리즘별 적합한 사용 시나리오

| 알고리즘 | 적합한 시나리오 |
|----------|----------------|
| FCFS | 단순한 환경, 공정한 처리 필요 시 |
| Priority | 긴급 요청 우선 처리 필요 시 (고객 문의 등) |
| MLFQ | 대화형/배치 작업 혼합 환경 |
| WFQ | 멀티테넌트 SaaS 서비스 |

### 5.2 주요 성과

1. **4개 알고리즘 구현 완료**: 모든 스케줄러가 정상 동작
2. **테스트 통과**: 69개 테스트 100% 통과 (98.65% 코드 커버리지)
3. **Priority 효과 입증**: URGENT 요청 대기 시간 62.3% 단축
4. **WFQ 차등 서비스**: Enterprise 테넌트 67.0% 개선

### 5.3 향후 과제

1. 실시간 환경에서의 MLFQ 성능 검증
2. WFQ 공정성 지수 개선 (가중치 정규화 적용)
3. 부하 변동 시나리오 테스트
4. 실제 LLM API (Ollama) 연동 테스트
