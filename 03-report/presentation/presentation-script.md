## 슬라이드 3: LLM API 사용 환경의 문제점 (1:00-1:40)

먼저 현재 LLM API 사용 환경의 문제점에 대해 이야기해보겠습니다.

첫째, 비용 폭증 문제입니다. 무분별한 API 요청으로 예산이 초과됩니다.

넷째, 자원 낭비입니다. 우선순위 없는 비효율적 처리로 자원이 낭비됩니다.

기존 시스템은 도착 순서대로 처리하므로, 만약 B의 요청이 먼저 도착했다면 긴급한 A의 요청도 100건의 대량 작업이 모두 처리될 때까지 기다려야 합니다.

이는 고객 불만, 서비스 품질 저하, 비즈니스 손실로 이어집니다.

## 슬라이드 7: 4가지 스케줄링 알고리즘 (3:40-4:30)

본 프로젝트에서는 4가지 스케줄링 알고리즘을 구현했습니다.

첫째, FCFS입니다. First-Come, First-Served로 도착 순서대로 처리하는 가장 기본적인 알고리즘입니다. 베이스라인으로 사용합니다.

둘째, Priority Scheduling입니다. 우선순위 기반으로 처리하며, Aging 메커니즘으로 기아 현상을 방지합니다.

셋째, MLFQ입니다. Multi-Level Feedback Queue로 4단계 큐를 통해 응답성과 처리량을 동시에 달성합니다.

넷째, WFQ입니다. Weighted Fair Queuing으로 가중치 기반 공정 스케줄링을 통해 멀티테넌트 환경을 최적화합니다.

각 알고리즘은 서로 다른 특성과 장점을 가지고 있습니다.

## 슬라이드 8: MLFQ 상세 설명 (4:30-5:20)

MLFQ에 대해 조 더 자세히 설명드리겠습니다.

MLFQ는 5가지 규칙으로 동작합니다.

Rule 1: 높은 우선순위 큐를 먼저 실행합니다.

Rule 2: 같은 우선순위는 Round-Robin 방식으로 처리합니다.

Rule 3: 새 작업은 최고 우선순위인 Q0에서 시작합니다.

Rule 4: 타임 슬라이스를 초과하면 다음 레벨로 강등됩니다.

Rule 5: 주기적으로 모든 작업을 Q0로 Boost합니다.

이 시스템은 4개의 큐로 구성됩니다. Q0는 1초 타임 퀀텀으로 짧은 대화형 요청을 처리하고, Q1은 3초로 중간 길이 요청을, Q2는 8초로 긴 요청을 처리합니다. Q3는 타임 제한이 없어 배치 작업에 적합합니다.

## 슬라이드 9: WFQ와 공정성 (5:20-6:00)

WFQ는 공정성을 보장하는 데 중점을 둡니다.

Virtual Time 개념을 사용합니다. Virtual Finish Time은 Virtual Start Time에 서비스 시간을 가중치로 나눈 값을 더해 계산합니다.

테넌트별 가중치는 Enterprise 100, Premium 50, Standard 10, Free 1로 부여합니다.

이를 통해 Jain's Fairness Index로 측정한 결과, 개별 테넌트 수준에서 0.92-0.98의 매우 높은 공정성을 달성했습니다.

전체 시스템 수준에서는 0.89인데, 이는 티어 간 의도된 가중치 차이를 반영한 것입니다.

결과적으로 가중치에 따른 자원 분배가 공정하게 이루어지고 있습니다.

## 슬라이드 10: 시스템 아키텍처 (6:00-6:40)

이제 시스템 아키텍처를 설명드리겠습니다.

본 시스템은 3계층 구조로 설계되었습니다.

최상위에는 API Layer가 있습니다. Request Controller, Scheduler Factory, Dashboard Service로 구성됩니다.

하위에는 Management Layer가 있습니다. Aging Manager, Boost Manager, Fairness Calculator 등의 관리 컴포넌트가 있습니다.

최하위에는 Storage Layer가 있습니다. Redis 기반의 BullMQ 큐, MongoDB 로그 저장소, LLM Service 연결이 포함됩니다.

## 슬라이드 12: 도메인 모델 (7:10-7:50)

핵심 데이터 구조를 설명드리겠습니다.

LLMRequest 모델은 id, prompt, provider, priority, status 등의 필드를 가집니다.

priority는 LOW(0)부터 URGENT(3)까지 4단계로 구성됩니다.

status는 pending에서 시작해 queued, processing을 거쳐 completed나 failed로 종료됩니다.

MLFQ를 위해 queueLevel, timeSliceUsed, totalCPUTime 필드를 추가했습니다.

WFQ를 위해 virtualStartTime, virtualFinishTime, weight, tenantId 필드를 추가했습니다.

이러한 확장 필드들은 각 스케줄링 알고리즘의 특성을 지원합니다.

## 슬라이드 13: 스케줄러 인터페이스 (7:50-8:30)

모든 스케줄러는 통일된 인터페이스를 구현합니다.

첫째, 런타임에 알고리즘을 교체할 수 있습니다.

둘째, 테스트가 용이합니다.

셋째, 확장성이 우수합니다.

새로운 알고리즘을 쉽게 추가할 수 있습니다.

## 슬라이드 14: 관리자 컴포넌트 (8:30-9:10)

스케줄링을 보조하는 관리자 컴포넌트들을 소개합니다.

AgingManager는 Priority Scheduler용으로, 기아 방지를 위해 우선순위를 동적으로 조정합니다. 30초 이상 대기하면 우선순위를 상향시킵니다.

BoostManager는 MLFQ용으로, 60초 주기로 모든 작업을 Q0로 Boost합니다. 이는 Rule 5를 구현한 것입니다.

FairnessCalculator는 WFQ용으로, Jain's Fairness Index를 실시간으로 계산하고 공정성 리포트를 생성합니다.

TenantRegistry는 WFQ용으로 테넌트별 가중치를 관리하고 동적으로 조정합니다.

이러한 컴포넌트들이 협력하여 스케줄링 품질을 높입니다.

## 슬라이드 15: FCFS & Priority 구현 (9:10-9:50)

구현 상세를 설명드리겠습니다.

FCFS는 BullMQ 큐의 FIFO 특성을 활용합니다. 모든 작업에 동일 우선순위를 부여하며, 시간 복잡도는 O(1)입니다.

Priority는 BullMQ priority 옵션을 활용합니다. 우선순위 변환 로직이 중요한데요.

BullMQ는 숫자가 작을수록 높은 우선순위를 가집니다. 따라서 변환식은 (MAX_PRIORITY - priority) * 2를 사용합니다.

예를 들어 URGENT(3)는 0으로, HIGH(2)는 2로, NORMAL(1)는 4로, LOW(0)는 6으로 변환됩니다.

AgingManager와 연동하여 장기 대기 요청의 우선순위를 상향시킵니다.

## 슬라이드 16: MLFQ 구현 (9:50-10:30)

MLFQ 구현의 핵심 포인트는 다음과 같습니다.

첫째, 4개의 독립 큐를 운영합니다. mlfq-q0, mlfq-q1, mlfq-q2, mlfq-q3입니다.

둘째, 각 작업의 CPU 사용량을 추적하여 타임 슬라이스 초과 여부를 판단합니다.

셋째, 타임 슬라이스 초과 시 다음 레벨로 큐를 강등합니다.

넷째, 60초마다 모든 작업을 Q0로 Boost합니다.

성능 특성을 보면, 대화형 요청은 Q0에서 빠르게 처리되고, 배치 작업은 점진적 강등 후 Q3에서 안정적으로 처리됩니다.

Boost 기능으로 장기 대기도 방지됩니다.

## 슬라이드 17: WFQ 구현 (10:30-11:10)

WFQ 구현은 Virtual Time 계산이 핵심입니다.

Virtual Finish Time은 Virtual Start Time에 서비스 시간을 가중치로 나눈 값을 더해 계산합니다.

구현 컴포넌트는 세 가지입니다.

VirtualTimeTracker는 전역 가상 시간을 관리합니다.

TenantRegistry는 테넌트 가중치를 관리합니다.

FairnessCalculator는 공정성 지표를 계산합니다.

공정성 보장 원리는 다음과 같습니다.

가중치가 높은 테넌트는 낮은 Virtual Finish Time을 가지므로 우선 처리됩니다.

가중치가 낮은 테넌트는 높은 Virtual Finish Time을 가지므로 지연 처리됩니다.

결과적으로 가중치 비율에 맞는 자원 분배가 이루어집니다.

## 슬라이드 19: 성능 비교 (11:50-12:30)

실험 결과를 통해 알고리즘별 성능을 비교해보겠습니다.

FCFS를 기준으로 했을 때, Priority는 평균 대기시간을 30% 개선했습니다.

MLFQ는 대기시간을 40% 개선하고 처리량을 20% 증가시켰습니다.

WFQ는 대기시간을 35% 개선했습니다.

공정성 측면에서는 FCFS와 Priority가 낮은 반면, MLFQ는 높고, WFQ는 매우 높습니다.

핵심 인사이트는 다음과 같습니다.

Priority는 긴급 요청의 대기시간을 효과적으로 단축합니다.

MLFQ는 대화형과 배치 작업을 모두 효과적으로 처리합니다.

WFQ는 멀티테넌트 환경에서 최적의 성능을 보입니다.

## 슬라이드 21: 알고리즘 선택 가이드 (13:00-13:30)

단순 환경에서는 FCFS를 권장합니다. 오버헤드가 최소화됩니다.

우선순위가 필요한 환경에서는 Priority를 권장합니다. 긴급 요청을 우선 처리할 수 있습니다.

혼합 워크로드 환경에서는 MLFQ를 권장합니다. 응답성과 처리량의 균형을 잡을 수 있습니다.

멀티테넌트 환경에서는 WFQ를 권장합니다. 공정성이 보장됩니다.

## 슬라이드 22: 라이브 데모 (13:30-14:30)

이제 실제 데모를 통해 시스템 동작을 확인하겠습니다.

데모 시나리오는 다음과 같습니다.

첫째, 시스템을 시작하여 서버를 구동하고 Redis, MongoDB에 연결합니다.

둘째, FCFS 테스트로 순차 처리를 확인합니다.

셋째, Priority 테스트로 우선순위별 처리 순서를 확인합니다.

넷째, MLFQ 테스트로 큐 강등 및 Boost 동작을 확인합니다.

다섯째, WFQ 테스트로 가중치별 처리량과 Fairness Index를 확인합니다.

데모 URL은 http://localhost:3000입니다.

대시보드에서 실시간 큐 상태, 처리 통계, 공정성 지표를 확인할 수 있습니다.

## 슬라이드 23: 결론 (14:30-15:10)

결론을 말씀드리겠습니다.

실용적 가치는 멀티테넌트 환경 지원, REST API 및 대시보드 제공, 오픈소스 공개입니다.
