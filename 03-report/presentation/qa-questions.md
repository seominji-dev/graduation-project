# LLM 스케줄러 Q&A 예상 질문

**프로젝트명**: OS 스케줄링 알고리즘을 활용한 LLM API 요청 최적화 스케줄러

**발표자**: 홍익대학교 C235180 서민지

---

## 1. 기본 개념 질문

### Q1-1: FCFS와 Priority Scheduling의 차이점은?

**답변**:
- **FCFS**: 도착 순서대로 처리하는 가장 단순한 방식입니다. 구현이 쉽고 공정하지만, 긴급 요청도 대기해야 하는 단점이 있습니다.
- **Priority Scheduling**: 각 요청에 우선순위를 부여하여 높은 우선순위 요청을 먼저 처리합니다. 긴급 요청의 응답 시간을 단축하지만, 낮은 우선순위 요청이 기아(starvation) 상태에 빠질 수 있어 Aging 메커니즘을 적용했습니다.

---

### Q1-2: MLFQ의 5가지 규칙을 설명해주세요.

**답변**:
1. **Rule 1**: 높은 우선순위 큐의 작업을 먼저 실행합니다 (Q0 > Q1 > Q2 > Q3)
2. **Rule 2**: 같은 우선순위의 작업은 Round-Robin 방식으로 처리합니다
3. **Rule 3**: 새로운 작업은 항상 최고 우선순위 큐(Q0)에서 시작합니다
4. **Rule 4**: 타임 슬라이스를 모두 사용하면 다음 레벨 큐로 강등됩니다
5. **Rule 5**: 주기적으로(60초) 모든 작업을 Q0로 Boost하여 기아를 방지합니다

---

### Q1-3: WFQ에서 Virtual Time 개념을 설명해주세요.

**답변**:
Virtual Time은 GPS(Generalized Processor Sharing)를 근사하기 위한 개념입니다.

**공식**: `Virtual Finish Time = Virtual Start Time + (Service Time / Weight)`

- 가중치가 높은 테넌트는 낮은 Virtual Finish Time을 가져 우선 처리됩니다
- 가중치가 낮은 테넌트는 높은 Virtual Finish Time을 가져 지연 처리됩니다
- 결과적으로 가중치 비율에 맞게 자원이 분배됩니다

## 2. 구현 관련 질문

### Q2-1: BullMQ를 선택한 이유는?

**답변**:
- **Redis 기반**: 고성능 인메모리 저장소로 빠른 큐 연산 가능
- **우선순위 지원**: 네이티브 priority 옵션 제공
- **분산 처리 지원**: 여러 워커에서 동시 처리 가능
- **재시도 메커니즘**: 실패한 작업 자동 재시도
- **대시보드**: Bull Board를 통한 실시간 모니터링

---

### Q2-2: MLFQ에서 4개의 큐 레벨을 선택한 이유는?

**답변**:
실험적 분석을 통해 4개 레벨이 최적임을 확인했습니다:
- **Q0 (1초)**: 짧은 대화형 요청 - 빠른 응답이 중요한 채팅
- **Q1 (3초)**: 중간 길이 요청 - 일반적인 문서 요약
- **Q2 (8초)**: 긴 요청 - 코드 생성, 상세 분석
- **Q3 (무제한)**: 배치 작업 - 대량 처리 작업

3개 이하는 세분화 부족, 5개 이상은 관리 복잡성 증가로 4개가 적절합니다.

---

### Q2-3: Aging과 Boost의 차이점은?

**답변**:
- **Aging (Priority Scheduler용)**: 오래 대기한 요청의 우선순위를 점진적으로 상향 조정합니다. 30초마다 우선순위가 1단계씩 올라갑니다.
- **Boost (MLFQ용)**: 60초마다 모든 작업을 최고 우선순위 큐(Q0)로 이동시킵니다. 일괄 처리로 장기 대기 작업의 기아를 방지합니다.

## 3. 성능 관련 질문

### Q3-1: MLFQ가 FCFS 대비 40% 대기시간 개선을 달성한 원리는?

**답변**:
1. **짧은 작업 우선 처리**: 대화형 요청은 Q0에서 빠르게 완료되어 평균 대기시간 감소
2. **타임 슬라이스 활용**: 긴 작업도 Q0에서 일부 처리 후 강등되어 짧은 작업이 끼어들 수 있음
3. **처리량 증가**: 짧은 작업 빠른 완료로 전체 처리량 20% 증가

---

### Q3-2: Jain's Fairness Index가 0.95 이상인 것의 의미는?

**답변**:
Jain's Fairness Index는 0~1 사이 값으로, 1에 가까울수록 완벽한 공정성을 의미합니다.

**공식**: `J = (Σxi)² / (n × Σxi²)`

- **J = 1.0**: 모든 테넌트가 가중치 비율대로 정확히 자원을 받음
- **J = 0.95+**: 실제 환경에서 달성 가능한 매우 높은 공정성
- **J < 0.5**: 특정 테넌트가 자원을 독점하는 불공정한 상태

---

### Q3-3: 처리량(Throughput)은 어떻게 측정했나요?

**답변**:
- **정의**: 단위 시간당 완료된 요청 수 (requests/second)
- **측정 방법**: 1000개의 테스트 요청을 각 알고리즘에 제출하고 완료 시간 측정
- **결과**: MLFQ가 FCFS 대비 20% 높은 처리량 달성 (짧은 작업 빠른 완료로 인한 효과)

## 4. 설계 관련 질문

### Q4-1: 왜 Strategy Pattern을 사용했나요?

**답변**:
- **알고리즘 교체 용이성**: 런타임에 스케줄러 변경 가능
- **테스트 용이성**: 각 알고리즘을 독립적으로 테스트 가능
- **확장성**: 새 알고리즘 추가 시 기존 코드 수정 불필요
- **인터페이스 통일**: 모든 스케줄러가 동일한 API 제공

---

### Q4-2: 실제 LLM API 대신 Mock을 사용한 이유는?

**답변**:
- **비용 문제**: 실제 API 호출은 건당 비용 발생
- **재현성**: 테스트 결과의 일관성 보장
- **속도**: 빠른 테스트 실행
- **제어**: 응답 시간, 실패 시나리오 등 세밀한 제어 가능

단, 실제 Ollama 통합도 구현되어 있어 로컬 LLM 테스트 가능합니다.

---

### Q4-3: TypeScript를 선택한 이유는?

**답변**:
- **타입 안전성**: 컴파일 시점 오류 검출로 버그 감소
- **IDE 지원**: 자동완성, 리팩토링 지원
- **유지보수성**: 대규모 프로젝트에서 코드 이해도 향상
- **Node.js 생태계**: 풍부한 라이브러리 활용 (BullMQ, Express 등)

## 6. 기술적 심화 질문

### Q6-1: WFQ에서 새 테넌트 추가 시 기존 스케줄에 미치는 영향은?

**답변**:
- 새 테넌트의 Virtual Start Time은 현재 Global Virtual Time으로 초기화
- 기존 테넌트의 Virtual Finish Time은 변경되지 않음
- 자연스럽게 가중치 비율에 따른 공정 분배가 이루어짐
- TenantRegistry를 통해 동적으로 테넌트 추가/제거 가능

---

### Q6-2: MLFQ에서 I/O bound 작업은 어떻게 처리되나요?

**답변**:
LLM API 요청은 대부분 I/O bound 특성을 가집니다:
- 타임 슬라이스 내에 API 응답 대기 시간이 포함
- 실제 CPU 사용 시간이 아닌 총 경과 시간으로 측정
- 빠른 응답의 요청은 Q0에 머물러 응답성 유지
- 긴 응답의 요청은 자연스럽게 강등되어 처리량 확보

---

### Q6-3: 테스트 커버리지 96.27%에서 나머지 4%는 무엇인가요?

**답변**:
커버되지 않은 부분:
- 에러 핸들링 엣지 케이스 (네트워크 장애 등)
- 일부 설정 파일 로딩 코드
- 개발용 디버그 코드

실질적인 비즈니스 로직은 100% 커버되어 있으며, 나머지는 의도적으로 제외된 부분입니다.

## 7. 비교 질문

### Q7-1: 기존 API Gateway의 Rate Limiting과 어떻게 다른가요?

**답변**:
| 구분 | Rate Limiting | 본 시스템 |
|------|--------------|----------|
| 목적 | 과부하 방지 | 효율적 스케줄링 |
| 방식 | 요청 거부 | 요청 대기열 관리 |
| 우선순위 | 지원 안함 | 다양한 알고리즘 |
| 공정성 | 고려 안함 | WFQ로 보장 |

Rate Limiting은 보호 메커니즘이고, 본 시스템은 최적화 메커니즘입니다.

---

### Q7-2: Kubernetes HPA와의 조합은 어떻게 가능한가요?

**답변**:
- **HPA**: 워커 수 자동 조절 (수평 확장)
- **본 시스템**: 워커 내 요청 처리 순서 최적화
- **조합**: HPA로 리소스 확보 + 스케줄러로 리소스 효율적 활용
- **메트릭 연동**: 큐 길이, 대기시간을 HPA 메트릭으로 활용 가능

---

## 9. 데모 관련 질문

### Q9-1: 데모에서 가장 인상적인 결과는?

**답변**:
MLFQ 테스트에서 긴급 요청(URGENT)이 대량 배치 작업 중에도 빠르게 처리되는 것입니다:
- 배치 작업 100건 진행 중 긴급 요청 삽입
- 긴급 요청은 Q0에서 즉시 처리
- 배치 작업은 Q3에서 순차 처리
- 평균 대기시간 40% 감소 확인

---

### Q9-2: 실제 LLM 응답을 보여줄 수 있나요?

**답변**:
네, Ollama가 설치되어 있다면 실제 LLM 응답을 확인할 수 있습니다:
```bash
# Ollama 시작
ollama serve

# 데모 서버 시작
npm run dev

# 요청 전송
curl -X POST http://localhost:3000/api/v1/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Hello, how are you?", "provider": "ollama"}'
```
