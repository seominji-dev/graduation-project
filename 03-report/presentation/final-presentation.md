## 슬라이드 7: MLFQ 선점형 구현

### MLFQ 구조

```
Q0 (최상위) ─────┐    TIME_QUANTUM[0] = 500ms
                  │    선점 가능
Q1 ──────────────┤    TIME_QUANTUM[1] = 1500ms
                  │
Q2 ──────────────┤    TIME_QUANTUM[2] = 4000ms
                  │
Q3 (최하위) ─────┘    TIME_QUANTUM[3] = 무제한
```

### 선점형 동작

1. 요청이 처리 시작되면 타이머 시작 (500ms)
2. 500ms 내에 완료되지 않으면 하위 큐로 강등
3. 짧은 요청은 Q0에서 빠르게 완료

## 슬라이드 8: WFQ 가중치 기반 공정 분배

### 테넌트 등급별 가중치

| 테넌트 | 가중치 | 비율 |
|--------|--------|------|
| Enterprise | 100 | 62.1% |
| Premium | 50 | 31.1% |
| Standard | 10 | 6.2% |
| Free | 1 | 0.6% |

### Virtual Time 계산

```
V(t) = V(t_prev) + (service_time / sum_of_active_weights)
```

### Jain's Fairness Index

```
JFI = (Σ xi)² / (n × Σ xi²)
```

- 시스템 수준 JFI: 0.89 (의도적 불공정)
- 테넌트 수준 JFI: 0.92-0.98 (동일 등급 내 공정)

# 부록: Q&A 준비 (Q&A Preparation)

## 예상 질문 및 답변

### Q1: 왜 TypeScript 대신 JavaScript를 사용했나요?

**A:** 학부 수준 프로젝트의 복잡도 관리와 의존성 최소화를 위해 JavaScript를 선택했습니다. TypeScript로 구현했던 초기 버전(Candidate-1)이 있었으나, 학부생 수준에서 접근 가능한 코드를 유지하기 위해 JavaScript로 전환했습니다.

### Q2: 실제 프로덕션 환경에서 사용 가능한가요?

**A:** 현재 구현은 단일 서버 환경을 가정합니다. 프로덕션 환경에서는 분산 캐시(Redis), 로드 밸런싱, 상태 동기화 등 추가 구현이 필요합니다. 다만 스케줄링 알고리즘 자체는 즉시 적용 가능합니다.

### Q3: MLFQ 선점형 구현의 가장 큰 어려움은 무엇이었나요?

**A:** 시간 슬라이스 경과를 정확히 추적하는 것과 선점 시 요청 상태를 안전하게 이동시키는 것이었습니다. `startProcessing()`, `checkPreemption()`, `preempt()` 메서드 체인을 구현하여 해결했습니다.

### Q4: JFI 0.89는 왜 낮은가요?

**A:** WFQ의 목적은 "공정한 분배"가 아니라 "의도한 가중치 비율에 따른 분배"입니다. 따라서 시스템 수준 JFI가 낮은 것은 정상이며, 동일 등급 내 테넌트 간 공정성(0.92-0.98)이 더 중요합니다.

### Q5: Rate Limiter는 다른 스케줄러와 어떻게 다른가요?

**A:** Rate Limiter는 "처리 순서"가 아니라 "처리 속도"를 제어합니다. 토큰 버킷 알고리즘을 사용하여 테넌트별 초당 요청 수를 제한합니다.

### Q6: Aging 메커니즘이 실제로 작동하나요?

**A:** PriorityScheduler에 Aging 코드가 구현되어 있으나, 현재 실험 환경에서는 기아 현상이 발생하지 않아 활성화되지 않습니다. 프로덕션 환경에서는 테스트 케이스 추가로 활성화할 수 있습니다.

### Q7: 왜 MongoDB 대신 JSON 파일을 사용했나요?

**A:** 의존성 패키지를 2개(express, jest)로 제한하기 위함입니다. MongoDB를 사용하면 Mongoose 등 추가 패키지가 필요하며, 본 연구의 핵심은 스케줄링 알고리즘이므로 단순한 저장소를 선택했습니다.

### Q8: 통계적 검증에 사용한 방법론은?

**A:** Power Analysis(검정력 ≥ 0.85), Cohen's d(효과 크기), 95% 신뢰구간, t-test(p-value)를 모든 실험 결과에 적용했습니다. 이는 학술적 엄밀함을 확보하기 위함입니다.

### Q9: MLFQ의 타임 퀀텀 값은 어떻게 결정했나요?

**A:** 실험을 통해 최적값을 찾았습니다. Q0: 500ms는 짧은 요청이 완료될 수 있는 최소 시간, Q1: 1500ms, Q2: 4000ms는 점진적 증가, Q3는 무제한으로 긴 작업 수용입니다.

### Q10: 이 시스템의 실제 비즈니스 가치는?

**A:** SaaS 멀티테넌트 서비스에 즉시 적용 가능하며, Enterprise 테넌트에게 5.8배 빠른 응답을 보장하는 SLA 준수 솔루션입니다. 또한 공정성 모니터링으로 Provider가 품질을 정량적으로 관리할 수 있습니다.

### 섹션 3: 시스템 설계 (3분)

"시스템은 Strategy Pattern으로 구현되어 런타임에 알고리즘을 교체할 수 있습니다. 클라이언트가 REST API로 요청을 보내면, 스케줄러 매니저가 선택된 알고리즘에 따라 처리 순서를 결정합니다.

MLFQ는 4단계 큐를 사용하고, 500ms 동안 처리되지 않은 요청은 하위 큐로 이동합니다. WFQ는 테넌트 등급별 가중치(100:50:10:1)에 비례하여 자원을 분배합니다."
