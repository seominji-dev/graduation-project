### 슬라이드 4: 멀티테넌트 환경의 공정성 문제

**핵심 메시지**: Provider 관점에서 발생하는 3가지 핵심 문제

**문제 1: 자원 독점 (Resource Monopolization)**
- 대량 요청을 제출하는 테넌트가 공유 자원 독점
- 다른 테넌트의 서비스 품질 저하

**문제 2: 기아 현상 (Starvation)**
- 우선순위 기반 스케줄링에서 낮은 등급 테넌트 무기한 대기
- SLA(Service Level Agreement) 위반 위험

**문제 3: 공정성 측정 부재**
- 테넌트 간 자원 분배가 공정한지 정량적 측정 불가
- 문제 인지 및 해결 어려움

**시각 자료**: 불공정 상황 다이어그램
```
[문제 상황]

                          └─────┘
```

### 슬라이드 7: OS 프로세스 스케줄링 개요

**핵심 메시지**: OS 스케줄링 목표와 멀티테넌트 환경의 유사성

**OS 스케줄링의 4가지 목표**:
1. **공정성(Fairness)**: 모든 프로세스에 적절한 CPU 할당
2. **기아 방지(Starvation Prevention)**: 낮은 우선순위도 실행 보장
3. **처리량(Throughput)**: 단위 시간당 완료 작업 수 최대화
4. **응답 시간(Response Time)**: 요청부터 첫 응답까지 최소화

**OS와 멀티테넌트 LLM의 개념 매핑**:

| OS 개념 | 멀티테넌트 LLM |
|---------|---------------|
| 프로세스 | LLM API 요청 |
| CPU 시간 | API 호출 권한 |
| 프로세스 우선순위 | 테넌트 등급 |
| 스케줄러 | 요청 스케줄러 |

**시각 자료**: OS와 LLM 개념 매핑 다이어그램

### 슬라이드 8: GPS (Generalized Processor Sharing)

**핵심 메시지**: 이상적인 공정 스케줄링 모델

**GPS 개념** (Parekh & Gallager, 1993):
- 모든 활성 사용자가 가중치에 비례하여 **동시에** CPU 시간 획득
- 완벽한 공정성을 가정하는 이론적 모델

**GPS 수식**:
```
Si(t1, t2) = (wi / Sigma wj) x (t2 - t1)

Si: 시간 구간 [t1, t2] 동안 테넌트 i가 받은 서비스
wi: 테넌트 i의 가중치
Sigma wj: 모든 활성 테넌트 가중치 합
```

**예시 계산** (10초 동안):
| 테넌트 | 가중치 | GPS 자원 비율 | 서비스 시간 |
|--------|--------|--------------|------------|
| Enterprise | 100 | 62.1% | 6.21초 |
| Premium | 50 | 31.1% | 3.11초 |
| Standard | 10 | 6.2% | 0.62초 |
| Free | 1 | 0.6% | 0.06초 |

**GPS의 한계**: 실제 CPU는 동시 처리 불가 → WFQ로 근사

### 슬라이드 9: WFQ (Weighted Fair Queuing)

**핵심 메시지**: GPS를 실용적으로 구현하는 알고리즘

**WFQ 개념** (Demers, Keshav, Shenker, 1989):
- GPS를 패킷 네트워크에서 구현하기 위한 알고리즘
- **Virtual Time** 개념으로 GPS의 공정성 근사

**Virtual Finish Time 수식**:
```
Fi = max(V(t), Pi) + (service_time / weighti)

Fi: 요청 i의 Virtual Finish Time
V(t): 현재 Virtual Time
Pi: 요청 i의 Virtual Start Time
weighti: 테넌트 i의 가중치
```

**WFQ 동작 원리**:
1. 새 요청 도착 → Virtual Finish Time 계산
2. **가장 낮은 Virtual Finish Time** 요청 먼저 처리
3. 처리 완료 후 Virtual Time 업데이트

**가중치의 효과**:
- 가중치 높음 (Enterprise, 100) → 낮은 Virtual Finish Time → 우선 처리
- 가중치 낮음 (Free, 1) → 높은 Virtual Finish Time → 후순위, **but 기아 방지**

**시각 자료**: WFQ Virtual Time 다이어그램

### 슬라이드 10: Jain's Fairness Index

**핵심 메시지**: 공정성을 0~1 사이 수치로 정량화

**Jain's Fairness Index** (Jain, Chiu, Hawe, 1984):
```
J(x1, x2, ..., xn) = (Sigma xi)^2 / (n x Sigma xi^2)

xi: i번째 테넌트가 받은 자원의 양
n: 활성 테넌트 수
```

**해석**:
| Jain's Index | 의미 |
|--------------|------|
| J = 1.0 | 완벽한 공정성 (모든 테넌트 동일 자원) |
| J = 1/n | 최악의 불공정 (한 테넌트가 모든 자원 독점) |
| J >= 0.9 | 우수한 공정성 (목표값) |

**예시 계산**:
```
4개 테넌트 처리량: [10, 8, 9, 7]

sum = 34
sum_squared = 294
J = (34)^2 / (4 x 294) = 1156 / 1176 = 0.983 (우수)
```

**시각 자료**: Jain's Fairness Index 계산 과정 다이어그램

### 슬라이드 14: 기아 방지 메커니즘

**핵심 메시지**: Aging과 Boosting으로 기아 현상 방지

**기아(Starvation) 현상**:
- 낮은 우선순위 테넌트가 무기한 대기하는 현상
- SLA 위반 및 고객 이탈 원인

**Aging 메커니즘** (Priority Scheduler):
```
Priority_new = Priority_current + (wait_time / aging_threshold)

설정:
- Aging 간격: 10초마다 확인
- Aging 임계값: 30초 이상 대기 시 우선순위 상승
- 최대 상승 횟수: 3회 (LOW -> NORMAL -> HIGH -> URGENT)
```

**Boosting 메커니즘** (MLFQ Scheduler):
```
설정:
- Boosting 주기: 30초마다
- 동작: 모든 큐의 작업을 Q0(최고 우선순위)로 이동
```

**WFQ의 자동 기아 방지**:
- Virtual Time 기반으로 모든 요청에 Virtual Finish Time 부여
- 가중치가 낮아도 결국 처리됨 → 별도 메커니즘 불필요

**시각 자료**: 기아 방지 메커니즘 흐름도

**Virtual Finish Time 계산 로직**:

### 슬라이드 17: FairnessCalculator 구현

**핵심 메시지**: Jain's Fairness Index 실시간 계산

**FairnessCalculator 핵심 로직**:
```javascript
function calculateJainsFairnessIndex(throughputs) {
  const n = throughputs.length;
  const sum = throughputs.reduce((acc, val) => acc + val, 0);
  const sumOfSquares = throughputs.reduce((acc, val) => acc + val * val, 0);

  const jfi = (sum * sum) / (n * sumOfSquares);
  return Math.min(1.0, jfi);
}
```

**공정성 메트릭**:

| 메트릭 | 설명 | 목표값 |
|--------|------|--------|
| jainsFairnessIndex | 전체 공정성 지수 | >= 0.9 |
| fairnessScore | 공정성 점수 (0-100) | >= 90 |
| mostFavoredTenant | 가장 유리한 테넌트 | - |
| leastFavoredTenant | 가장 불리한 테넌트 | - |
| disparityRatio | 처리량 격차 비율 | 가중치 비율 이내 |

**시각 자료**: FairnessCalculator 계산 흐름도

## 예상 Q&A

### Q1: "왜 Customer 관점이 아니라 Provider 관점인가요?"

**A**: Provider 관점은 멀티테넌트 환경에서 더 중요한 문제를 다룹니다.
- Customer 관점: 개인 사용자의 비용 최적화
- Provider 관점: 여러 고객사에 공정한 서비스 제공
- Provider가 공정성을 보장하지 못하면 SLA 위반, 고객 이탈 발생

### Q2: "Jain's Fairness Index 목표가 0.95인데 Free 테넌트는 0.92 아닌가요?"

**A**: 공정성 지수는 두 가지 관점에서 측정됩니다:

1. **개별 테넌트 수준 (목표: 0.90+)**: 동일 등급 내 테넌트 간 공정성
   - Free 등급의 0.92는 목표(0.90+)를 달성
   - 이는 Free 등급 내 테넌트 A, B, C 간의 공정성입니다

2. **가중치 정규화 (목표: 0.95+)**: 처리량/가중치 비율의 공정성
   - 전체 시스템에서 0.95 달성
   - 이는 가중치에 비례한 공정 분배를 의미합니다

3. **전체 시스템 (0.89)**: 절대 처리량 비교
   - 이 값이 1보다 낮은 것은 가중치 차이(100:50:10:1)가 의도대로 반영된 정상 결과입니다

### Q3: "테넌트 가중치 100:50:10:1은 어떻게 결정했나요?"

**A**: 업계 사례와 비즈니스 논리를 기반으로 결정:
- Enterprise: Free 대비 100배 비용 지불 → 100배 가중치
- GPS 이론에 따라 가중치 비율 = 자원 분배 비율
- 극단적 차이(1000:1)는 기아 유발 가능성으로 회피

### Q4: "WFQ가 FCFS보다 Enterprise 테넌트에게 불리한 것 아닌가요?"

**A**: 맞습니다. WFQ에서 Enterprise 대기 시간이 약간 증가합니다.
- FCFS: 48.3ms → WFQ: 52.1ms (+7.9%)
- 그러나 이는 **공정성을 위한 의도된 trade-off**입니다.
- Enterprise는 여전히 62%의 자원을 받으며, 다른 테넌트의 기아를 방지합니다.
