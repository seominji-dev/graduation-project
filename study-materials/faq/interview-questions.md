# 예상 면접 질문 및 답변

## 프로젝트 개요 질문

### Q1: 이 프로젝트를 한 문장으로 설명해주세요.
**A:** 운영체제의 프로세스 스케줄링 알고리즘(FCFS, Priority, MLFQ, WFQ)을 LLM API 요청 관리에 적용하여, API 비용을 절감하고 응답 시간을 최적화하는 시스템입니다.

### Q2: 왜 이 프로젝트를 선택했나요?
**A:** 
1. OS 수업에서 배운 스케줄링 이론을 실제로 적용해보고 싶었습니다.
2. ChatGPT, Claude 같은 LLM API가 비싸서, 효율적으로 사용하는 방법이 필요했습니다.
3. 이론과 실무를 연결하는 의미 있는 프로젝트라고 생각했습니다.

### Q3: 비슷한 프로젝트가 있나요?
**A:** AWS Lambda, Google Cloud Functions 같은 서버리스 플랫폼에서 요청 스케줄링을 하지만, LLM API에 특화된 OS 알고리즘 적용은 새로운 접근입니다.

## 기술 질문

### Q4: MLFQ와 Priority 스케줄링의 차이점은?
**A:**
- **Priority:** 고정된 우선순위, 사용자가 지정
- **MLFQ:** 동적 우선순위, 시스템이 작업 특성을 보고 자동 조정

MLFQ가 더 유연하고 Starvation 문제를 해결합니다.

### Q5: WFQ에서 Virtual Time은 왜 필요한가요?
**A:** Virtual Time은 공정성을 수학적으로 보장하기 위한 개념입니다.
- 실제 시간 대신 가상 시간으로 진행 상황을 추적
- 가중치가 2배인 테넌트는 2배 빠르게 Virtual Time이 증가
- Virtual Time이 가장 작은(덜 서비스받은) 테넌트를 먼저 처리

### Q6: 테스트 커버리지 98%는 어떻게 달성했나요?
**A:**
1. 단위 테스트: 각 스케줄러 알고리즘별 독립 테스트
2. 통합 테스트: API 엔드포인트 테스트
3. 엣지 케이스: 빈 큐, 동시 요청, 오버플로우 등

## 성능/결과 질문

### Q7: 어떤 성능 개선이 있었나요?
**A:**
- MLFQ: 평균 대기 시간 40% 감소 (짧은 요청 우선 처리)
- WFQ: 테넌트 간 공정성 지수 0.95 달성 (1.0이 완벽한 공정)
- 전체: API 호출 비용 예측 가능성 향상

### Q8: 왜 Ollama(로컬 LLM)를 주로 사용했나요? OpenAI나 Claude API가 더 좋지 않나요?
**A:** Ollama를 주요 실험 환경으로 선택한 것은 학술적으로 타당한 결정입니다:

**Ollama 선택 이유:**
1. **재현성 (Reproducibility)**: 학술 연구의 핵심 요소입니다. 로컬 환경에서 동일 조건으로 반복 실험이 가능하며, 외부 API 상태에 영향받지 않습니다.
2. **통제된 환경**: 네트워크 지연 없이 순수 스케줄링 알고리즘 성능만 측정할 수 있습니다.
3. **비용 효율성**: 777개 테스트를 수백 번 실행해도 비용이 발생하지 않습니다.
4. **학술 표준**: CS 연구에서는 통제된 환경 실험 후 실제 환경 검증을 순차적으로 진행하는 것이 일반적입니다.

**OpenAI/Claude API와의 차이점:**
- 네트워크 지연: Ollama 0ms vs OpenAI 100-500ms (가변적)
- Rate Limiting: Ollama 무제한 vs OpenAI 분당 요청 수 제한
- 응답 시간 변동성: Ollama 일정 vs OpenAI 시간대별 변동

**설계의 확장성:**
시스템은 Strategy Pattern 기반 Provider 추상화가 적용되어 있어, 후속 연구에서 다양한 LLM 제공자 환경으로 확장 연구가 가능합니다.

### Q8-1: 그래도 실제 API로 테스트해야 하는 것 아닌가요?
**A:** 클라우드 API 환경에서의 추가 실험은 향후 연구 과제입니다. 본 연구의 핵심 학술적 기여는:
- **알고리즘 유효성 검증**: 4개 OS 스케줄링 알고리즘이 LLM 요청 관리에 적용 가능함을 실증
- **통제된 비교 분석**: 외부 변수를 제거하고 알고리즘 자체의 성능 특성 비교
- **재현 가능한 연구**: 777개 테스트로 실험 결과 재현 가능

학술 연구에서 통제된 환경 실험 후 실제 환경 검증을 순차적으로 진행하는 것이 표준적인 방법론입니다.

## 어려웠던 점/배운 점

### Q9: 가장 어려웠던 점은?
**A:**
1. **알고리즘 이해:** MLFQ의 Queue 간 이동 로직, WFQ의 Virtual Time 계산
2. **동시성 처리:** 여러 요청이 동시에 들어올 때 Race Condition 방지
3. **테스트 설계:** 비결정적인 타이밍을 가진 시스템 테스트

### Q10: 이 프로젝트를 통해 배운 점은?
**A:**
1. OS 이론이 실제로 어떻게 적용되는지 체험
2. TypeScript로 대규모 프로젝트 구조화
3. 테스트 주도 개발(TDD)의 중요성
4. 문서화와 코드 품질의 상관관계

## 심화 질문 (대학원/기업 면접)

### Q11: 분산 환경에서 이 시스템을 확장하려면?
**A:**
- Redis Cluster로 큐 상태 공유
- Consistent Hashing으로 요청 분배
- 스케줄러 노드 간 리더 선출 (Raft)

### Q12: 현재 시스템의 한계점은?
**A:**
1. 단일 노드에서만 동작 (분산 미지원)
2. 스케줄러 알고리즘 변경 시 재시작 필요
3. 실시간 모니터링 대시보드 미완성

### Q13: 후속 연구로 탐구하고 싶은 부분은?
**A:**
1. 분산 환경에서의 스케줄링 알고리즘 특성 비교 연구
2. 머신러닝 기반 워크로드 예측 및 알고리즘 자동 선택 연구
3. 실제 클라우드 LLM API 환경에서의 성능 비교 연구

### Q14: 분산 환경으로 확장할 수 있나요?
**A:** 이론적으로 분산 환경 확장이 가능하며, 이는 향후 연구 주제로 적합합니다:

**이론적 확장성 분석 (Theoretical Scalability Analysis):**

1. **Redis Cluster 활용 가능성**
   - 현재 BullMQ가 Redis Cluster를 네이티브 지원
   - 분산 환경에서의 스케줄링 알고리즘 특성 비교 연구 가능

2. **분산 스케줄링 연구 주제**
   - Consistent Hashing 기반 요청 분산 전략 연구
   - 분산 환경에서의 공정성(Fairness) 보장 문제

3. **합의 알고리즘 연구 연계**
   - Raft/Paxos 기반 스케줄러 상태 동기화 연구
   - CAP 정리와 스케줄링 일관성 트레이드오프 분석

현재 아키텍처의 인터페이스 기반 설계(IScheduler)는 이러한 후속 연구의 기반이 됩니다.

### Q15: 이 프로젝트의 핵심 학술적 기여는 무엇인가요?
**A:** 4가지 핵심 기여가 있습니다:

1. **OS 이론의 AI 도메인 확장**: CPU 스케줄링 알고리즘을 LLM API 요청 관리에 최초 적용
2. **LLM 특화 메커니즘 개발**: Aging(30초), Boost(60초) 등 기아 방지 메커니즘
3. **정량적 성능 비교**: 4개 알고리즘의 대기시간, 처리량, 공정성 실증 분석
4. **오픈소스 기여**: 777개 테스트와 함께 MIT License로 전체 코드 공개

이는 단순한 구현을 넘어, OS 이론의 범용성과 AI 시스템으로의 확장 가능성을 입증하는 연구입니다.
