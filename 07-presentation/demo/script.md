## [슬라이드 3] 연구 배경 · 연구 질문 · 약 1분

ChatGPT로 대표되는 LLM API는 다중 사용자가 동시에 접속하는 환경입니다.
그러나 현재 대부분의 LLM 서빙 도구에는 **요청 처리 순서를 결정하는 기준**이 없습니다.
그 결과 세 가지 문제가 발생합니다.
첫째, 유료 구독자와 무료 사용자 사이에 **등급에 따른 차등 처리가 이루어지지 않습니다**.
둘째, 한 사용자가 한꺼번에 많은 요청을 보내면 **뒤에 도착한 다른 사용자는 계속 기다리게 됩니다**.
셋째, 긴급한 요청과 일상적인 요청을 **구분할 방법이 없습니다**.

저는 이 문제를 **운영체제 스케줄링 이론**으로 접근해보기로 했습니다.
CPU 스케줄링에서 오랫동안 검증된 알고리즘을, LLM API 요청 관리에 적용할 수 있는지를 확인하는 것이 목표입니다.

이를 세 개의 연구 질문으로 정리했습니다.
RQ1은 **구현 가능성** — FCFS, Priority, MLFQ, WFQ 네 알고리즘을 LLM API 환경에 구현할 수 있는가.
RQ2는 **성능 비교** — 평균 대기시간과 처리량 면에서 알고리즘 간 효율성 차이가 있는가.
RQ3은 **공정성 분석** — JFI 지표로 사용자별 공정성을 어떻게 측정할 수 있는가.

(전환) 이 연구 질문을 풀기 위해 먼저 시스템 구조를 설계했습니다.

## [슬라이드 4] 시스템 아키텍처 · 약 1분

시스템은 4개 계층으로 구성됩니다.
**클라이언트 계층**은 REST API로 요청을 보냅니다.
**API 계층**은 Express.js 기반이며, 여기에 **Rate Limiter가 전처리 단계로 위치**합니다. 등급별 한도를 초과한 요청은 이 단계에서 HTTP 429로 거부됩니다.
**스케줄러 계층**은 네 개 알고리즘이 공통 인터페이스, enqueue와 dequeue를 구현합니다. 환경변수 `SCHEDULER_TYPE` 하나로 네 스케줄러를 실시간 전환할 수 있습니다.
**저장소·LLM 계층**은 메모리 큐로 요청을 관리하고, 최종적으로 로컬 Ollama에 요청을 전달합니다.

기술 스택은 Node.js 22, Express 4.18, JavaScript ES2024이며, 외부 라이브러리는 Express 하나만 사용했습니다.
스케줄링 알고리즘을 직접 구현해 학습 효과를 높이는 것이 목표였기 때문입니다.

(전환) 이제 네 가지 알고리즘이 어떻게 다른지 살펴보겠습니다.

## [슬라이드 5] 4개 알고리즘 비교 · 약 1분 30초

**FCFS, First-Come First-Served**는 도착 순서대로 처리하는 가장 단순한 방식입니다. 기준선 역할을 합니다.
**Priority**는 URGENT, HIGH, NORMAL, LOW 네 단계의 요청 우선순위를 기준으로 처리합니다.
**MLFQ, Multi-Level Feedback Queue**는 여러 큐를 두고, 짧은 요청은 상위 큐에 유지해 먼저 처리하고 긴 요청은 하위 큐로 강등하는 방식입니다.
**WFQ, Weighted Fair Queuing**은 본래 네트워크 QoS용 알고리즘으로, 가중치가 큰 사용자의 가상 완료 시각(VFT)이 천천히 증가하여 결과적으로 더 자주 선택됩니다. 저희 프로젝트에서는 Enterprise 100, Premium 50, Standard 10, Free 1의 가중치를 부여했습니다.

(전환) 이 WFQ 가중치가 실제 어떤 등급 체계에 매핑되는지 보여드리겠습니다.

---

## [슬라이드 6] 등급 체계 · 약 1분

등급은 두 축으로 구성됩니다.
왼쪽의 **요청 우선순위**는 요청 자체의 긴급도를 나타내며, Priority Scheduler에서 사용됩니다.
오른쪽의 **구독 등급**은 사용자 분류이며, WFQ의 가중치와 Rate Limiter의 분당 한도에 동시에 적용됩니다.

수치를 보시면, Enterprise는 Free보다 **처리 가중치는 100배, 분당 요청 한도는 20배** — 5건 대 100건 — 로 설계되어 있습니다.
이 설계가 뒤에 보실 실험에서 **약 12배의 대기시간 차이**로 나타납니다.

(전환) 이제 실험 결과로 넘어가겠습니다. 네 가지 실험 중 첫 번째입니다.

## [슬라이드 7] 실험 1 — 500건 평균 대기시간 · 약 1분

첫 번째 실험은 네 알고리즘을 **동일한 500건 요청**에 적용한 결과입니다.
WFQ가 약 **11,846밀리초**로 가장 짧은 평균 대기시간을 보였고, FCFS와 MLFQ 비선점형이 **모두 12,203밀리초로 동일**했습니다.
여기서 중요한 발견이 있습니다. MLFQ 비선점형이 FCFS와 정확히 같은 값이 나온 이유는, **모든 요청이 최상위 큐 Q0의 할당 시간인 1,000밀리초 안에 완료되어 강등이 일어나지 않았고**, 비선점형이라 꺼내는 순서가 도착 순서와 같아졌기 때문입니다.
즉, 이 실험은 MLFQ의 이론적 효과를 보여주지 못했습니다.

(전환) 그래서 MLFQ의 본래 장점을 확인하기 위해 선점형을 별도 실험으로 수행했습니다.

## [슬라이드 8] 실험 2 — MLFQ 선점형 · 약 1분

MLFQ의 이론적 효과를 검증하기 위해 선점형을 추가 구현했습니다.
요청 500건, 처리 시간 최대 10초의 버스트 도착 구조로 5개 시드에 걸쳐 반복했습니다.

결과는 명확했습니다.
**짧은 요청은 대기시간이 약 73% 감소** — 635초에서 170초로 — 했고, 5개 시드 전체에서 71.8%~74.3% 범위로 일관되게 재현되었습니다.
**반면 긴 요청은 약 89% 증가**했고, 중간 길이 요청은 약 13% 증가했습니다.
이는 MLFQ가 짧은 요청 중심 서비스에는 매우 효과적이지만, **긴 요청이 많은 환경에서는 주의가 필요함**을 보여줍니다. Rule 5 부스팅으로 기아 상태는 방지됩니다.

단 한 가지 실용적 한계가 있습니다. **선점형 MLFQ는 LLM 추론 도중 중단이 필요한데, 대부분의 LLM 서빙 도구는 추론 중단을 지원하지 않습니다**. 따라서 실서버에는 적용할 수 없습니다.

(전환) 다음은 WFQ의 차등 서비스 실험입니다.

## [슬라이드 9] 실험 3 — WFQ 차등 서비스 · 약 1분

WFQ 실험을 구독 등급별로 분해한 결과입니다.
**Enterprise 1,862밀리초, Premium 8,384밀리초, Standard 15,109밀리초, Free 22,029밀리초**.
Enterprise가 Free 대비 **약 12배 짧은 대기시간**을 보였고, 이는 가중치 100:50:10:1 비율에 대응되는 결과입니다.

가중치 비율은 100대 1인데 실제 차이가 12배에 그친 이유는 **요청이 순차적으로 큐에 도착하는 실험 설계**의 영향입니다. 동시 경쟁이 극단적으로 일어나지 않은 것이죠.
그럼에도 FCFS 계열이 모두 균등했던 앞 실험과 대조적으로, **WFQ는 의도적인 차등 서비스를 수치로 제공함**을 확인할 수 있습니다.

(전환) 시뮬레이션만으로는 부족하니, 실제 LLM 환경에서도 재현되는지 검증했습니다.

## [슬라이드 10] 실험 4 — Ollama 실서버 검증 · 약 1분

시뮬레이션 결과가 실제 LLM에서도 재현되는지 확인하기 위해 Ollama 로컬 서버로 검증했습니다.
**Google Gemma 4 계열의 gemma4:e4b 모델**, 등급별 5건씩 **총 20건**을 동시에 큐에 투입한 뒤 FCFS, Priority, WFQ 세 알고리즘을 비교했습니다.

세 알고리즘 모두 **평균 처리 시간이 거의 동일**했습니다.
이는 핵심 발견입니다. **스케줄러는 "누구를 먼저 처리할지"만 결정할 뿐, LLM 추론 속도 자체를 바꾸지 못합니다.**
등급별 대기시간 패턴은 시뮬레이션과 동일하게 나타났습니다. FCFS는 도착 순서대로, Priority는 우선순위대로, WFQ는 가중치에 따라 처리되었습니다.
이 실험에서 또 한 가지, **선점형 MLFQ는 실서버에 적용할 수 없다**는 사실이 재확인되었습니다.

(전환) 이상의 네 실험을 바탕으로 세 연구 질문에 답하겠습니다.

## [슬라이드 11] 결론 · 한계 · 향후 과제 · 약 1분 30초

세 연구 질문 모두에 답할 수 있었습니다.

**RQ1 구현 가능성**: FCFS, Priority, MLFQ, WFQ 네 알고리즘을 모두 구현하고, Ollama 실서버에서도 검증을 마쳤습니다.

**RQ2 성능 비교**: 비선점형 환경에서는 알고리즘 간 처리량이 유사했습니다. 다만 **WFQ가 평균 대기시간은 가장 짧았고**, MLFQ 선점형은 짧은 요청에 약 73% 감소라는 뚜렷한 효과를 보였습니다.

**RQ3 공정성**: FCFS, Priority, 비선점형 MLFQ의 JFI는 모두 **1.000 — 완전 균등 배분**입니다. WFQ는 0.316으로 낮은 값이지만, 이는 **가중치 기반 의도적 차등**의 결과이며, Enterprise가 Free 대비 약 12배 빨리 처리됩니다.

한계는 두 가지입니다. 첫째, **시뮬레이션 기반**이라는 점으로 실제 대규모 부하와는 차이가 있을 수 있습니다. 둘째, **선점형 MLFQ는 실서버에 적용할 수 없습니다**.
향후 과제로는 **토큰 기반 비용 모델**, **대시보드 고도화(WebSocket 기반 실시간 갱신)**, **동일 등급 내 공정성 검증**, **대규모 부하 실험** 네 가지를 남깁니다.

(전환) 이상으로 이론 발표를 마치고, 실제로 시스템이 어떻게 동작하는지 데모로 보여드리겠습니다.

## [Q&A 대응 포인트]

자주 나올 질문과 준비된 답변

1. **"WFQ가 가장 빨랐는데 차이가 크지 않은 이유는?"**
   → 500건 실험에서 FCFS 12,203ms 대비 WFQ 11,846ms로 약 3% 차이. 모든 알고리즘이 같은 총 처리 시간을 공유하고 비선점형이라 총 대기 시간의 총합은 일정하기 때문. WFQ의 진짜 가치는 **등급별 분배 차이**에 있으며 이는 실험 3에서 12배 차이로 드러납니다.

2. **"MLFQ 선점형을 왜 실서버에서 검증하지 않았나?"**
   → Ollama를 비롯한 대부분의 LLM 서빙 도구가 추론 중단을 지원하지 않습니다. 이는 실용적 제약이며, 본 연구의 기여 중 하나로 보고합니다.

3. **"실서버 실험이 20건뿐인데 일반화 가능한가?"**
   → 본 실험의 목적은 "시뮬레이션 결과의 정성적 재현 여부 확인"입니다. 정량적 일반화는 5.6절 한계와 향후 과제에 명시했습니다.

4. **"토큰 기반 비용 모델이란?"** (향후 과제 관련)
   → 실제 LLM 요금이 입력·출력 토큰 수에 비례하므로, WFQ의 "처리 비용"을 단순 건수가 아닌 토큰 수로 환산해 더 정교한 차등을 구현할 수 있습니다.

5. **"교수님이 시나리오 3(Ollama 실호출)을 보고 싶다고 하시면?"**
   → `demo-scenario.md` 시나리오 3 (2~3분) 실행. 12건 축소 재현이며, 대기시간이 등급별로 차등임을 즉석에서 재현 가능.
