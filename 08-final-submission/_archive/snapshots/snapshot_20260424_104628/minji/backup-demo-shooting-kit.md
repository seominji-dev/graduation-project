# 백업 녹화본 촬영 콘티

> 이 문서의 목적: 발표장에서 **실시간 데모가 실패할 때를 대비해** 미리 촬영할 백업 영상의 콘티와 절차.
> 결과물: `07-presentation/demo/backup-demo.mov` (해상도 1080p 권장, 오디오 없음)
> 촬영 시기: 발표 **1주 전 ~ 3일 전**

---

## 🎬 촬영 목적

- **실시간 데모가 실패할 때 전환하는 보험**
- 발표 중 "잠시만요, 준비된 녹화본으로 보여드리겠습니다" 한 멘트 후 재생
- **자막은 없어도 됨** (본인이 실시간으로 구두 설명)
- **길이 목표**: 3분 ~ 3분 30초 이내

---

## 📹 촬영 준비물

- [ ] 노트북 (Ollama + 서버 코드 포함)
- [ ] QuickTime Player (macOS 기본 앱)
- [ ] 조용한 촬영 환경 (소음 없는 방)
- [ ] Ollama 사전 warmup 완료 (`ollama run gemma4:e4b "hi"` → `/bye`)
- [ ] 터미널 폰트 18pt 이상으로 크게
- [ ] 대시보드는 시크릿 창에서 열기

---

## 🎯 콘티 A — 필수 녹화 (시나리오 1, 시나리오 2)

### Take 1-A — 시나리오 1 (FCFS→WFQ 순서 비교, 약 2분 30초)

**0:00 ~ 0:15** — 초기 환경 (화면)
- 터미널 1에 `SCHEDULER_TYPE=FCFS npm start` 실행
- "FCFS 스케줄러 시작됨" 출력 보임
- 브라우저 대시보드 `http://localhost:3000/dashboard.html` 로드 (빈 상태 확인)

**0:15 ~ 0:45** — 요청 전송 (터미널 2)
- 4개 등급 × 1건씩 총 4건을 curl로 전송 (`demo-scenario.md` 시나리오 1-2 참조)
- 각 curl 간 1~2초 간격 (시연 효과)
- 대시보드 전환해 대기열에 4건 쌓인 모습 보여주기

**0:45 ~ 1:15** — FCFS 처리 순서 (대시보드)
- "다음 처리" 버튼 4회 연속 클릭
- 처리 순서가 **도착 순서 (Enterprise → Premium → Standard → Free)** 와 같은지 확인
- 대기열 → 처리 중 → 완료로 이동하는 모습

**1:15 ~ 1:30** — FCFS 종료 및 WFQ 전환
- 터미널 1에서 `Ctrl+C`로 서버 중단
- `SCHEDULER_TYPE=WFQ npm start` 실행
- "WFQ 스케줄러 시작됨" 출력

**1:30 ~ 2:00** — 같은 4건 다시 전송
- 같은 curl 4개를 다시 실행
- 대시보드에 4건 쌓임

**2:00 ~ 2:30** — WFQ 처리 순서 (핵심!)
- "다음 처리" 버튼 4회
- 처리 순서가 **Enterprise → Premium → Standard → Free** (가중치 순)으로 나오는지 확인
- FCFS와 비교해 **확실히 다르다**는 것이 눈에 보여야 함

**촬영 중 확인사항**:
- [ ] 터미널 출력이 읽힘
- [ ] 대시보드의 테넌트 컬럼 (Enterprise/Premium 등)이 또렷
- [ ] 처리 순서가 틀림없이 다르게 나옴 (WFQ가 잘못 설정되면 FCFS와 같은 순서 나올 수 있음)

### Take 2-A — 시나리오 2 (Rate Limiter 429, 약 1분)

**0:00 ~ 0:15** — Free 등급으로 6건 요청 준비
- 터미널 2에 미리 6개 curl 명령 준비 (같은 Free 등급에서 빠르게 연속 호출)

**0:15 ~ 0:45** — 6개 연속 호출
- 6개 요청을 1초 간격으로 연속 전송
- 터미널 출력에서 6번째 요청이 **HTTP 429 Too Many Requests** 받는 모습 확인

**0:45 ~ 1:00** — 대시보드 확인
- 대기열에는 5건만 있음 (6번째는 거부됐으므로)
- 거부된 요청은 처리되지 않음

**촬영 중 확인사항**:
- [ ] 429 응답이 또렷이 보여야 함
- [ ] 대시보드에서 거부된 요청이 대기열에 안 들어간 것을 확인

## 🎯 콘티 B — 옵션 녹화 (시나리오 3, 교수님 요청 시)

### Take 3-B — Ollama 실서버 축소 재현 (약 2~3분)

**0:00 ~ 0:30** — 환경 확인
- 터미널 3에서 `ollama serve` 실행 중임 확인
- 터미널 1에서 `SCHEDULER_TYPE=WFQ npm start` 실행
- "Ollama 연결 확인" 메시지 출력 확인

**0:30 ~ 1:30** — 12건 축소 실험
- 등급별 3건씩 총 12건 전송 (빠르게 연속)
- `curl` 명령어는 `demo-scenario.md` 시나리오 3 참조
- 대시보드에 12건 쌓임

**1:30 ~ 2:30** — LLM 실제 호출 + 대기시간 차등
- "다음 처리" 버튼을 빠르게 클릭 (자동 처리 모드면 생략)
- 각 요청이 Ollama에 전달되어 gemma4:e4b에서 실제 응답 생성
- 대시보드에서 **Enterprise가 먼저 완료, Free가 나중** 확인

**2:30 ~ 3:00** — 결과 요약
- 각 등급의 평균 대기시간 대시보드 출력
- Enterprise < Premium < Standard < Free 순서 확인

**촬영 중 확인사항**:
- [ ] 실제 LLM 응답이 나오는지 (`처리 완료: gemma4:e4b 응답` 메시지 등)
- [ ] Ollama 서버 로그에도 요청 기록 남는지

## 🎥 촬영 기법

### QuickTime Player 사용

1. **새로운 화면 기록 열기**: `QuickTime Player 열기` → `파일 → 새로운 화면 기록` 또는 `Cmd + Ctrl + N`
2. **녹화 설정**:
   - 마이크 **OFF** (오디오 없음)
   - 클릭 위치 **ON** (마우스 클릭 시각적 표시)
3. **녹화 영역**: "전체 화면" 선택 (또는 사용할 영역만)
4. **녹화 시작**: 녹화 버튼 클릭 → 원하는 영역 클릭 (전체 화면이면 아무 곳이나)
5. **녹화 종료**: 메뉴 바의 녹화 아이콘 클릭 또는 `Cmd + Ctrl + Esc`
6. **저장**: `파일 → 저장` → `backup-demo.mov` 파일명

### 녹화 품질

- **해상도**: 1080p 또는 1440p (노트북 해상도에 따라)
- **프레임**: 30fps (충분)
- **오디오**: OFF (발표 시 본인이 구두 설명)

## 🔄 재촬영 기준

다음 경우 **다시 촬영**:

1. 터미널/대시보드 글자가 읽히지 않음 → 폰트 크기 재확인 후 재촬영
2. 스케줄링 순서가 예상대로 안 나옴 → 코드 재확인 후 재촬영
3. 5초 이상 공백 (로딩 시간) → 편집 또는 재촬영
4. 마우스 커서가 화면 밖을 오래 돌아다님 → 동선 개선
