## 보고서 장별 핵심 암기 포인트

### 1장 (서론)

- **연구 배경**: LLM API 서비스가 다중 사용자 환경에서 요청 관리 문제에 직면
- **문제**: 요청 처리 순서 기준 없음, 특정 사용자 독점, 구독 등급 차등 불가
- **연구 목적**: OS 스케줄링 알고리즘(FCFS, Priority, MLFQ, WFQ)을 LLM API에 적용
- **RQ 3개**: 구현 가능성(RQ1), 성능 비교(RQ2), 공정성 분석(RQ3)

### 2장 (관련연구)

- **FCFS**: 도착 순서 처리, 단순하나 우선순위 반영 불가
- **Priority**: 우선순위 기반, 에이징으로 기아 방지
- **MLFQ**: 4개 큐(Q0~Q3), 큐별 독립 시간 할당, 주기적 부스팅
- **WFQ**: Virtual Finish Time(VFT) 기반, 가중치에 비례 처리
- **JFI**: 공정성 지표, 1이면 완전 공평, 1/n이면 한 사용자 독점
- **Ollama**: 로컬 LLM 서빙 도구, 추론 중단 불가 → 선점형 불가

### 3장 (시스템 설계)

- **5계층**: 클라이언트 → API → Rate Limiter → 스케줄러 → 저장소/LLM
- **역할 분담**: Rate Limiter는 진입 결정, 스케줄러는 처리 순서 결정
- **등급 체계 이원화**:
  - 요청 우선순위: URGENT/HIGH/NORMAL/LOW (Priority Scheduler용)
  - 구독 등급: Enterprise/Premium/Standard/Free (WFQ·Rate Limiter용)

### 4장 (구현)

- **BaseScheduler 인터페이스**: `enqueue(request)`, `dequeue()` 두 메서드
- **MLFQ 큐별 독립 할당량**: Q0에서 2초 쓰고 Q1로 강등되면 Q1에서는 4초 새로 시작
- **WFQ VFT 비용=1 고정**: LLM 응답 토큰을 미리 알 수 없어 실제 처리 시간 예측 불가
- **Rate Limiter (토큰 버킷)**: Free 5/분, Standard 10/분, Premium 50/분, Enterprise 100/분

### 5장 (실험 및 평가)

- **기본 실험**: 500건, 처리시간 10~100ms, 4개 등급 동일 비율
- **결과 1 (평균 대기시간)**: WFQ 약 11,846ms로 최소, MLFQ 비선점형 = FCFS
- **결과 2 (MLFQ 선점형)**: 짧은 요청 대기시간 약 73% 감소, 긴 요청 약 89% 증가
- **결과 3 (WFQ 차등)**: Enterprise 약 1,862ms vs Free 약 22,029ms (약 12배 차이)
- **결과 4 (Ollama 실서버)**: 처리시간 약 116~119ms, 스케줄러 순서 유지 확인

### 6장 (논의)

- **시뮬레이션 한계**: 실제 부하·네트워크 지연 미반영
- **실서버 실험 규모**: 20건 수준, 검증 목적이지 부하 시험은 아님
- **연속 배치 미적용**: 실제 LLM 서빙 도구(vLLM, TGI)는 연속 배치를 쓰므로 스케줄링 효과가 달라질 수 있음
- **알고리즘별 적합 상황**:
  - FCFS: 단순 기본값
  - Priority: 긴급 요청이 있는 환경
  - MLFQ: 짧은 요청 위주 혼합 환경
  - WFQ: 구독 등급 차등 서비스가 필요한 환경

### 7장 (결론)

- **RQ1 답**: 4개 알고리즘 모두 구현 완료, Ollama 실서버 연동까지 검증
- **RQ2 답**: WFQ가 평균 대기시간 최소, MLFQ 선점형은 짧은 요청에 유리
- **RQ3 답**: FCFS·Priority·MLFQ는 JFI ≈ 1.0 (공평), WFQ는 JFI ≈ 0.32 (의도적 차등)
- **향후 연구 3가지**: 연속 배치 환경, 대규모 부하 시험, 토큰 수 예측 기반 VFT 개선

## 슬라이드별 발표 멘트 연습 체크리스트

각 슬라이드를 **입으로 소리 내어** 2~3회 반복한 뒤 체크한다.

- [ ] 슬라이드 1 (표지) — 이름·학과·연구 제목 차분하게 소개
- [ ] 슬라이드 2 (목차) — 5개 섹션을 손가락으로 짚듯 간결하게
- [ ] 슬라이드 3 (연구 배경) — 문제 상황 4개 → 제안 접근법 → RQ 3개 순
- [ ] 슬라이드 4 (아키텍처) — 5계층을 위에서 아래로 짚으며 데이터 흐름 설명
- [ ] 슬라이드 5 (알고리즘 비교) — 4개 박스를 왼쪽 위·오른쪽 위·왼쪽 아래·오른쪽 아래 순으로
- [ ] 슬라이드 6 (등급 체계) — 왼쪽(요청 우선순위) → 오른쪽(구독 등급) 순으로, "다른 체계임"을 강조
- [ ] 슬라이드 7 (실험 1) — WFQ 최소, MLFQ=FCFS 결과 주목
- [ ] 슬라이드 8 (MLFQ 선점형) — 짧은 요청 73% 감소, 긴 요청 89% 증가 트레이드오프 강조
- [ ] 슬라이드 9 (WFQ 차등) — Enterprise vs Free 약 12배 차이, 가중치 비율 일관성 강조
- [ ] 슬라이드 10 (Ollama 실서버) — 처리시간은 LLM이 결정, 대기시간은 스케줄러가 결정
- [ ] 슬라이드 11 (결론) — RQ 3개에 순차 답변, 한계 인정
- [ ] 슬라이드 12 (데모) — "지금부터 실시간으로 보여드리겠습니다" 후 터미널로 전환

## Q&A 답변 연습 (11문항)

demo-scenario.md의 "Q&A 예상 질문 및 답변 포인트"를 참조하여 각 질문에 대해 다음을 연습한다:

- [ ] Q1~Q11 각 질문을 **소리 내어 읽고** → 답변을 **입으로 말하기** (2회씩)
- [ ] 답변 길이가 15~30초 범위인지 확인 (너무 길면 지루함, 너무 짧으면 성의 없어 보임)
- [ ] 모르는 것은 "그 부분은 본 연구에서 다루지 못한 한계입니다"로 솔직히 인정하는 연습
- [ ] 보고서 절 번호(예: "5.4절", "6장")를 답변에 포함시키는 연습
