**주의**: 이 슬라이드에서 청중의 "왜"가 결정된다. 문제 3개는 천천히 강조.

## 🎴 Card 04 — 시스템 아키텍처

**핵심 메시지**: 4계층 구조 + Rate Limiter 위치 + 환경변수로 스케줄러 전환.

**필수 말해야 할 것** (1분):
1. **4계층**: 클라이언트 → API(Rate Limiter 포함) → 스케줄러 → 저장소/LLM
2. **Rate Limiter**: 스케줄러 전 단계, 등급별 한도 초과 시 HTTP 429
3. **스케줄러 전환**: 환경변수 `SCHEDULER_TYPE` 하나로 4개 알고리즘 즉시 전환
4. **기술 스택**: Node.js 22, Express 4.18, ES2024, 외부 라이브러리 Express만

**암기 단어**: 4계층 / Rate Limiter / 429 / SCHEDULER_TYPE / Express 하나만

**이것만 말해도 OK**: "네 계층 구조이고, 환경변수 하나로 네 스케줄러를 전환합니다."

**주의**: "Rate Limiter는 스케줄링이 아닌 전처리"라는 점이 포인트.

## 🎴 Card 05 — 4개 알고리즘 비교

**핵심 메시지**: 각 알고리즘의 핵심 한 줄씩.

**필수 말해야 할 것** (1분 30초):
1. **FCFS**: 도착 순서대로 (기준선)
2. **Priority**: URGENT/HIGH/NORMAL/LOW 4단계 우선순위
3. **MLFQ**: 여러 큐, 짧은 요청 위, 긴 요청 아래
4. **WFQ**: 가상 완료 시각(VFT) 기반 가중치 처리. Enterprise 100 / Premium 50 / Standard 10 / Free 1

**암기 단어**: 도착순 / 우선순위 / 짧은거위 긴거아래 / VFT / 100:50:10:1

**이것만 말해도 OK**: "FCFS는 도착순, Priority는 긴급도, MLFQ는 짧은요청우대, WFQ는 가중치기반입니다."

**주의**: WFQ의 VFT 언급 시 **"가상 완료 시각"**이라고 한글 풀이 병기. 가중치 100:50:10:1은 외워둘 것.

## 🎴 Card 06 — 등급 체계

**핵심 메시지**: **두 축**(요청 우선순위 vs 구독 등급)이 다른 개념임.

**필수 말해야 할 것** (1분):
1. **왼쪽 축 = 요청 우선순위**: URGENT/HIGH/NORMAL/LOW → Priority 스케줄러가 사용
2. **오른쪽 축 = 구독 등급**: Enterprise/Premium/Standard/Free → WFQ 가중치 + Rate Limiter 모두에 사용
3. **수치**: Enterprise는 Free보다 가중치 100배, 분당 한도 20배 (5건 ↔ 100건)
4. **미리 예고**: "이 설계가 나중에 **약 12배** 대기시간 차이로 나타납니다"

**암기 단어**: 두 축 / 요청 우선순위 / 구독 등급 / 100배 / 20배 / 12배

**이것만 말해도 OK**: "요청 우선순위와 구독 등급은 다른 축이고, Enterprise가 Free보다 가중치 100배입니다."

**주의**: 두 축이 다르다는 걸 손동작으로 표현하면 좋다 (좌우로 가리키기).

## 🎴 Card 07 — 실험 1: 500건 평균 대기시간

**핵심 메시지**: WFQ가 최소, MLFQ 비선점형 = FCFS. 왜 같은가를 설명.

**필수 말해야 할 것** (1분):
1. **수치**: WFQ **11,846ms**, FCFS = MLFQ 비선점 **12,203ms**
2. **MLFQ = FCFS인 이유**:
   - 요청 처리 시간이 10~100ms로 짧음
   - 모두 Q0의 1초 할당 안에 끝남 → **강등 없음**
   - 비선점형 → 도착 순서 그대로 유지
3. **결론**: 이 실험에서는 MLFQ의 이론적 장점 안 보임

**암기 단어**: 11,846 / 12,203 / 강등 없음 / 비선점

**이것만 말해도 OK**: "짧은 요청뿐이어서 MLFQ가 FCFS와 같아졌습니다."

**주의**: 이 슬라이드에서 "왜 같은 값이 나왔나"의 설명이 교수님 관심 포인트. 차분하게 풀어서 설명.

## 🎴 Card 08 — 실험 2: MLFQ 선점형

**핵심 메시지**: 짧은 요청 73% 감소, 긴 요청 89% 증가, 실서버 적용 불가.

**필수 말해야 할 것** (1분):
1. **실험 조건**: 500건 × 5 시드, 처리 시간 최대 10초, 버스트 도착
2. **결과**:
   - Short: **약 73% 감소** (635초 → 170초)
   - Medium: 약 13% 증가
   - Long: 약 89% 증가
   - 시드 5개 모두 71.8~74.3% 범위 (우연이 아님)
3. **한계**: 선점형은 LLM 추론 중단 불가 → 실서버 적용 불가

**암기 단어**: 73% 감소 / 89% 증가 / 5 시드 / 선점 불가

**이것만 말해도 OK**: "짧은 요청은 73% 빨라졌지만, 긴 요청은 89% 느려졌고, 실서버에는 적용 불가입니다."

**주의**: "부스팅(Rule 5)"을 말할 때는 "**아래 큐에 오래 있던 요청을 위로 다시 올려주는 기능**"으로 풀이 병기.

## 🎴 Card 09 — 실험 3: WFQ 차등 서비스

**핵심 메시지**: Enterprise 약 12배 빠름. 100:1 가중치가 왜 12배에 그쳤는지 설명.

**필수 말해야 할 것** (1분):
1. **4개 숫자**: Enterprise 약 **1,900** / Premium 약 **8,400** / Standard 약 **15,000** / Free 약 **22,000** ms (정확치는 슬라이드)
2. **배율**: Enterprise가 Free의 **약 12배** 빠름
3. **100:1이 12배에 그친 이유**: 요청이 순차적으로 큐에 도착하는 설계 → 극단적 동시 경쟁 없음

**암기 단어**: 12배 / 1,900 - 8,400 - 15,000 - 22,000 / 순차 도착

**이것만 말해도 OK**: "Enterprise가 Free보다 약 12배 빠르게 처리됐습니다. 가중치 기반 차등이 숫자로 확인됐습니다."

**주의**: 4개 숫자를 말할 때 **"약 2천 / 8천 / 1만5천 / 2만2천"**으로 단순화해 말해도 된다 (슬라이드에 정확한 숫자 있음).

## 🎴 Card 10 — 실험 4: Ollama 실서버 검증

**핵심 메시지**: 스케줄러는 순서만 바꿀 뿐 LLM 속도는 못 바꾼다 (처리 시간 208~209ms로 동일).

**필수 말해야 할 것** (1분):

2. **실험**: 20건 (등급별 5건), FCFS/Priority/WFQ 3개 비교
3. **핵심 발견**: 세 알고리즘 평균 처리 시간 약 **208~209ms로 동일**
4. **해석**: 스케줄러는 처리 순서만 결정, LLM 추론 속도는 그대로

## 🎴 Card 11 — 결론·한계·향후 과제

**핵심 메시지**: RQ 3개에 답 + 한계 2개 + 향후 4개.

**필수 말해야 할 것** (1분 30초):
1. **RQ1 구현 가능성**: 네 알고리즘 모두 구현, 실서버까지 검증
2. **RQ2 성능 비교**: WFQ가 대기시간 최소, MLFQ 선점형은 Short 약 73% 감소
3. **RQ3 공정성 (JFI 정의 포함)**:
   - JFI = Jain's Fairness Index
   - 1이면 완전 공평, 1/n이면 독점
   - FCFS·Priority·MLFQ 비선점 = **1.000**
   - WFQ = **0.316** (의도적 차등)
4. **한계 2개**: 시뮬레이션 기반, 선점형 실서버 불가
5. **향후 과제 4개**: 토큰 기반 비용 모델 / 대시보드 고도화 / 동일 등급 내 공정성 / 대규모 부하

**암기 단어**: RQ1 구현 / RQ2 WFQ·73% / RQ3 1.000 0.316 / 한계 2개 / 향후 4개

**이것만 말해도 OK**: "네 알고리즘 모두 구현했고, WFQ의 차등 서비스와 MLFQ 선점형의 효과를 확인했으며, 공정성 지표도 수치로 분석했습니다."

**주의**: JFI 정의를 여기서 한 번 더 짚어줄 것. 앞에서는 RQ3으로만 언급됐을 수 있음.

## 🎴 특수 카드 — Q&A 진입 전략

### 질문이 들어왔을 때 3초 루틴

1. **듣기** (3초): 끝까지 듣고 요지 파악
2. **되묻기** (선택): "질문 요지가 ~인 것 맞으신가요?"로 확인
3. **답변**:
   - 알면: 보고서 절 번호 포함 답변 (예: "5.4절에 해당 내용이 있는데요...")
   - 모르면: "그 부분은 본 연구에서 다루지 못한 한계입니다"
4. **마무리**: "혹시 추가 설명이 필요하실까요?"

### 절대 금지

- "어... 그러니까..." 같은 망설임 표현
- "잘 모르겠는데 아마..." (추측 금지)
- 질문에 대한 방어적 반응 ("그건 제 의도가 아니었어요")
- 답변 중 불필요한 웃음

### 시간 관리

- 답변 1개 평균 **30초~1분 이내**
- 너무 길게 답하면 청중이 지루해짐
- 모호한 부분은 "보고서에 자세히 기술했습니다"로 끝맺음 가능

## 🎴 리허설 활용법

### 1단계 — 카드 순서대로 낭독 (1주 전)

각 카드의 "필수 말해야 할 것"을 보며 **소리 내어 읽기**. 한 번에 전체 12장 하기 어려우면 6장씩 나누어 해도 된다.

### 2단계 — 카드 없이 발표 (3일 전)

카드를 덮고 **"이것만 말해도 OK"**를 외워서 슬라이드 순서대로 말해보기. 모든 카드가 나와야 한다.

### 3단계 — 풀 리허설 (1일 전)

슬라이드를 띄우고 원본 `script.md` 또는 `script-v2.md`를 옆에 두고 **시간 측정**. 슬라이드 1~11은 약 10분 20초 목표.

### 4단계 — 당일 아침

카드 12장 **썸네일 훑기** (한 카드당 10초 정도, 총 2분).
