# 용어 사전 — 서민지가 "자기 말로" 설명하기

> 이 문서의 목적: 발표·Q&A에서 등장할 모든 전문 용어를 **서민지가 입으로 설명할 수 있는 수준**으로 정리한다.
> 사용 방법: 각 용어는 (1) 정식 정의 (2) 쉽게 말하면 (3) "이렇게 말하면 안전해" 표현 (4) 예상 Q&A 순으로 구성되어 있다. 3번 표현을 외워두면 된다.

---

## A. 스케줄링 알고리즘

### FCFS (First-Come First-Served)

- **정식**: 도착한 순서대로 요청을 처리하는 가장 단순한 큐잉 알고리즘.
- **쉽게 말하면**: "줄 선 순서대로 처리한다."
- **이렇게 말하면 안전해**: "FCFS는 First-Come First-Served의 약자로, 말 그대로 먼저 도착한 요청부터 차례대로 처리하는 가장 단순한 방식입니다. 다른 알고리즘의 기준선 역할을 합니다."
- **예상 Q&A**: "FCFS의 단점은?" → "우선순위나 등급을 반영할 수 없어서, 긴 요청이 맨 앞에 오면 뒤의 짧은 요청이 계속 기다리게 됩니다."

### Priority Scheduling

- **정식**: 요청마다 부여된 우선순위 값에 따라 처리 순서를 결정하는 알고리즘.
- **쉽게 말하면**: "급한 요청부터 먼저 처리한다."
- **이렇게 말하면 안전해**: "Priority 스케줄러는 요청마다 붙어 있는 URGENT, HIGH, NORMAL, LOW 네 단계의 우선순위를 보고, 높은 순위부터 먼저 처리합니다."
- **예상 Q&A**: "기아 문제는 어떻게 해결했나요?" → "5초 이상 기다린 요청은 자동으로 우선순위를 한 단계 올리는 **에이징**을 적용했습니다."

### Aging (에이징)

- **정식**: 대기 시간이 길어진 요청의 우선순위를 점진적으로 올려 기아를 방지하는 기법.
- **쉽게 말하면**: "오래 기다린 사람을 뒤늦게라도 챙겨주는 장치."
- **이렇게 말하면 안전해**: "에이징은 오래 기다린 요청의 우선순위를 한 단계씩 올려주는 방식입니다. 저희 구현에서는 5초마다 한 단계씩 올리고, 최대 URGENT까지 승격됩니다."

### MLFQ (Multi-Level Feedback Queue)

- **정식**: 여러 개의 큐를 두고, 실행 시간에 따라 요청을 상하위 큐로 이동시키는 알고리즘.
- **쉽게 말하면**: "빨리 끝날 일은 위 줄, 오래 걸리는 일은 아래 줄."
- **이렇게 말하면 안전해**: "MLFQ는 Multi-Level Feedback Queue의 약자입니다. 말 그대로 여러 단계의 큐를 두고, 짧은 요청은 위쪽 큐에 남겨서 먼저 처리하고, 오래 걸리는 요청은 아래 큐로 내려보내는 방식입니다. 저희는 4단계 큐를 구현했습니다."

### 비선점형 vs 선점형 (Non-preemptive vs Preemptive)

- **정식 (비선점형)**: 한 번 실행을 시작한 요청은 완료될 때까지 중단되지 않는 방식.
- **정식 (선점형)**: 실행 중인 요청을 중단하고 다른 요청에 CPU를 넘길 수 있는 방식.
- **쉽게 말하면**: "비선점형은 한 번 잡으면 끝까지, 선점형은 중간에 빼앗을 수 있음."
- **이렇게 말하면 안전해**: "비선점형은 한 번 처리가 시작되면 그 요청이 끝날 때까지 중단되지 않는 방식이고, 선점형은 중간에 다른 요청에 처리 시간을 넘길 수 있는 방식입니다. 저희 프로젝트의 기본 MLFQ는 비선점형이고, 추가로 선점형 MLFQ를 구현해 시뮬레이션에서만 비교했습니다."
- **예상 Q&A**: "왜 선점형을 실서버에서 안 했나요?" → "LLM은 추론 중간에 멈추는 게 거의 불가능해서, Ollama를 포함한 대부분의 LLM 도구가 선점을 지원하지 않습니다."

### 강등 (Demotion)

- **정식**: MLFQ에서 할당 시간을 초과한 요청을 하위 큐로 내려보내는 동작.
- **쉽게 말하면**: "오래 걸리는 요청은 아래 큐로 떨어뜨린다."
- **이렇게 말하면 안전해**: "강등은 MLFQ에서 요청이 그 큐의 할당 시간을 다 써버리면 한 단계 아래 큐로 내려보내는 동작입니다. 짧은 요청을 위 큐에 남겨두기 위한 장치입니다."

### 부스팅 (Boosting) / Rule 5

- **정식**: 하위 큐에 오래 머문 요청을 주기적으로 최상위 큐로 되돌려 기아를 방지하는 기법.
- **쉽게 말하면**: "아래 큐에 너무 오래 있던 요청은 다시 위로 올려준다."
- **이렇게 말하면 안전해**: "부스팅은 MLFQ에서 오래 기다린 요청을 주기적으로 최상위 큐로 되돌리는 기능입니다. 저희는 5초 주기로 부스팅을 적용해서, 긴 요청도 계속 기다리기만 하지 않고 언젠가는 처리되도록 했습니다."

### 기아 상태 (Starvation)

- **정식**: 특정 요청이 다른 요청에 의해 영원히 처리되지 못하고 대기만 하는 상태.
- **쉽게 말하면**: "영원히 순서가 오지 않는 요청."
- **이렇게 말하면 안전해**: "기아는 우선순위가 낮거나 처리 시간이 긴 요청이 계속 뒤로 밀려서 영영 처리되지 못하는 상태입니다. 저희는 Priority에서는 에이징, MLFQ에서는 부스팅으로 기아를 방지했습니다."

### WFQ (Weighted Fair Queuing)

- **정식**: 각 요청에 가상 완료 시각(VFT)을 계산해 가장 작은 값부터 처리하는 공정 큐 알고리즘.
- **쉽게 말하면**: "가중치가 큰 사용자의 요청이 더 자주 처리되도록 한다."
- **이렇게 말하면 안전해**: "WFQ는 Weighted Fair Queuing의 약자로, 원래 네트워크 QoS에서 쓰던 알고리즘입니다. 각 요청에 가상 완료 시각, 즉 VFT를 계산해서 가장 작은 값부터 처리합니다. 가중치가 큰 사용자는 VFT 증가 속도가 느려서 결과적으로 더 자주 선택됩니다."
- **예상 Q&A**: "VFT 구체적으로 어떻게 계산해요?" → "요청 비용을 1로 고정하고, 이전 요청의 VFT에 1을 가중치로 나눈 값을 더하는 방식입니다. Enterprise는 가중치 100이니 VFT 증가분이 0.01, Free는 가중치 1이니 증가분이 1이 됩니다."

### VFT (Virtual Finish Time, 가상 완료 시각)

- **정식**: WFQ에서 각 요청이 "가상 시계"에서 언제 끝날지를 계산한 값.
- **쉽게 말하면**: "이 요청의 완료 예상 순번."
- **이렇게 말하면 안전해**: "VFT는 가상 완료 시각이라는 뜻입니다. 실제 시간이 아니라 각 사용자의 가중치를 반영한 '가상의 시간'을 계산한 값이고, 이 값이 작은 요청부터 먼저 처리됩니다."
- **예상 Q&A**: "왜 비용을 1로 고정했어요?" → "LLM은 응답 토큰 수를 미리 알 수 없어서 실제 처리 비용을 예측하기 어렵기 때문입니다. 보고서 4.3절에 설명돼 있습니다."

### QoS (Quality of Service, 서비스 품질)

- **정식**: 네트워크에서 서로 다른 트래픽에 차별적 서비스 품질을 제공하기 위한 기법의 총칭.
- **쉽게 말하면**: "중요한 트래픽을 더 잘 처리해주는 기술."
- **이렇게 말하면 안전해**: "QoS는 네트워크에서 트래픽마다 다른 수준의 서비스 품질을 제공하는 기법입니다. WFQ도 QoS를 구현하기 위한 대표적 알고리즘 중 하나입니다."

## B. 공정성·통계 용어

### JFI (Jain's Fairness Index)

- **정식**: 자원 분배의 공정성을 0부터 1 사이 점수로 측정하는 지표.
- **쉽게 말하면**: "1에 가까우면 공평, 1/n에 가까우면 독점."
- **이렇게 말하면 안전해**: "JFI는 Jain's Fairness Index로, 자원이 얼마나 고르게 분배됐는지를 0부터 1 사이 점수로 나타냅니다. 1이면 완전 공평, 1을 사용자 수로 나눈 값이면 한 사람 독점을 뜻합니다. 저희 실험에서 FCFS·Priority·비선점형 MLFQ는 모두 1.000이었고, WFQ는 0.316이었는데 이건 가중치 기반 의도적 차등의 결과입니다."
- **예상 Q&A**: "WFQ의 JFI가 0.316이면 불공평한 거 아닌가요?" → "'불공평'과 '의도적 차등'은 다릅니다. JFI는 숫자일 뿐이고, WFQ는 가중치대로 차등 서비스를 제공하도록 설계됐습니다. 즉 이 0.316이라는 숫자 자체가 WFQ가 가중치를 반영했다는 증거입니다."

### 95% 신뢰구간 (95% Confidence Interval)

- **정식**: 같은 실험을 반복했을 때 참값이 그 범위 안에 있을 확률이 95%인 구간.
- **쉽게 말하면**: "이 정도 범위 안에 참값이 있다고 95% 확신할 수 있다."
- **이렇게 말하면 안전해**: "95% 신뢰구간은 실험을 반복했을 때 참값이 그 범위 안에 있을 확률이 95%라는 뜻입니다. 저희 MLFQ 선점형 실험에서 Short 개선율의 95% 신뢰구간은 71.9%부터 74.5% 사이로 나왔습니다. 좁은 범위라는 건 개선 효과가 안정적이라는 뜻입니다."
- **예상 Q&A**: "신뢰구간의 95%는 구체적으로 어떻게 계산했나요?" → "쌍체 t-검정 기반의 표준오차를 이용해 계산했고, 재현 스크립트는 `08-final-submission/experiments/compute-stats.js`에 있습니다."

### Cohen's d

- **정식**: 두 그룹 평균 차이를 표준편차로 나눈 효과 크기 지표.
- **쉽게 말하면**: "차이가 얼마나 큰지를 크기 순으로 나타내는 수치. 0.8 이상이면 큰 효과."
- **이렇게 말하면 안전해**: "Cohen's d는 두 그룹 평균 차이가 얼마나 큰지를 표준편차로 나눠 나타낸 값입니다. 저희 실험에서는 20.39라는 아주 큰 값이 나왔고, 이건 MLFQ 선점형이 FCFS 대비 확실히 효과적이라는 뜻입니다."

### 쌍체 t-검정 (Paired t-test)

- **정식**: 같은 조건에 두 방법을 적용했을 때 그 차이가 우연인지 유의미한지 확인하는 통계 검정.
- **쉽게 말하면**: "이 차이가 우연이 아니라는 걸 통계로 확인하는 방법."
- **이렇게 말하면 안전해**: "쌍체 t-검정은 같은 실험 조건에서 두 방식을 비교했을 때 차이가 우연인지 아닌지 판단하는 통계 방법입니다. 저희 결과는 t = 45.59, p 값 0.001 미만으로, 우연에 기댄 결과가 아니라는 게 확인됐습니다."

### 시드 (Seed)

- **정식**: 난수 생성기의 초기값. 같은 시드로 실행하면 같은 난수 순서가 반복된다.
- **쉽게 말하면**: "난수의 씨앗. 같은 씨앗이면 같은 꽃이 핀다."
- **이렇게 말하면 안전해**: "시드는 난수 생성의 초기값입니다. 같은 시드를 쓰면 같은 요청 패턴을 다시 만들 수 있어서 실험 재현이 가능합니다. 저희는 12345, 23456, 34567, 45678, 56789 이렇게 다섯 개 시드로 반복했습니다."

### 버스트 (Burst) 도착

- **정식**: 요청이 일시에 몰려 들어오는 트래픽 패턴.
- **쉽게 말하면**: "한꺼번에 우르르 몰려옴."
- **이렇게 말하면 안전해**: "버스트 도착은 요청이 한꺼번에 몰려 들어오는 패턴입니다. 저희 MLFQ 선점형 실험은 이런 버스트 상황을 가정해서 짧은 요청과 긴 요청이 섞여 들어오도록 설정했습니다."

## C. 성능·처리 지표

### 대기시간 (Wait Time)

- **정식**: 요청이 큐에 들어간 시점부터 처리가 시작된 시점까지의 시간.
- **쉽게 말하면**: "큐에서 기다린 시간."
- **이렇게 말하면 안전해**: "대기시간은 요청이 큐에 들어간 순간부터 처리가 시작될 때까지의 시간입니다. 저희 주 실험의 성능 지표입니다."

### 응답시간 (Response Time)

- **정식**: 요청 도착 시점부터 완료 시점까지의 전체 소요 시간 (대기 + 처리).
- **쉽게 말하면**: "도착부터 끝까지 총 걸린 시간."
- **이렇게 말하면 안전해**: "응답시간은 요청이 도착한 순간부터 완료될 때까지의 전체 시간입니다. 대기시간과 처리 시간을 합친 값이고, MLFQ 선점형처럼 처리 중 중단이 있는 경우에는 이 응답시간을 쓰는 게 더 정확합니다."

### 처리 시간 (Processing Time)

- **정식**: 요청이 실제로 실행되어 완료되기까지의 시간 (LLM 추론 시간).
- **쉽게 말하면**: "LLM이 실제로 답 만드는 데 걸린 시간."
- **이렇게 말하면 안전해**: "처리 시간은 요청이 실제로 실행되는 데 걸리는 시간입니다. Ollama 실서버에서는 평균 208에서 209밀리초였습니다. 이 시간은 LLM 자체의 속도라서 스케줄러가 바꿀 수 없다는 게 저희 실서버 실험의 핵심 발견입니다."

### 처리량 (Throughput)

- **정식**: 단위 시간당 처리되는 요청 수 (req/s 등).
- **쉽게 말하면**: "1초에 몇 건 처리했는가."
- **이렇게 말하면 안전해**: "처리량은 초당 처리된 요청 수입니다. 저희 기본 실험에서 네 알고리즘이 모두 18.56부터 18.72 req/s 사이로 비슷했고, 이건 비선점형 환경에서는 처리량에 큰 차이가 없음을 보여줍니다."

## D. 시스템·프로토콜 용어

### Rate Limiter

- **정식**: 사용자별 요청 속도를 제한해 과도한 요청을 차단하는 전처리 장치.
- **쉽게 말하면**: "너무 많이 보내면 잠깐 거절하는 문지기."
- **이렇게 말하면 안전해**: "Rate Limiter는 스케줄러로 들어가기 전 단계에서 등급별 분당 요청 한도를 걸어두는 문지기 역할입니다. 저희는 Enterprise 100건, Premium 50건, Standard 10건, Free 5건으로 설정했고, 한도를 넘으면 HTTP 429를 돌려줍니다."

### 토큰 버킷 (Token Bucket)

- **정식**: 일정 시간마다 발급되는 토큰을 요청마다 하나씩 소모하는 방식의 Rate Limiting 알고리즘.
- **쉽게 말하면**: "토큰이 든 바구니에서 요청할 때마다 하나씩 뽑는다."
- **이렇게 말하면 안전해**: "토큰 버킷은 시간에 따라 토큰이 채워지고, 요청할 때마다 토큰을 하나씩 소비하는 방식입니다. 저희 구현은 1분 단위 슬라이딩 윈도우로 같은 효과를 내도록 단순화해 구현했습니다."
- **예상 Q&A**: "보고서에는 토큰 버킷이라고 써 있는데 실제 코드는 슬라이딩 윈도우던데요?" → "네, 저희는 개념적으로는 토큰 버킷, 구체 구현은 1분 슬라이딩 윈도우 카운터로 단순화했습니다. 분당 N건이라는 결과는 동일합니다."

### HTTP 429 (Too Many Requests)

- **정식**: 서버가 설정한 요청 한도를 클라이언트가 초과했을 때 반환하는 표준 HTTP 상태 코드.
- **쉽게 말하면**: "너무 많이 보냈어요, 잠깐 쉬세요."
- **이렇게 말하면 안전해**: "HTTP 429는 '너무 많은 요청'이라는 뜻의 표준 응답 코드입니다. Rate Limiter가 한도를 넘은 요청을 거절할 때 쓰입니다."

### Ollama

- **정식**: 로컬 환경에서 대규모 언어 모델(LLM)을 실행할 수 있게 해주는 오픈소스 도구.
- **쉽게 말하면**: "내 컴퓨터에서 ChatGPT 비슷한 걸 돌리는 프로그램."
- **이렇게 말하면 안전해**: "Ollama는 LLM을 로컬 컴퓨터에서 실행할 수 있게 해주는 오픈소스 도구입니다. 저희는 실서버 실험에서 Ollama로 Google의 Gemma 4 계열 gemma4:e4b 모델을 돌렸습니다."

### gemma4:e4b

- **정식**: Google Gemma 4 계열의 효율화(efficient) 4B 파라미터 모델.
- **쉽게 말하면**: "구글이 만든 중간 크기의 LLM. 파라미터 40억 개짜리."
- **이렇게 말하면 안전해**: "gemma4:e4b는 2026년 4월 공개된 Google Gemma 4 계열의 모델 중 하나로, 약 40억 개 파라미터를 사용하는 중형 모델입니다. 이름의 'e4b'는 효율화된 4B 파라미터라는 뜻입니다."

### 연속 배치 (Continuous Batching)

- **정식**: 여러 요청을 실행 중에도 새로운 요청을 합류시켜 함께 처리하는 LLM 서빙 최적화 기법.
- **쉽게 말하면**: "이미 달리고 있는 버스에 중간 정류장에서 손님을 계속 태우는 방식."
- **이렇게 말하면 안전해**: "연속 배치는 실제 LLM 서빙에서 여러 요청을 동시에 묶어서 처리하는 방식입니다. 실행 중인 배치에 새 요청을 계속 끼워넣을 수 있어서 처리량을 높입니다. vLLM이나 TGI 같은 상용 도구가 이 방식을 씁니다. 저희 프로젝트는 단순 순차 처리이므로 이 효과는 검증하지 못했고, 향후 연구 과제로 남겼습니다."

### vLLM, TGI

- **정식 (vLLM)**: UC Berkeley에서 개발한 고성능 LLM 서빙 엔진.
- **정식 (TGI)**: Hugging Face의 Text Generation Inference, 상용 LLM 서빙 도구.
- **이렇게 말하면 안전해**: "vLLM과 TGI는 실제 상용 LLM 서빙에서 널리 쓰이는 도구들입니다. 연속 배치, PagedAttention 같은 고급 기법을 사용해 처리량을 극대화합니다. 저희 프로젝트에서는 직접 다루지 않았지만, 보고서 6장에서 이런 도구와의 차이를 언급하고 있습니다."

## E. 모델·파라미터 용어

### 파라미터 (Parameter)

- **정식**: 신경망이 학습 과정에서 조정한 가중치 값들의 총 개수.
- **쉽게 말하면**: "모델의 뇌세포 개수 같은 것. 많을수록 크고 똑똑한 편."
- **이렇게 말하면 안전해**: "파라미터는 모델이 학습하면서 조정한 가중치 값의 개수입니다. gemma4:e4b는 약 40억 개로, 중형 모델에 해당합니다."

### 토큰 (Token)

- **정식**: LLM이 입출력 단위로 사용하는 텍스트 조각 (단어 또는 부분 단어).
- **쉽게 말하면**: "LLM이 읽고 쓰는 단위. 한 단어가 보통 1~2토큰."
- **이렇게 말하면 안전해**: "토큰은 LLM이 다루는 기본 단위로, 한 단어가 보통 1~2토큰 정도입니다. 저희 실서버 실험에서는 출력을 최대 20토큰으로 제한했습니다."

## F. "나도 처음 들어" 안전망 표현

모르는 질문이 나왔을 때 당황하지 않고 쓸 수 있는 안전한 답변 패턴.

1. **"그 부분은 본 연구에서 다루지 못한 한계입니다."** — 기본 패턴
2. **"저희 연구 범위 밖이라 정확히 답변드리기 어렵습니다. 보고서 6장 한계 부분에서 관련 내용을 언급하고 있습니다."** — 깊은 질문
3. **"흥미로운 지적입니다. 향후 연구 과제로 이 부분을 검토해보겠습니다."** — 창의적 비판
4. **"구체적으로는 보고서 N절에 기술되어 있는데, 지금 바로 정확한 수치가 기억나지는 않습니다."** — 기억이 안 날 때
5. **"제가 정확히 이해했는지 다시 확인하고 싶습니다. 질문의 요지는 ~인가요?"** — 질문이 복잡할 때
