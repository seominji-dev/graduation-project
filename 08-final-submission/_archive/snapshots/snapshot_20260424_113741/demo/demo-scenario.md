- **LLM 모델 다운로드 확인**: `ollama list` — 기본 모델은 `gemma4:e4b`, 없으면 `ollama pull gemma4:e4b` (다른 모델 사용 시 서버 실행 시점에 `OLLAMA_MODEL=<모델명>` 환경변수 설정)

### 시스템 동작 방식 (숙지 필수)

본 시스템은 다음 두 가지 원칙으로 동작한다:

1. **Ollama 자동 연결**: Ollama 서버(포트 11434)가 실행 중이면 서버 기동 시 자동으로 연동된다. 별도 환경 변수는 없다.
2. **수동 처리 트리거**: 요청을 `POST /api/requests`로 보내면 큐에 적재될 뿐, 자동 처리되지 않는다. 대시보드의 **"다음 처리"** 버튼을 누르거나 `POST /api/scheduler/process`를 호출해야 한 건씩 꺼내 LLM에 전달된다.

**데모에서 처리 트리거는 주로 대시보드 버튼으로 수행**한다. 청중이 버튼 클릭과 대시보드 변화를 동시에 볼 수 있기 때문이다. curl로 처리 트리거가 필요한 경우는 Q&A 시 추가 시연용으로만 사용한다:

```bash
# 한 건만 처리 (필요 시)
curl -X POST http://localhost:3000/api/scheduler/process
```

## 시나리오 1: FCFS vs WFQ 처리 순서 비교 + Ollama 증거샷 (약 2분 30초)

### 목적
FCFS는 도착 순서대로, WFQ는 Enterprise 등급을 먼저 처리함을 시각적으로 보여준다.

### 1-1. FCFS 서버 시작

터미널 1에서:

```bash
cd 졸업프로젝트/02-implementation
SCHEDULER_TYPE=FCFS node src-simple/server.js
```

예상 출력 (ASCII 박스 형태로 출력됨):
```
╔════════════════════════════════════════════════╗
║         LLM Scheduler 서버 시작                 ║
╠════════════════════════════════════════════════╣
║  포트: 3000                                    ║
║  스케줄러: FCFS                                ║
║  API 문서: http://localhost:3000               ║
╚════════════════════════════════════════════════╝
```

이 박스가 출력되면 서버 기동에 성공한 것이다. Ollama 연결 상태는 별도 health check로 확인한다 (터미널 2에서 `curl http://localhost:3000/api/health` 실행 시 `"llm":"connected"` 필드가 나오면 정상 연결).

### 1-2. 브라우저 대시보드 열기

브라우저에서 접속: `http://localhost:3000`

대시보드에서 확인:
- 현재 스케줄러 타입 (FCFS)
- 대기 중인 요청 목록
- 처리된 요청 통계

### 1-3. 4개 등급 요청 전송

터미널 2에서 순서대로 전송 (Free → Standard → Premium → Enterprise 순):

```bash
# Free 등급 요청 (먼저 도착)
curl -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Free user request","tenantId":"user-free","tier":"free","priority":"NORMAL"}'

# Standard 등급 요청
curl -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Standard user request","tenantId":"user-standard","tier":"standard","priority":"NORMAL"}'

# Premium 등급 요청
curl -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Premium user request","tenantId":"user-premium","tier":"premium","priority":"HIGH"}'

# Enterprise 등급 요청 (나중에 도착)
curl -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Enterprise user request","tenantId":"user-enterprise","tier":"enterprise","priority":"HIGH"}'
```

이 시점에서는 4건이 모두 **대기 큐에 쌓여 있는 상태**이고 아직 처리되지 않았다. 대시보드에서 `pending` 상태 4건을 청중에게 보여준다.

### 1-4. FCFS 처리 순서 확인 (대시보드 "다음 처리" 버튼 4회 클릭)

브라우저 대시보드로 전환하여 **"다음 처리"** 버튼을 4번 순차적으로 누른다.
누를 때마다 큐에서 한 건씩 꺼내져 `processing` → `completed` 상태로 바뀐다.

예상 처리 순서 (FCFS):
1. Free (먼저 도착)
2. Standard
3. Premium
4. Enterprise (나중에 도착)

설명 멘트:
> "FCFS에서는 도착 순서대로 처리됩니다. Free가 먼저 왔으니 Free가 먼저 처리됩니다. Enterprise가 가장 늦게 도착했으므로 가장 나중에 처리됩니다."

### 1-5. WFQ로 서버 전환

터미널 1에서 Ctrl+C로 서버 중단 후:

```bash
SCHEDULER_TYPE=WFQ node src-simple/server.js
```

### 1-6. 동일 요청 재전송 및 처리

위 1-3 명령어 4건을 동일한 순서(Free → Standard → Premium → Enterprise)로 다시 전송한 뒤, 대시보드에서 **"다음 처리"** 버튼을 4번 누른다.

예상 처리 순서 (WFQ, 가중치 Enterprise:Premium:Standard:Free = 100:50:10:1):
1. **Enterprise** (가장 먼저 처리 — 가중치 100)
2. Premium (가중치 50)
3. Standard (가중치 10)
4. Free (마지막 처리 — 가중치 1)

설명 멘트:
> "WFQ에서는 구독 등급에 비례한 처리 횟수를 보장합니다. Enterprise가 Free보다 100배 높은 가중치를 가지므로, 도착 순서가 뒤여도 먼저 처리됩니다."

브라우저 대시보드에서 처리 순서 확인: Enterprise → Premium → Standard → Free 순.

### 1-7. Ollama 실호출 증거샷 (약 30초)

목적: 보고서 5.4절 실서버 실험이 실제로 동작 가능함을 증명한다.

**별도 재시작 불필요**: Ollama 서버(포트 11434)가 기동 중이라면 현재 실행 중인 WFQ 서버가 이미 자동으로 Ollama와 연결되어 있다. 즉, 이전 단계에서 "다음 처리"를 누를 때마다 실제로는 Ollama가 응답을 생성하고 있었다.

터미널 2에서 Enterprise 등급으로 짧고 명확한 프롬프트 1건 전송:

```bash
curl -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"한 문장으로 인사해주세요","tenantId":"user-enterprise","tier":"enterprise","priority":"HIGH"}'
```

응답 예시 (요청 ID만 즉시 반환, 처리 전):
```json
{
  "message": "요청이 제출되었습니다",
  "requestId": "<UUID 형식>",
  "status": "pending"
}
```

HTTP 상태 코드는 **201 Created**로 반환된다. `requestId`는 서버가 생성한 UUID(예: `12379b12-76e2-4fd2-b50c-144cdaf09c76`) 형식이며, 이후 대시보드에서 해당 ID의 상태 변화를 추적할 수 있다.

대시보드에서 **"다음 처리"** 버튼을 누른 뒤, 해당 요청이 `processing` → `completed` 상태로 바뀌며 실제 LLM이 생성한 응답 텍스트(예: "안녕하세요! 반갑습니다.")가 표시되는 것을 청중에게 확인시킨다.

설명 멘트:
> "지금까지의 비교는 시뮬레이션 기반이지만, 실제 Ollama를 통한 LLM 호출도 동일한 시스템에서 작동합니다. 방금 보신 응답 텍스트는 실제 Ollama 로컬 LLM이 생성한 결과입니다. 보고서 5.4절의 실서버 실험은 이 방식으로 구독 등급별 5건씩 총 20건을 수행한 결과입니다."

## 시나리오 2: Rate Limiter 동작 시연 (1분)

### 목적
구독 등급별 요청 한도가 적용되어, Free 등급은 일정 수 이상 요청 시 거부됨을 보여준다.

### 2-1. Rate Limiter 활성화 서버 시작

터미널 1에서 (Ctrl+C로 기존 서버 중단 후):

```bash
RATE_LIMIT_ENABLED=true SCHEDULER_TYPE=WFQ node src-simple/server.js
```

### 2-2. Free 등급 요청 빠르게 전송

터미널 2에서 루프로 6번 전송:

```bash
for i in 1 2 3 4 5 6; do
  curl -s -o /dev/null -w "요청 $i: %{http_code}\n" \
    -X POST http://localhost:3000/api/requests \
    -H "Content-Type: application/json" \
    -d "{\"prompt\":\"Request $i\",\"tenantId\":\"user-free\",\"tier\":\"free\",\"priority\":\"NORMAL\"}"
done
```

예상 출력:
```
요청 1: 201
요청 2: 201
요청 3: 201
요청 4: 201
요청 5: 201
요청 6: 429
```

설명 멘트:
> "Free 등급은 분당 5건으로 제한됩니다. 한도를 초과하면 HTTP 429 응답을 받습니다. Enterprise 등급은 분당 100건으로 훨씬 높은 한도를 가집니다."

### 2-3. Enterprise는 정상 처리 확인

```bash
curl -s -o /dev/null -w "Enterprise 요청: %{http_code}\n" \
  -X POST http://localhost:3000/api/requests \
  -H "Content-Type: application/json" \
  -d '{"prompt":"Enterprise request","tenantId":"user-enterprise","tier":"enterprise","priority":"HIGH"}'
```

예상 출력: `Enterprise 요청: 201`

설명 멘트:
> "같은 상황에서 Enterprise는 분당 100건 한도 안이므로 정상적으로 접수됩니다. 201은 Created 응답으로, Rate Limiter를 통과해 요청 객체가 생성되고 스케줄러 큐에 적재되었다는 의미입니다."

**처리 트리거**: 이 시나리오의 핵심은 Rate Limiter가 **큐 진입 자체를 거부**하는 것(429)이므로 실제 처리까지 보여줄 필요는 없다. 대시보드에서 Free 5건(한도까지)이 `pending`으로 쌓여 있고 6번째가 거부된 상태만 확인하면 충분하다. Free 한도가 5건이므로 루프 6회 중 5회는 201 Created, 1회(6번째)는 429 Too Many Requests가 반환된다.

## 시나리오 3: Ollama 실서버 축소 재현 (예비, 2~3분)

**⚠️ 이 시나리오는 기본 발표에 포함되지 않는다.**
교수님이 "실제 LLM 환경에서도 보여달라"고 요청할 경우에만 실행한다.

### 목적

보고서 5.4절 실서버 실험을 발표장에서 축소 재현하여, 스케줄러가 바뀌어도 LLM 추론 속도는 거의 동일(약 116~119ms)하지만 **대기시간은 스케줄러에 따라 달라진다**는 핵심 결과를 실시간으로 보여준다.

### 3-1. WFQ 서버 시작 (Ollama 자동 연결)

Ollama 서버가 이미 기동 중이라면 별도 환경 변수 없이 자동 연결된다. 터미널 1에서 (기존 서버 Ctrl+C 후):

```bash
SCHEDULER_TYPE=WFQ node src-simple/server.js
```

서버 기동 직후 터미널 2에서 health check로 Ollama 연결 상태를 확인한다:

```bash
curl http://localhost:3000/api/health
```

응답의 `"llm":"connected"` 필드가 출력되면 Ollama와 정상 연결된 상태다. (Ollama 서버가 꺼져 있으면 `"llm":"mock"`으로 Fallback되어 시뮬레이션 응답을 반환한다.)

### 3-2. 등급별 요청 3건씩 전송 (총 12건, 보고서 5.4절의 60% 축소)

터미널 2에서:

```bash
for TIER in free standard premium enterprise; do
  for i in 1 2 3; do
    curl -s -o /dev/null -w "$TIER-$i: %{http_code}\n" \
      -X POST http://localhost:3000/api/requests \
      -H "Content-Type: application/json" \
      -d "{\"prompt\":\"Hi\",\"tenantId\":\"user-$TIER-$i\",\"tier\":\"$TIER\",\"priority\":\"NORMAL\"}"
  done
done
echo "총 12건 전송 완료"
```

### 3-3. 순차 처리 실행 및 결과 관찰 (약 60~90초 소요)

12건을 순차 처리하기 위해 대시보드 **"다음 처리"** 버튼을 12번 반복해서 누른다(또는 아래 curl 반복문으로 일괄 트리거).

```bash
# 선택: 12건을 자동으로 순차 처리
for i in $(seq 1 12); do
  curl -s -o /dev/null -w "처리 $i 완료\n" -X POST http://localhost:3000/api/scheduler/process
  sleep 0.5
done
```

브라우저 대시보드에서 확인:
- 각 요청의 **처리 시간**은 스케줄러와 무관하게 약 116~119ms로 유사
- **Enterprise 등급의 평균 대기시간**이 Free 등급보다 현저히 짧음
- 가중치 100:50:10:1 비율에 따라 Enterprise가 먼저 처리됨

### 3-4. FCFS와 비교 (시간 여유 시)

터미널 1에서 Ctrl+C 후:

```bash
SCHEDULER_TYPE=FCFS node src-simple/server.js
```

3-2 명령을 다시 실행한 뒤 대시보드 **"다음 처리"** 버튼을 반복해서 누르면 "도착 순서대로 처리"되는 것을 볼 수 있다.

설명 멘트:
> "보고서에서 제시한 결과와 동일한 경향이 발표장에서도 재현됩니다. 처리 시간은 LLM 자체가 결정하지만, 대기시간은 스케줄러가 어떤 요청을 먼저 고르느냐에 따라 달라집니다."

### 3-5. 리스크 대응

- Ollama 응답이 30초 이상 지연되면: 시연 중단 후 백업 녹화본으로 전환
- 특정 요청 타임아웃 발생 시: 해당 요청은 건너뛰고 나머지 진행
- 짧은 프롬프트(`"Hi"`)로 부하를 최소화했으므로 GPU/CPU 과열은 일반적으로 문제되지 않음

## Q&A 예상 질문 및 답변 포인트

### 기술적 질문

**Q1. 왜 실제 LLM 호출이 아닌 시뮬레이션이 주 실험인가요?**
> 알고리즘 자체의 스케줄링 특성을 비교하려면 네트워크 지연이나 LLM 상태에 따른 변동을 배제해야 합니다. 시뮬레이션으로 알고리즘의 순수한 차이를 확인한 뒤, 실서버 실험(5.4절)으로 실제 환경에서도 경향이 유지됨을 검증했습니다.

**Q2. JFI(Jain's Fairness Index)는 무엇이고 왜 사용했나요?**
> 자원이 얼마나 고르게 분배되었는지를 0에서 1 사이 점수로 나타내는 지표입니다. 1이면 완전히 공평, 1/n이면 한 사용자가 독점하는 상태를 뜻합니다. 구독 등급별 차등 서비스가 의도한 비율대로 이뤄졌는지 수치로 확인하기 위해 사용했습니다.

**Q3. MLFQ 선점형이 왜 실제 LLM 환경에서 안 되나요?**
> LLM 추론은 한 번 시작되면 중간에 멈출 수 없습니다. Ollama를 포함한 대부분의 LLM 서빙 도구가 이러한 특성을 가지므로, 선점형 MLFQ는 시뮬레이션에서만 효과를 검증했습니다.

**Q4. WFQ의 가중치는 어떻게 결정했나요?**
> 구독 등급별 상대적 서비스 비율을 Enterprise:Premium:Standard:Free = 100:50:10:1로 설정했습니다. 일반적인 SaaS 서비스 등급 체계를 참고하여, 최상위 등급이 최하위 등급보다 100배 수준의 서비스 비중을 받도록 배정했습니다.

**Q5. Rate Limiter와 스케줄러는 어떻게 구분되나요?**
> Rate Limiter는 요청의 시스템 진입 여부를 결정하는 전처리 단계이고, 스케줄러는 이미 진입한 요청 중 어떤 것을 먼저 처리할지 결정하는 단계입니다. 두 기능은 독립적으로 동작합니다.

**Q6. 왜 Node.js와 Express를 선택했나요?**
> 학부 수준에서 구현·이해가 용이하며, 비동기 처리 모델이 LLM 호출과 같은 I/O 중심 작업에 적합하기 때문입니다.

### 연구 범위 질문

**Q7. 실서버 실험 규모가 너무 작지 않나요?**
> 보고서 6장 제한사항에서 인정한 부분입니다. 20건 규모는 시뮬레이션 결과의 방향성이 실제 환경에서도 유지되는지 확인하는 수준이며, 실무 적용을 위해서는 수백~수천 건 규모의 부하 시험이 필요합니다.

**Q8. 연속 배치(Continuous Batching)는 고려했나요?**
> 본 프로젝트에서 다루지 못한 한계입니다. 실제 LLM 서빙 시스템(vLLM, TGI 등)은 연속 배치로 여러 요청을 동시에 처리하는데, 이 경우 스케줄링 효과가 달라질 수 있습니다. 향후 연구 과제로 남겨두었습니다.

**Q9. VFT 계산에서 처리 비용을 1로 고정한 이유는?**
> LLM 요청은 응답 토큰 수를 미리 알 수 없어 정확한 처리 시간을 예측하기 어렵기 때문입니다. 보고서 4.3절에 상세히 기술했으며, 향후 토큰 수 예측 모델을 결합하는 개선 방향을 고려하고 있습니다.

### 의의 질문

**Q10. 이 연구가 왜 의미가 있나요?**
> ChatGPT 같은 LLM 서비스가 보편화되면서, 여러 사용자가 동시에 요청하는 환경에서의 공정성 문제가 중요해지고 있습니다. 운영체제에서 오래 연구된 스케줄링 이론을 LLM 환경에 적용해본 것 자체가 교육적 의의가 있습니다.

**Q11. 앞으로 더 발전시킨다면?**
> 세 가지 방향을 생각하고 있습니다. 첫째, 연속 배치 환경에서의 스케줄링. 둘째, 더 큰 규모의 부하 시험. 셋째, 응답 길이를 고려한 비용 예측(VFT 개선)입니다.
