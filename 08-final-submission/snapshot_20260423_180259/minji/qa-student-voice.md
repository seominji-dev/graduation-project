> 사용 방법: 각 질문마다 **[원본]** (정식 답변)과 **[학부생 말투]** (실제 발표용)이 있다. 학부생 말투를 소리 내어 2회씩 연습한다.

## 📋 Q&A 답변 공통 원칙

### 답변 길이

- **이상**: 30초 ~ 1분
- **너무 짧으면**: 성의 없어 보임
- **너무 길면**: 청중 지루함 · 다른 질문 기회 사라짐

### 답변 구조

1. **핵심 결론을 먼저** (1문장)
2. **근거/설명** (2~3문장)
3. **마무리·확인** (선택, "추가 설명 필요하실까요?")

### 보고서 절 번호 활용

- 답변 중 보고서 절 번호를 자연스럽게 넣으면 신뢰도 상승
- 예: "5.4절에 자세히 기술되어 있는데요..."

### 모르는 것은 솔직히

- "그 부분은 본 연구에서 다루지 못한 한계입니다."
- "흥미로운 지적입니다. 향후 연구 과제로 검토하겠습니다."
- 절대 추측 금지 · 거짓말 금지

## 🎯 Q1. 왜 실제 LLM 호출이 아닌 시뮬레이션이 주 실험인가요?

**[원본]**
> 알고리즘 자체의 스케줄링 특성을 비교하려면 네트워크 지연이나 LLM 상태에 따른 변동을 배제해야 합니다. 시뮬레이션으로 알고리즘의 순수한 차이를 확인한 뒤, 실서버 실험(5.4절)으로 실제 환경에서도 경향이 유지됨을 검증했습니다.

**[학부생 말투]**
> 네, 좋은 질문이십니다. (호흡) 이 연구의 목적이 스케줄링 알고리즘 자체의 특성 비교였는데요, 실제 LLM 호출을 하면 네트워크 지연이나 LLM 상태가 매번 달라져서 알고리즘의 순수한 차이를 보기가 어렵습니다. (호흡) 그래서 시뮬레이션으로 먼저 알고리즘의 차이를 확인하고, 5.4절에서 Ollama 실서버 실험으로 같은 경향이 실제 환경에서도 나타나는지를 검증하는 순서로 진행했습니다.

**핵심 포인트**: 목적이 알고리즘 비교 → 변동 요소 배제 → 실서버로 경향 확인

**꼬리 질문 대비**:
- "실서버 실험을 더 크게 했어야 하지 않나요?" → Q7 참조
- "실서버에서 수치가 다르던데요?" → "실서버는 20건 규모라 수치 크기는 다르지만, FCFS·Priority·WFQ의 상대적 순서 차이는 시뮬레이션과 같은 패턴이 나왔습니다."

## 🎯 Q2. JFI(Jain's Fairness Index)는 무엇이고 왜 사용했나요?

**[원본]**
> 자원이 얼마나 고르게 분배되었는지를 0에서 1 사이 점수로 나타내는 지표입니다. 1이면 완전히 공평, 1/n이면 한 사용자가 독점하는 상태를 뜻합니다. 구독 등급별 차등 서비스가 의도한 비율대로 이뤄졌는지 수치로 확인하기 위해 사용했습니다.

**[학부생 말투]**
> 네, JFI는 Jain's Fairness Index의 약자인데요. (호흡) 자원이 사용자들 사이에 얼마나 고르게 분배됐는지를 0부터 1 사이의 점수로 나타내는 지표입니다. 1이면 완전 공평, 1을 사용자 수 n으로 나눈 값이면 한 사용자가 독점한 상태를 뜻합니다. (호흡) 저희가 이걸 쓴 이유는, 스케줄러의 공정성을 말로만 "공평하다, 차등적이다"라고 하기보다는 **숫자로 비교**하기 위해서였습니다. 결과적으로 FCFS, Priority, MLFQ 비선점형은 JFI가 1.000으로 완전 균등, WFQ는 0.316으로 가중치에 따른 의도적 차등이 수치로 확인됐습니다.

**핵심 포인트**: 약어 풀기 → 정의 → 왜 사용 → 실제 결과

**꼬리 질문 대비**:
- "WFQ 0.316은 불공평한 거 아닌가요?" → "0.316은 낮은 숫자지만, 이건 '불공평'이 아니라 '의도적 차등'입니다. WFQ는 가중치대로 차등 서비스를 제공하도록 설계됐고, 이 0.316이라는 값 자체가 설계대로 작동했다는 증거입니다."
- "JFI 공식을 아세요?" → "정확히는 (합의 제곱) 나누기 (n × 제곱의 합)으로 계산됩니다. 저희 구현에서는 WFQ 스케줄러 내부에 있습니다."

## 🎯 Q3. MLFQ 선점형이 왜 실제 LLM 환경에서 안 되나요?

**[원본]**
> LLM 추론은 한 번 시작되면 중간에 멈출 수 없습니다. Ollama를 포함한 대부분의 LLM 서빙 도구가 이러한 특성을 가지므로, 선점형 MLFQ는 시뮬레이션에서만 효과를 검증했습니다.

**[학부생 말투]**
> 네, 그 부분이 이 연구의 중요한 발견 중 하나인데요. (호흡) LLM은 한 번 응답을 생성하기 시작하면 중간에 멈추기가 거의 불가능합니다. Ollama를 포함해서 vLLM, TGI 같은 대부분의 LLM 서빙 도구가 추론 중단 기능을 지원하지 않기 때문이에요. (호흡) 선점형 MLFQ는 실행 중인 요청을 잠깐 멈추고 다른 요청으로 넘어가야 하는데, LLM에서 이게 안 되기 때문에 시뮬레이션에서만 효과를 검증할 수 있었습니다. 이 제약은 저희 연구의 한계이자 발견이기도 합니다.

**핵심 포인트**: LLM 중단 불가 → 대부분 도구가 미지원 → 시뮬레이션만 가능

**꼬리 질문 대비**:
- "중단을 지원하는 도구는 없어요?" → "제가 아는 범위에서는 거의 없습니다. 실험적인 토큰 단위 스트리밍 중단 같은 연구는 있지만, 표준 도구에는 들어가 있지 않습니다."

## 🎯 Q4. WFQ의 가중치는 어떻게 결정했나요?

**[원본]**
> 구독 등급별 상대적 서비스 비율을 Enterprise:Premium:Standard:Free = 100:50:10:1로 설정했습니다. 일반적인 SaaS 서비스 등급 체계를 참고하여, 최상위 등급이 최하위 등급보다 100배 수준의 서비스 비중을 받도록 배정했습니다.

**[학부생 말투]**
> 네, Enterprise부터 Free까지 **100, 50, 10, 1**로 설정했습니다. (호흡) 이 값은 저희 실험 목적을 위해 직접 정한 숫자인데요, 최상위 등급이 최하위 등급보다 100배 수준의 서비스 비중을 받도록 설계했습니다. 일반적인 SaaS 서비스의 등급별 요금 차이를 참고했고, 의도적으로 극단적인 비율로 설정해서 차등 효과가 명확히 드러나도록 했습니다. (호흡) 그래서 실험 결과 Enterprise가 Free보다 약 12배 빠르게 처리되는 패턴이 확인된 것입니다.

**핵심 포인트**: 100:50:10:1 → 직접 설정 → SaaS 참고 → 극단적 비율 의도

**꼬리 질문 대비**:
- "왜 100배인데 실제론 12배밖에 안 차이 나요?" → Q9 답변 뒷부분과 유사. "요청이 순차적으로 큐에 들어가는 설계라서 극단적 동시 경쟁이 안 일어났기 때문입니다."
- "숫자 근거가 있나요?" → "정확한 산업 데이터를 참조하기보다는, 극단적인 가중치 차이를 의도적으로 부여해 차등 효과를 실험하는 것이 목적이었습니다."

## 🎯 Q5. Rate Limiter와 스케줄러는 어떻게 구분되나요?

**[원본]**
> Rate Limiter는 요청의 시스템 진입 여부를 결정하는 전처리 단계이고, 스케줄러는 이미 진입한 요청 중 어떤 것을 먼저 처리할지 결정하는 단계입니다. 두 기능은 독립적으로 동작합니다.

**[학부생 말투]**
> 네, 이 둘은 역할이 완전히 다릅니다. (호흡) Rate Limiter는 **요청이 들어오느냐 마느냐**를 결정하는 문지기 역할이에요. 등급별 분당 한도를 초과한 요청은 HTTP 429로 거절합니다. (호흡) 반면 스케줄러는 **이미 들어온 요청 중에 어떤 걸 먼저 처리할지**를 결정합니다. 즉 Rate Limiter는 입장 심사, 스케줄러는 줄 정리라고 보시면 됩니다. 그래서 이 둘은 서로 독립적으로 동작하고, 저희 시스템에서도 Rate Limiter는 API 계층, 스케줄러는 그 다음 계층에 따로 있습니다.

**핵심 포인트**: 역할 차이 (입장 vs 순서) → 계층 위치 다름 → 독립 동작

**꼬리 질문 대비**:
- "Rate Limiter 알고리즘은 뭐 쓰나요?" → "토큰 버킷 방식입니다. 개념적으로는 토큰 버킷이고, 구현은 1분 단위 슬라이딩 윈도우 카운터로 단순화했습니다."
- "429는 HTTP 상태 코드인가요?" → "네, HTTP 429 Too Many Requests는 '요청이 너무 많다'는 뜻의 표준 응답 코드입니다."

## 🎯 Q6. 왜 Node.js와 Express를 선택했나요?

**[원본]**
> 학부 수준에서 구현·이해가 용이하며, 비동기 처리 모델이 LLM 호출과 같은 I/O 중심 작업에 적합하기 때문입니다.

**[학부생 말투]**
> 두 가지 이유가 있습니다. 첫째, 학부 수준에서 구현하고 이해하기 쉬운 환경이라는 점입니다. (호흡) 둘째, Node.js의 비동기 이벤트 루프 모델이 LLM 호출 같은 I/O 중심 작업에 잘 맞기 때문입니다. 요청을 받아서 LLM에 보내고 응답을 기다리는 동안 다른 요청도 처리할 수 있어야 하는데, 이게 Node.js의 특기예요. (호흡) Express는 Node.js의 가장 표준적인 웹 프레임워크라서 외부 라이브러리를 Express 하나만 쓸 수 있어서, 실제 스케줄링 구현에 집중할 수 있었습니다.

**핵심 포인트**: 학부 수준 적합 → 비동기/IO 특성 → Express는 최소 의존성

**꼬리 질문 대비**:
- "Python으로 했으면 어땠을까요?" → "Python도 가능하지만, LLM 요청은 I/O 대기가 많아서 Node.js의 이벤트 루프가 한 프로세스에서 여러 요청을 처리하기에 더 직관적이었다고 생각합니다."
- "Go나 Rust는 고려 안 했나요?" → "Go는 고성능 서버에 적합하지만 학부 수준에서 학습 곡선이 높고, Rust도 마찬가지라서 이번 프로젝트 범위를 벗어난다고 판단했습니다."

## 🎯 Q7. 실서버 실험 규모가 너무 작지 않나요?

**[원본]**
> 보고서 6장 제한사항에서 인정한 부분입니다. 20건 규모는 시뮬레이션 결과의 방향성이 실제 환경에서도 유지되는지 확인하는 수준이며, 실무 적용을 위해서는 수백~수천 건 규모의 부하 시험이 필요합니다.

**[학부생 말투]**
> 네, 맞습니다. 저희도 보고서 6장의 한계 부분에서 이 점을 인정했습니다. (호흡) 20건 규모는 실제 부하 시험이라기보다는, **시뮬레이션에서 확인한 방향성이 실제 LLM 환경에서도 유지되는지 확인하는 검증 수준**이라고 말씀드릴 수 있습니다. 실무 적용을 위해서는 수백에서 수천 건 규모의 부하 시험이 필요하고, 이건 향후 연구 과제로 남겨두었습니다.

**핵심 포인트**: 한계 인정 → 목적이 방향성 확인 → 부하 시험은 향후 과제

**꼬리 질문 대비**:
- "왜 더 크게 안 했어요?" → "로컬 Ollama 환경에서 한 건당 실제 LLM 추론 시간이 필요해서, 20건이라도 충분히 시간이 걸렸습니다. 개인 환경의 한계였다고 생각합니다."
- "결과가 믿을 만한가요?" → "20건 규모는 작지만, 시뮬레이션 500건과 같은 상대적 순서 패턴이 나왔다는 점에서 최소한의 검증은 됐다고 보고 있습니다."

## 🎯 Q8. 연속 배치(Continuous Batching)는 고려했나요?

**[원본]**
> 본 프로젝트에서 다루지 못한 한계입니다. 실제 LLM 서빙 시스템(vLLM, TGI 등)은 연속 배치로 여러 요청을 동시에 처리하는데, 이 경우 스케줄링 효과가 달라질 수 있습니다. 향후 연구 과제로 남겨두었습니다.

**[학부생 말투]**
> 네, 그 부분은 본 프로젝트에서 다루지 못한 한계입니다. (호흡) 연속 배치는 실제 상용 LLM 서빙 도구인 vLLM이나 TGI에서 쓰는 기법인데요, **실행 중인 배치에 새로운 요청을 계속 끼워넣어 동시에 처리하는 방식**입니다. 이 경우에는 저희처럼 한 건씩 순차 처리할 때와 스케줄링 효과가 달라질 수 있습니다. (호흡) 저희 프로젝트는 단순 순차 처리 기반이어서 이 부분을 다루지 못했고, 보고서 6장과 7장에서 향후 연구 과제로 명시했습니다.

**핵심 포인트**: 한계 인정 → 연속 배치 설명 → vLLM/TGI 언급 → 향후 과제

**꼬리 질문 대비**:
- "연속 배치를 구현하면 어떻게 될까요?" → "흥미로운 질문입니다. 연속 배치에서는 '어떤 요청을 현재 배치에 끼워 넣을지'가 스케줄링의 새로운 차원이 될 수 있고, 저희 알고리즘과 결합 방식에 대해 추가 연구가 필요할 것 같습니다."

## 🎯 Q9. VFT 계산에서 처리 비용을 1로 고정한 이유는?

**[원본]**
> LLM 요청은 응답 토큰 수를 미리 알 수 없어 정확한 처리 시간을 예측하기 어렵기 때문입니다. 보고서 4.3절에 상세히 기술했으며, 향후 토큰 수 예측 모델을 결합하는 개선 방향을 고려하고 있습니다.

**[학부생 말투]**
> 네, 이게 저희 WFQ 구현의 **가장 중요한 설계 결정** 중 하나였습니다. (호흡) 원래 WFQ에서는 요청마다 처리 비용이 다를 수 있고, VFT 계산에 그 비용이 반영됩니다. 그런데 LLM 요청은 응답을 만들기 전까지는 **토큰이 몇 개 나올지 미리 알 수가 없어요**. (호흡) 그래서 저희는 비용을 일단 모든 요청에 대해 1로 고정했습니다. 이렇게 하면 가중치만으로 차등이 이루어지지만, 실제 처리 비용의 차이는 반영되지 않는 한계가 생깁니다. (호흡) 향후 과제로는 토큰 수를 미리 예측하는 모델을 붙여서 VFT 계산을 더 정교하게 만드는 방향을 생각하고 있습니다. 보고서 4.3절에 자세히 기술했습니다.

**핵심 포인트**: 설계 결정 → 토큰 수 예측 불가 → 비용=1로 단순화 → 한계 인정 → 향후 과제

**꼬리 질문 대비**:
- "토큰 수 예측이 가능한가요?" → "가능한 연구 방향이지만 정확도가 높지는 않습니다. 입력 프롬프트의 길이·복잡도 등에서 출력 토큰 수를 추정하는 연구가 일부 있지만, 저희 프로젝트 범위를 벗어납니다."

## 🎯 Q10. 이 연구가 왜 의미가 있나요?

**[원본]**
> ChatGPT 같은 LLM 서비스가 보편화되면서, 여러 사용자가 동시에 요청하는 환경에서의 공정성 문제가 중요해지고 있습니다. 운영체제에서 오래 연구된 스케줄링 이론을 LLM 환경에 적용해본 것 자체가 교육적 의의가 있습니다.

**[학부생 말투]**
> 세 가지 의의가 있다고 생각합니다. (호흡)

> 첫째, ChatGPT 같은 LLM 서비스가 많은 사용자에게 쓰이면서 **공정성 문제**가 점점 중요해지고 있습니다. 유료 사용자와 무료 사용자, 긴급한 요청과 일상적 요청을 구분하는 스케줄링 기준이 필요한데, 이 분야가 아직 초기 단계입니다.

> 둘째, 운영체제에서 수십 년간 연구된 스케줄링 이론을 LLM 환경에 적용해봄으로써, **기존 이론이 얼마나 적용 가능한지**와 **어떤 한계가 있는지**를 학부 수준에서 검증한 것 자체가 교육적으로 의미가 있다고 봅니다.

> 셋째, MLFQ 선점형이 실서버에 적용 불가능하다는 점이나, WFQ의 가중치 차등이 실제로 어떻게 드러나는지 등, **실제로 해봐야 알 수 있는 지점들**을 숫자로 확인했다는 실용적 의의도 있습니다.

**핵심 포인트**: 3가지 의의 (공정성 중요성, 이론 적용 가능성 검증, 실증 지점)

**꼬리 질문 대비**:
- "실제 LLM 서비스 업체에 쓰일 만한가요?" → "저희 구현 그대로는 실제 서비스에 바로 쓰기 어렵습니다. 하지만 핵심 아이디어, 특히 구독 등급별 WFQ 가중치 차등 같은 개념은 확장해서 적용할 수 있을 것이라고 생각합니다."

## 🎯 Q11. 앞으로 더 발전시킨다면?

**[원본]**
> 세 가지 방향을 생각하고 있습니다. 첫째, 연속 배치 환경에서의 스케줄링. 둘째, 더 큰 규모의 부하 시험. 셋째, 응답 길이를 고려한 비용 예측(VFT 개선)입니다.

**[학부생 말투]**
> 네 가지 방향을 생각하고 있습니다. (호흡)

> 첫째, **토큰 기반 비용 모델**입니다. 아까 Q9에서 말씀드린 것처럼, VFT 계산에 토큰 수 예측을 결합하는 연구입니다.

> 둘째, **대시보드 고도화**입니다. 현재는 HTTP 폴링 방식인데, WebSocket 기반 실시간 갱신으로 바꾸면 더 직관적으로 스케줄링 동작을 관찰할 수 있을 것 같습니다.

> 셋째, **동일 등급 내 공정성 검증**입니다. 저희 실험은 등급당 1명만 두었는데, 같은 등급 내에서 여러 사용자가 경쟁할 때의 공정성도 검증이 필요합니다.

> 넷째, **대규모 부하 실험**입니다. 수백에서 수천 건 규모로 스케일을 키워서 실제 서비스 환경에 가까운 부하를 재현하는 것입니다.

**핵심 포인트**: 4가지 (토큰 비용, 대시보드, 동일 등급 공정성, 대규모 부하)

**꼬리 질문 대비**:
- "그중에서 가장 중요한 건 뭐예요?" → "개인적으로는 동일 등급 내 공정성 검증이 가장 흥미롭습니다. 실제 서비스에서 같은 요금제를 쓰는 사용자들 사이의 공정성이 문제가 될 테니까요."

## 🎯 추가 예상 질문 (Q12~Q20)

demo-scenario.md에 없는 질문이지만 나올 수 있는 것들.

### Q12. 왜 TypeScript가 아닌 JavaScript인가요?

> 학부 수준에서 TypeScript의 타입 시스템 학습이 프로젝트 본래 목적인 **스케줄링 알고리즘 구현**에 집중할 시간을 빼앗는다고 판단했습니다. 저희는 JSDoc 주석으로 타입 힌트를 제공하는 수준에서 마무리했습니다.

### Q13. MongoDB 같은 DB 안 쓰나요?

> 프로젝트 범위가 "스케줄러의 동작 검증"이었기 때문에, 메모리 기반 큐로 처리 순서만 관리하면 충분했습니다. 영속화나 고가용성은 다루지 않았습니다.

### Q14. Redis로 분산 큐 구현은 고려했나요?

> 분산 환경은 저희 프로젝트 범위를 벗어납니다. 학부 수준에서 단일 서버 기반으로 알고리즘 검증에 집중했다는 점이 한계이기도 하고 의도한 범위이기도 합니다.

### Q15. 실제 ChatGPT는 어떤 스케줄링을 쓸까요?

> 정확한 내부 구현은 공개되어 있지 않아서 답변드리기 어렵습니다. 공개된 연구로는 vLLM 쪽의 PagedAttention이나 연속 배치 기법이 있고, 스케줄링 자체보다는 GPU 메모리 관리에 더 초점이 맞춰져 있는 것 같습니다.

### Q16. 선점형 MLFQ를 스트리밍 방식으로 흉내 낼 수 있을까요?

> 흥미로운 질문입니다. 토큰 단위로 LLM 출력을 스트리밍하면서 중간에 다른 요청을 끼워 넣는 연구가 일부 있지만, Ollama 표준 API에는 이런 기능이 없습니다. 저희 프로젝트에서는 시도하지 않았고, 향후 과제에 추가할 수 있을 것 같습니다.

### Q17. 시뮬레이션에서 요청 도착 패턴은 어떻게 생성했나요?

> 기본 실험은 처리 시간이 10~100ms 사이로 균일 랜덤하게 분포하도록 생성했고, MLFQ 선점형 실험은 Short/Medium/Long 세 유형을 섞은 버스트 패턴이었습니다. 자세한 설정은 보고서 5.1절과 표 5~7에 있습니다.

### Q18. 왜 4개 등급인가요? 3개나 5개도 가능할 텐데요.

> 업계 SaaS에서 흔히 쓰이는 Enterprise-Premium-Standard-Free 4단계 모델을 참고했습니다. 알고리즘 자체는 등급 수를 늘이거나 줄일 수 있지만, 실험 범위를 제한하기 위해 4개로 고정했습니다.

### Q19. 이 연구로 어떤 후속 연구가 가능할까요?

> Q11 답변과 겹치는데요, 특히 **토큰 수 기반 비용 예측 모델**을 VFT에 결합하는 연구가 가장 흥미로울 것 같습니다. 그리고 연속 배치 환경에서의 스케줄링도 실무적으로 중요한 방향이라고 생각합니다.

### Q20. 발표 중 데모가 실패하면 어떻게 하실 건가요?

> 네, 그런 경우에 대비해서 백업 녹화본을 준비해 두었습니다. 실시간 데모가 실패하면 녹화본으로 전환해서 같은 내용을 보여드릴 예정입니다.
