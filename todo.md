## 2. Priority vs WFQ 입력 차이 명확화 (3.3.2절, 3.3.4절)

방향: Priority는 등급+긴급도 사용, WFQ는 등급만 사용. 차이를 명시적으로 서술.

### 2-1. Priority 우선순위 결정 로직 명확화

- [x] line 110의 "테넌트 등급과 요청 헤더의 긴급도 필드를 조합" 서술 구체화
  - 현재 매핑이 사실상 "등급 = 우선순위"로 읽힘 (긴급도의 역할 불명확)
  - 수정 방안: 등급이 기본 우선순위, 긴급도 플래그가 1단계 상승시키는 구조로 명확화
  - 또는: 등급이 곧 우선순위이며 긴급도는 같은 등급 내 정렬 기준이라고 명시

### 2-2. WFQ에서 긴급도 미사용 명시

- [x] 3.3.4절(line 120-124)에 긴급도를 사용하지 않는다는 내용 추가
  - 예: "WFQ는 테넌트 등급별 가중치만을 기준으로 자원을 배분하며, 개별 요청의 긴급도 필드는 사용하지 않는다. 이는 WFQ의 목적이 긴급 요청 우선 처리가 아니라 등급 간 비례적 자원 배분이기 때문이다."
- [x] VFT 계산식 분모 명확화: Weight = 테넌트 등급 가중치 (Enterprise=100, Premium=50, Standard=10, Free=1)

### 2-3. 두 알고리즘의 입력/동작 차이 비교 추가

- [x] 3.3.4절 끝이나 3.4절에 간단한 비교 서술 추가 (표는 불필요, 1~2문장)
  - Priority: 등급+긴급도 → 절대적 순서 (높은 우선순위가 항상 먼저)
  - WFQ: 등급만 → 비례적 배분 (모든 등급이 가중치 비율대로 서비스)
