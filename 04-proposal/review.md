2.1 스케줄링 알고리즘

- MLFQ 부분의 ‘타임 퀀텀(Time Quantum)이란, 각 작업에 주어지는 최대 실행 시간으로’
    - 작업 당 할당되는 총 시간처럼 느껴짐 (전체 실행 시간)
    - ‘한 번에 할당되는 실행 시간’이라고 수정하는게 어떤지?
- WFQ 부분의 GPS 설명 부분에서, 마치 가중치를 고려하지 않고 똑같이 배분하는 것처럼 느껴짐
    - "GPS(Generalized Processor Sharing)는 자원을 **가중치에 비례하여 동시에** 배분하는 이상적인 수학적 모델이며, WFQ는 이를 개별 요청(Discrete request) 단위의 현실 시스템에서 비슷하게 구현한 실용적인 스케줄링 기법이다."로 수정

2.3 공정성

"토큰 수 기반의 자원량 측정이 어렵다"는 이유가 정확하지 않음.

- JFI xi는 **처리 완료 후** 측정하는 거니까 입력+출력 토큰 합산 가능
- 즉 토큰 수 기반으로 할 수 **있는데** 안 하는 거예요

더 정확한 이유는:

**"Cost=1로 단순화했기 때문에 xi도 요청 수로 정의하여 스케줄링 기준과 측정 기준을 일치시킨다"**

이렇게 수정하는 게 더 정확함. 즉:

- 기존: "토큰 수 측정이 어려워서" → ❌ 틀림
- 수정: "Cost=1 단순화와 일관성을 위해" → ✅ 정확

3.4 핵심 기술 특징 - JFI

**첫 번째 지적:**

맞아요. "등급이 아닌 행동 패턴"이라고 강조했는데, 사실 **어떤 기준으로 큐를 조정하든** 처리량 차이가 생기면 JFI는 낮아질 수 있어요. 저 표현은 MLFQ의 특징을 설명한 거지, JFI가 낮아지는 이유를 정확히 설명한 건 아니에요.
더 정확하게는 **"하위 큐로 강등된 테넌트는 처리 횟수가 줄어들 수 있어 시스템 수준 JFI가 낮아질 수 있다"** 가 맞아요.

**두 번째 지적 (FCFS):**

맞아요! FCFS도 테넌트별로 보내는 요청 수가 다르면 JFI가 낮아질 수 있어요.
제안서에서 "FCFS는 JFI가 1에 가깝다"고 쓴 건 암묵적으로 **모든 테넌트가 동일한 수의 요청을 보낸다고 가정**한 거예요. 실험 설계에서 이걸 명시적으로 통제해야 해요.
실험할 때 테넌트별 요청 수를 동일하게 맞추지 않으면 FCFS도 JFI가 낮게 나올 수 있어서, 이 가정을 실험 설계 부분에 명시하면 좋을 것 같아요 😊

**비교 가능한 그룹:**

- **FCFS vs MLFQ**: 둘 다 등급 무관하게 동작하니까 JFI로 직접 비교 의미있음
- **Priority**: 등급별 차등이 의도된 거라 FCFS/MLFQ랑 단순 비교는 어렵고, **에이징이 기아를 얼마나 잘 방지하는지** 관점으로 봐야함

**WFQ는:**

- 차등 배분이 설계 의도니까 나머지 셋이랑 JFI 숫자로 직접 비교는 의미없고
- **"가중치 비율대로 잘 배분됐는가"** 를 별도로 검증해야 함 → 이때 가중치 JFI 사용

그래서 결국:

- **FCFS vs MLFQ**: JFI로 직접 비교 ✅
- **Priority**: 에이징 효과 관점으로 ✅
- **WFQ**: 가중치 비례 달성 여부로 별도 검증 ✅

→ 가중치 JFI는 WFQ 자체의 공정성을 검증할때 사용해야하는데, 그에 관련된 내용이 없음

3.5 성능 평가 지표

tokens/s는 엄밀히 말하면 **Ollama가 생성한 출력 토큰 수 / 시간** 임
그래서 입력 토큰이 아무리 길어도 출력이 짧으면 tokens/s가 낮게 나오고, 입력이 짧아도 출력이 길면 높게 나옴
 tokens/s는 **"시스템이 얼마나 많은 텍스트를 생성했는가"** 를 나타내는 거지, 
**"얼마나 많은 작업을 처리했는가"** 와는 조금 다름

그래서 사실 tokens/s를 "처리량의 부가 지표"로 쓰는 건 살짝 애매함
 더 정확하게는:

- **req/s**: 처리량 지표
- **tokens/s**: LLM 생성 속도 지표

즉, 성능 평가 지표의 새로운 섹션으로 넣는게 더 깔끔하지 않나?
