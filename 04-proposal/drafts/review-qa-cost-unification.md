# Cost=1 단순화와 요청 단위 통일 분석

## 질문

> Rate Limiter의 토큰 버킷이랑 JFI xi, 처리량 계산에서도 요청 수 기준으로 단순화하는 거 맞나?
>
> Rate Limiter는 처리 완료 후 입력+출력 토큰 합산해서 사후 차감하는 방식으로 토큰 수 기반도 가능하고, xi도 처리 완료 후 토큰 수 합산이 가능하며, 처리량 계산도 마찬가지임.
>
> 근데 Cost=1 단순화와 일관성을 맞추기 위해 셋 다 요청 단위로 통일한다는 건가? 스케줄링 기준(요청 수)이랑 측정 기준(xi, Rate Limiter)을 일치시키는 게 실험 결과 해석이 더 깔끔하므로?

## 답변

맞다. 세 지표 모두 **요청 수 기반으로 통일**하되, 각각의 역할과 이유가 조금씩 다르다.

### 1. Rate Limiter 토큰 버킷: 1 요청 = 1 버킷 토큰

코드(`RateLimiterScheduler.js`)에서 확인:

```javascript
this.currentTokens -= 1;  // 요청 1건 = 버킷 토큰 1개 차감
```

- **역할**: 전처리 게이트 (스케줄러 진입 전 유입량 제한)
- **요청 단위인 이유**: Rate Limiter는 요청이 도착하는 시점에 즉시 판단해야 하므로, 아직 처리되지 않은 요청의 토큰 수를 알 수 없다. 따라서 요청 단위가 자연스럽다.
- **토큰 수 기반 가능 여부**: 이론적으로는 처리 완료 후 입력+출력 토큰을 합산하여 사후 차감하는 방식도 가능하지만, 이는 Rate Limiter의 본래 목적(사전 유입 제어)과 맞지 않고 구현 복잡도가 올라간다.

### 2. JFI $x_i$: 측정 구간 내 처리 완료 요청 수

제안서 2.3절에서 정의:

> $x_i$를 측정 구간 동안 테넌트 $i$가 처리 완료한 요청 수로 정의한다.

- **역할**: 공정성 측정의 입력값
- **요청 단위인 이유**: 스케줄러가 Cost=1로 모든 요청을 동일하게 취급하므로, 측정 기준도 요청 수로 맞추면 "스케줄링 결정이 공정했는가"를 직접 평가할 수 있다.
- **토큰 수 기반 가능 여부**: 처리 완료 후 토큰 수 합산이 가능하다. 그러나 이 경우 "스케줄러는 요청 단위로 결정했는데 평가는 토큰 단위"라는 불일치가 생겨, JFI가 낮게 나왔을 때 그것이 스케줄러의 불공정 때문인지 응답 길이 차이 때문인지 구분할 수 없다.

### 3. 처리량(Throughput): req/s

제안서 3.5절에서 정의:

> 단위 시간당 처리 완료되는 요청의 수(req/s)

- **역할**: 시스템 효율성 측정
- **요청 단위인 이유**: 스케줄러의 디스패치 효율을 직접 반영한다. LLM이 생성한 텍스트 양은 별도의 "생성 속도(tokens/s)" 지표로 분리하여 측정한다.
- **토큰 수 기반 가능 여부**: tokens/s로도 측정 가능하며, 이는 "생성 속도"로 이미 별도 지표화했다. 처리량과 생성 속도를 분리함으로써, 스케줄러 성능과 LLM 백엔드 성능을 독립적으로 평가할 수 있다.

### 정리: 왜 셋 다 요청 단위인가

| 구분 | 단위 | 이유 |
|------|------|------|
| **스케줄링 기준** (Cost=1) | 요청 수 | 모든 요청을 동일 비용으로 취급 |
| **Rate Limiter** | 요청 수 | 사전 유입 제어 → 토큰 수 사전 예측 불가 |
| **JFI $x_i$** | 요청 수 | 스케줄링 기준과 측정 기준 일치 → 인과 해석 가능 |
| **처리량** | 요청 수 (req/s) | 스케줄러 디스패치 효율 직접 반영 |
| **생성 속도** | 토큰 수 (tokens/s) | LLM 출력량 → 스케줄러와 독립적 지표 |

**핵심 원리**: 스케줄링 기준(요청 수)과 측정 기준(xi, Rate Limiter, 처리량)을 일치시키면, 실험 결과에서 **"스케줄러의 결정이 공정했는가"**를 직접 해석할 수 있다. 만약 측정 기준이 토큰 수라면, JFI 변동이 스케줄러 때문인지 응답 길이 차이 때문인지 혼재되어 해석이 어려워진다.

### 결론

제안서 수정은 불필요하다. 현재 제안서의 설계가 이미 이 원리를 정확히 반영하고 있다:

- 2.3절: $x_i$ = 처리 완료 요청 수 (Cost=1 일관성 명시)
- 3.3.1절: WFQ Cost=1 단순화
- 3.5절: 처리량(req/s)과 생성 속도(tokens/s) 분리
- 3.6절: Rate Limiter 1 요청 = 1 버킷 토큰
